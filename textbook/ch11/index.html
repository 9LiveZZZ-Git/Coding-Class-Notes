11.1 Introduction: Why Process Matters

Creative coding exists at the intersection of art and engineering. Unlike traditional software development, where requirements are specified in advance, creative coding projects evolve through experimentation. Unlike traditional art-making, where the medium is familiar, creative coding requires learning new tools and debugging technical problems alongside aesthetic exploration.

Many beginning creative coders focus exclusively on output — the visual or sonic result of their sketch. This is understandable: the output is what you see and share. But as projects grow in complexity, the process behind that output becomes equally important. How do you keep track of what you have tried? How do you find and fix bugs? How do you organize code so that you can return to it a month later and understand what it does?

Professional software engineers spend a great deal of time on process: planning sprints, reviewing code, writing tests, documenting APIs. Creative coders do not need all of that machinery, but they do need a lightweight version of it. The practices in this chapter will save you hours of frustration, help you produce better work, and prepare you for collaboration — whether with peers, clients, or future versions of yourself.

💡 Key Concept

The creative coding workflow is a loop, not a line. You do not plan, then build, then finish. You plan a little, build a little, observe what happens, adjust your plan, and build again. The practices in this chapter support that iterative loop.

11.1.1 The Iteration Cycle

Every successful creative coding project follows some variation of this cycle:

Conceive → Prototype → Observe → Evaluate → Adjust → (repeat)

At each turn of the cycle, you make the project a little better, a little more interesting, a little closer to the vision forming in your mind — a vision that itself evolves as you work. This is not a flaw in the process; it is the process. Many of the most compelling creative coding works emerged from productive accidents: a bug that produced an interesting visual, a parameter value that created an unforeseen pattern.

The practices in this chapter — planning, organizing, debugging, reviewing, optimizing, documenting, and sharing — are all in service of making this iteration cycle faster and more productive. When your code is organized, you can change things quickly. When you save checkpoints, you can recover from mistakes. When you debug systematically, you spend less time stuck and more time creating.

11.2 Project Planning

You might think that planning is antithetical to creative exploration. In fact, the opposite is true. A lightweight plan gives you a foundation to explore from. Without a plan, you wander aimlessly. With a plan, you have a direction — and you can decide intentionally when to deviate from it.

11.2.1 The Minimum Viable Sketch

In software engineering, a Minimum Viable Product (MVP) is the simplest version of a product that delivers value. In creative coding, we can think in terms of a Minimum Viable Sketch (MVS): the simplest version of your idea that you can run and evaluate.

📖 Definition

Minimum Viable Sketch (MVS) — The simplest runnable version of a creative coding idea that demonstrates its core concept. It should take no more than one work session to build.

For example, if your project idea is "an interactive particle system that responds to music," your MVS might be "ten circles that move randomly on screen." That is enough to test your basic architecture (a particle array, an update loop, a render step) without worrying about audio input, visual polish, or performance.

11.2.2 Breaking Down a Project

Once you have an idea, decompose it into layers of increasing complexity:

LayerExample (Audio-Reactive Particles)
1. Static geometryDraw 100 circles at random positions
2. AnimationMake circles move with velocity vectors
3. InteractionParticles respond to mouse position
4. Data inputAdd FFT audio analysis, map bass to size
5. Visual polishAdd color gradients, trails, blending modes
6. PerformanceOptimize for 1000+ particles at 60fps

Each layer builds on the previous one. If you complete layers 1 through 3, you have a working interactive sketch. Layers 4 through 6 add richness, but the project is viable without them.

11.2.3 Milestones and Scope Management

For a multi-week project, define explicit milestones with dates:

PhaseDurationActivity
Exploration1–2 weeksMake many small sketches. Try different techniques.
Selection1 sessionReview sketches. Identify 2–3 candidates.
Development2–3 weeksDevelop chosen direction. Add interaction, polish aesthetics.
Finalization1 weekBug fixes, documentation, presentation preparation.
⚠ Common Mistake

Scope creep. The biggest risk in creative coding projects is trying to do too much. It is far better to have a polished, simple sketch than a broken, complex one. If you find yourself constantly adding features without finishing any of them, stop and ask: "What is the one thing that makes this sketch interesting?" Focus on that.

11.2.4 The Work Log

A work log is the single most important documentation practice for a creative coding project. It is a chronological journal of your development process.

Each entry should include:

  1. Date and approximate time spent
  2. Session goal — What were you setting out to accomplish?
  3. Actions taken — What did you actually do? Be specific about code changes, parameter values, and approaches tried.
  4. Results — What happened? Include screenshots for visual results.
  5. Failures and dead ends — What did you try that did not work? Why do you think it failed?
  6. Tools and resources — What documentation, tutorials, or references did you consult?
  7. AI tool usage — If you used an LLM, what did you ask and how did you use the response?
  8. Next steps — What will you do in the next session?

The work log serves as a memory aid (you will forget what you tried), a debugging tool ("it worked on Feb 8 with a noise scale of 0.003"), a reflection surface, and evidence of your engagement with the project.

11.3 Code Organization

When a sketch is 20 lines long, organization does not matter much. When it grows to 200 or 2000 lines, organization is the difference between a project you can continue developing and one you have to abandon because you cannot understand your own code.

11.3.1 Naming Conventions

Names are the most important form of documentation. A well-named variable explains itself:

// Bad: What do these mean?
let n = 100;
let s = 0.003;
let d = 2;
let c;

// Good: Self-documenting names
let particleCount = 100;
let noiseScale = 0.003;
let damping = 2;
let canvas;

Follow consistent conventions:

  • Variables and functions: camelCase — particleCount, updatePosition()
  • Constants: UPPER_SNAKE_CASE — MAX_PARTICLES, NOISE_SCALE
  • Classes: PascalCase — Particle, FlowField
  • Boolean variables: prefix with is, has, or should — isRunning, hasCollided

11.3.2 Constants vs. Magic Numbers

📖 Definition

Magic number — A numeric literal in code whose meaning is not obvious from context. Magic numbers make code hard to read and hard to modify.

Extract magic numbers into named constants at the top of your sketch:

// Bad: magic numbers scattered throughout
function setup() {
  createCanvas(800, 600);
  for (let i = 0; i < 200; i++) {
    particles.push(new Particle(random(800), random(600)));
  }
}

function draw() {
  background(0, 25);
  for (let p of particles) {
    p.applyForce(0, 0.1);
    p.update(0.98);
  }
}

// Good: constants at the top
const CANVAS_W = 800;
const CANVAS_H = 600;
const NUM_PARTICLES = 200;
const TRAIL_ALPHA = 25;
const GRAVITY = 0.1;
const DAMPING = 0.98;

function setup() {
  createCanvas(CANVAS_W, CANVAS_H);
  for (let i = 0; i < NUM_PARTICLES; i++) {
    particles.push(new Particle(random(CANVAS_W), random(CANVAS_H)));
  }
}

function draw() {
  background(0, TRAIL_ALPHA);
  for (let p of particles) {
    p.applyForce(0, GRAVITY);
    p.update(DAMPING);
  }
}

This approach has two major benefits: you can see all tunable parameters at a glance, and changing a value in one place updates it everywhere.

11.3.3 Separating Concerns

A common pattern in creative coding is to separate your sketch into logical sections:

// ========== CONFIGURATION ==========
const CANVAS_W = 800;
const CANVAS_H = 600;
const NUM_PARTICLES = 500;

// ========== STATE ==========
let particles = [];
let flowField;

// ========== SETUP ==========
function setup() {
  createCanvas(CANVAS_W, CANVAS_H);
  initParticles();
  flowField = new FlowField();
}

// ========== DRAW LOOP ==========
function draw() {
  background(0, 20);
  flowField.update();
  updateParticles();
  renderParticles();
  drawUI();
}

// ========== PARTICLE FUNCTIONS ==========
function initParticles() { /* ... */ }
function updateParticles() { /* ... */ }
function renderParticles() { /* ... */ }

// ========== UI FUNCTIONS ==========
function drawUI() { /* ... */ }

// ========== INPUT HANDLERS ==========
function mousePressed() { /* ... */ }
function keyPressed() { /* ... */ }

11.3.4 Comments: When and How

Good comments explain why, not what. The code already says what it does; comments should explain the reasoning behind non-obvious decisions:

// Bad: restates the code
i++; // increment i

// Good: explains the reasoning
// Wrap particles at edges instead of bouncing, so the
// flow field appears infinite rather than contained
if (this.pos.x > width) this.pos.x = 0;

11.3.5 Refactoring

📖 Definition

Refactoring — Changing the structure of code without changing its behavior. The goal is to make code more readable, maintainable, or efficient.

Common refactoring moves in creative coding:

  • Extract function: Take a block of code that does one thing and turn it into a named function
  • Extract constant: Replace a magic number with a named constant
  • Rename: Give variables and functions more descriptive names
  • Extract class: Group related data and functions into a class (e.g., a Particle class)
  • Simplify conditionals: Replace complex if/else chains with clearer logic

11.4 Version Control Concepts

Have you ever had a sketch working perfectly, made some changes, and then wished you could go back to the working version? Have you ever named files sketch_final.js, sketch_final_v2.js, sketch_ACTUALLY_final.js? Version control solves these problems.

11.4.1 What Is Version Control?

📖 Definition

Version control — A system that records changes to files over time so that you can recall specific versions later. It is like an unlimited undo history for your entire project.

The most widely used version control system is Git, created by Linus Torvalds in 2005. Git tracks every change you make, who made it, and when. It lets you create branches to experiment without affecting your main code, and merge successful experiments back in.

11.4.2 Core Git Concepts

You do not need to master Git for this course, but understanding the core concepts will serve you well:

ConceptWhat It MeansAnalogy
RepositoryA project folder tracked by GitA filing cabinet for all versions of your project
CommitA snapshot of your project at a point in timeTaking a photo of your desk
BranchAn independent line of developmentA parallel universe where you try something different
MergeCombining changes from two branchesBringing successful experiments from the parallel universe back to the main timeline
HistoryThe chronological list of all commitsA photo album showing every version of your project

11.4.3 Git for Creative Coders: A Minimal Workflow

If you are using Git from the command line, here is a minimal workflow:

// Initialize a new Git repository
git init

// Check what has changed
git status

// Stage changes for commit
git add sketch.js

// Save a snapshot with a descriptive message
git commit -m "Working flow field with color trails"

// Create a branch for an experiment
git branch experiment-audio
git checkout experiment-audio

// ... make changes, test them ...

// If the experiment works, merge it back
git checkout main
git merge experiment-audio

// If the experiment fails, just switch back
git checkout main
💡 Key Concept

Commit messages matter. A good commit message describes what changed and why. "Added audio" is vague. "Add FFT analysis with bass-driven particle size" tells you exactly what that snapshot contains. When you need to find a specific version later, good messages are invaluable.

11.4.4 Saving Iterations Without Git

If you are not ready to use Git, you can still practice version control manually:

  • p5.js Web Editor: Use File > Duplicate before making major changes. Name copies descriptively: "Particles v3 - before audio input"
  • Local files: Copy your project folder and add a date: my-sketch-2026-02-12
  • Cloud storage: Use Google Drive or Dropbox, which keep file version history

Whatever method you use, the key habit is: save a checkpoint before making a significant change. When your experiment does not work, you can always go back.

11.4.5 Platforms for Hosting Code

If you decide to use Git, you will likely want a remote host for your repository. The most popular platforms are:

  • GitHub — The largest code hosting platform. Free for public and private repositories. Has a strong creative coding community. GitHub Pages lets you host sketches as live websites for free.
  • GitLab — Similar to GitHub with built-in CI/CD features. Also free for individuals.
  • Bitbucket — Another alternative, popular in some professional settings.

For creative coding, GitHub is the most common choice because of its community features (starring projects, forking sketches, GitHub Pages hosting) and because many creative coding libraries (p5.js, three.js, openFrameworks) are hosted there.

💡 Key Concept

Your GitHub profile is a portfolio. Each repository you create is visible to potential collaborators, employers, and fellow artists. Keeping your creative coding projects on GitHub builds a body of work over time. Add descriptions and screenshots to your repositories to make them discoverable.

✍ Knowledge Check

In Git, what is a commit?

A commit is a saved snapshot of your entire project at a moment in time. It includes a message describing what changed and why. You can return to any commit in your history, making it safe to experiment freely.

11.5 Debugging Strategies

Debugging is the art of finding and fixing errors in your code. It is one of the most important skills a programmer can develop, and it is a skill that improves with practice. Experienced programmers are not people who write bug-free code — they are people who find and fix bugs efficiently.

11.5.1 Reading Error Messages

When your sketch crashes, the browser console displays an error message. These messages look intimidating but follow a consistent pattern:

Uncaught TypeError: Cannot read properties of undefined (reading 'x')
    at Particle.update (sketch.js:42)
    at draw (sketch.js:18)

Breaking this down:

  • Error type: TypeError — you tried to use a value in a way that does not match its type
  • Description: Cannot read properties of undefined — you tried to access .x on something that is undefined
  • Location: sketch.js:42 in the Particle.update function, called from draw at line 18
💡 Key Concept

Read the error message carefully. Most beginners glance at the red text and panic. Instead, read it like a sentence. The browser is telling you exactly what went wrong and where. The stack trace (the indented lines) shows you the chain of function calls that led to the error.

11.5.2 Common p5.js Errors

Here are the errors you will encounter most frequently in p5.js, along with their causes and fixes. Run this sketch to see common errors in action:

Cannot read properties of undefined Accessing a property on a variable that has no value Check that the object is properly initialized before you access its properties.

11.5.3 Console.log Debugging

The console.log() function is your primary debugging tool. Use it to inspect values at specific points in your code:

function draw() {
  // Check if variables have expected values
  console.log("mouseX:", mouseX, "mouseY:", mouseY);

  // Check if a condition is being reached
  if (dist(mouseX, mouseY, targetX, targetY) < 20) {
    console.log("HIT! Target reached");
  }

  // Check array contents
  console.log("particles:", particles.length, "first:", particles[0]);
}
⚠ Common Mistake

Logging inside draw() without throttling. Since draw() runs 60 times per second, a console.log() inside it will flood the console with thousands of messages. Use frameCount % 60 === 0 to log once per second, or log only when a condition changes.

// Log once per second instead of 60 times per second
if (frameCount % 60 === 0) {
  console.log("FPS:", frameRate().toFixed(1), "Particles:", particles.length);
}

11.5.4 Isolating the Problem

When something goes wrong in a complex sketch, isolate the problem. This means systematically removing parts of your code until you find the part that causes the error:

  1. Comment out sections. If you have particles, a flow field, and audio input, comment out two of them. Does the remaining one work? Add the others back one at a time.
  2. Simplify values. Replace complex expressions with hardcoded values. Instead of let angle = noise(x * 0.01, y * 0.01) * TWO_PI;, try let angle = PI / 4;. If that works, the problem is in the noise calculation.
  3. Create a minimal example. Start a new sketch with just the problematic code. If the bug disappears, the problem is an interaction between parts.

11.5.5 Binary Search Debugging

When you have a long stretch of code and do not know where the bug is, use binary search debugging:

  1. Add a return or comment block at the midpoint of your draw() function
  2. If the bug disappears, it is in the second half. If it persists, it is in the first half.
  3. Move the divider to the midpoint of the remaining suspect region
  4. Repeat until you have narrowed it down to a few lines

This is much faster than reading through hundreds of lines trying to spot the problem. For $n$ lines of code, binary search finds the bug in at most $\lceil \log_2 n \rceil$ steps.

11.5.6 The Rubber Duck Method

📜 Historical Note

The rubber duck debugging method comes from the book The Pragmatic Programmer (Hunt and Thomas, 1999). The idea is simple: explain your code, line by line, to a rubber duck (or any inanimate object). The act of articulating what each line does often reveals the bug, because you are forced to think about what the code actually does rather than what you think it does.

This method works because bugs often arise from a mismatch between your mental model and the code's actual behavior. When you explain the code out loud, you confront every assumption.

You can also use this method with an LLM: paste your code and ask the AI to explain what each line does. Then compare the AI's explanation with your intention. Where they diverge, you have likely found the bug.

11.5.7 Debugging Checklist

When you encounter a bug, work through this checklist in order:

  1. Read the error message. What type of error is it? What line does it point to?
  2. Check the obvious. Typos? Missing semicolons? Mismatched brackets? Wrong variable name?
  3. Add console.log(). Log the values of variables near the error. Are they what you expect?
  4. Isolate the problem. Comment out sections until the error disappears. The last section you uncommented contains the bug.
  5. Check your assumptions. Is width what you think it is? Is the array actually populated? Is the function being called?
  6. Simplify. Replace complex expressions with hardcoded values. If noise(x * 0.01) is suspicious, try 0.5 instead.
  7. Explain it aloud. Walk through the code line by line and say what each line does. The rubber duck method catches assumptions.
  8. Take a break. Sometimes stepping away for 10 minutes reveals what hours of staring cannot. Your subconscious keeps working.
  9. Ask for help. Share the specific error, what you have tried, and the relevant code. An LLM, a peer, or a teaching assistant can offer fresh perspective.
💡 Key Concept

Debugging is a learnable skill. It feels random at first, but with practice you develop intuition for where bugs hide. Keep a mental catalog of bugs you have encountered — the same types recur across projects. Off-by-one errors, uninitialized variables, scope confusion, and timing issues account for the vast majority of creative coding bugs.

Binary search debugging is the most efficient approach. By adding a return at the midpoint, you immediately cut the search space in half. 11.6 Code Review Practices

Code review is the practice of having someone else (or yourself, deliberately) read through code to find problems, suggest improvements, and verify correctness. In professional software development, code review is considered essential. In creative coding, it is equally valuable but often overlooked.

11.6.1 What Code Review Catches

A fresh pair of eyes can spot things the original author misses:

  • Bugs — Logic errors, off-by-one mistakes, unhandled edge cases
  • Readability issues — Unclear variable names, missing comments, overly complex logic
  • Performance problems — Unnecessary computations inside loops, redundant draw calls
  • Missing features — Edge cases not considered (what happens on window resize? when there are zero particles?)
  • Stylistic inconsistencies — Mixed naming conventions, inconsistent formatting

11.6.2 Giving Constructive Feedback

When reviewing a peer's creative code, follow these guidelines:

  1. Start with what works well. "The color palette is beautiful" or "The particle physics feels really natural" builds trust before criticism.
  2. Be specific. "This is confusing" is not helpful. "The variable d on line 42 could be renamed to distToCenter for clarity" is helpful.
  3. Suggest, do not dictate. "Have you considered using a class here?" is better than "You need to use a class here."
  4. Distinguish between bugs and style. "This will crash when the array is empty" is a bug report. "I prefer for...of over index-based loops" is a style preference.
  5. Respect the artist's intent. Creative code is expressive. Do not review it the way you would review banking software.

11.6.3 Self-Review Checklist

Before sharing your code or submitting a project, review it yourself using this checklist:

CategoryCheck
RunsDoes the sketch run without errors in the console?
NamesAre variables and functions named descriptively?
ConstantsAre magic numbers extracted into named constants?
CommentsIs non-obvious code commented with why, not what?
StructureIs code organized into logical sections or functions?
Edge casesWhat happens at canvas edges? With zero items? On window resize?
PerformanceIs the frame rate acceptable? Are there unnecessary computations?
AttributionIs external code and AI usage properly credited?

11.6.4 Reviewing Creative Code

Creative code has dimensions that traditional code review does not consider:

  • Aesthetic quality: Does the visual/sonic output achieve what the artist intended?
  • Interactivity: Is the interaction intuitive? Does it feel responsive?
  • Surprise and delight: Does the piece offer moments of discovery?
  • Conceptual depth: Does the code embody an interesting idea?

These are harder to evaluate than correctness, but they are what make creative coding an art form.

11.7 Performance Optimization

A sketch that runs at 5 frames per second is painful to interact with. Performance matters in creative coding because smoothness is an aesthetic quality — jerky animation looks and feels wrong, regardless of how beautiful each individual frame might be.

11.7.1 Measuring Performance

Before optimizing, measure. p5.js provides frameRate() which returns the current frames per second:

function draw() {
  // Display frame rate in the corner
  if (frameCount % 30 === 0) {
    console.log("FPS:", frameRate().toFixed(1));
  }
  // ... rest of sketch
}

Target 60 fps for smooth interaction. If you are below 30 fps, optimization is needed. If you are at 60 fps, further optimization will not produce visible improvement — do not optimize code that is already fast enough.

💡 Key Concept

Premature optimization is the root of all evil. This famous quote (attributed to Donald Knuth) means: do not spend time optimizing code until you have measured and confirmed that it is actually slow. Write clear, correct code first. Optimize only the parts that are measurably too slow.

11.7.2 Common Performance Bottlenecks

In p5.js, the most common causes of slow performance are:

  1. Too many draw calls. Every ellipse(), rect(), line(), and text() call has overhead. Drawing 10,000 individual shapes per frame is slow.
  2. Expensive math in loops. Functions like sin(), cos(), sqrt(), dist(), and noise() are relatively expensive. Calling them thousands of times per frame adds up.
  3. Creating objects every frame. Allocating new p5.Vector objects, arrays, or colors inside draw() creates garbage collection pressure.
  4. Pixel manipulation without loadPixels(). Using get() and set() is much slower than direct pixel array access.
  5. Unnecessary background() calls with complex rendering. Sometimes you can use layering to avoid redrawing everything.

11.7.3 Optimization Techniques

Pre-compute values. If a value does not change each frame, compute it once in setup():

// Slow: computing every frame
function draw() {
  for (let i = 0; i < 360; i++) {
    let x = centerX + cos(radians(i)) * radius;
    let y = centerY + sin(radians(i)) * radius;
    point(x, y);
  }
}

// Fast: pre-compute the circle points
let circlePoints = [];
function setup() {
  // ...
  for (let i = 0; i < 360; i++) {
    circlePoints.push({
      x: cos(radians(i)) * radius,
      y: sin(radians(i)) * radius
    });
  }
}

function draw() {
  for (let pt of circlePoints) {
    point(centerX + pt.x, centerY + pt.y);
  }
}

Avoid sqrt() for distance comparisons. If you only need to compare distances (not compute the actual distance), compare squared distances instead:

// Slow: uses sqrt via dist()
if (dist(x1, y1, x2, y2) < threshold) { ... }

// Fast: compare squared distances
let dx = x2 - x1;
let dy = y2 - y1;
if (dx * dx + dy * dy < threshold * threshold) { ... }

This avoids the expensive square root operation entirely. For $n$ particles checking distance to each other, this saves $O(n^2)$ square root calls per frame.

Use pixel arrays directly. For pixel manipulation, use the pixels[] array instead of get() and set():

// Slow: get() and set()
for (let x = 0; x < width; x++) {
  for (let y = 0; y < height; y++) {
    let c = get(x, y);
    set(x, y, color(red(c) * 0.5, green(c), blue(c)));
  }
}
updatePixels();

// Fast: direct pixel array access
loadPixels();
for (let i = 0; i < pixels.length; i += 4) {
  pixels[i] *= 0.5;     // red
  // pixels[i+1]         // green (unchanged)
  // pixels[i+2]         // blue (unchanged)
  // pixels[i+3]         // alpha (unchanged)
}
updatePixels();

11.7.4 When to Use Shaders

If your sketch involves per-pixel computation (color effects, Mandelbrot sets, reaction-diffusion, blur effects), consider moving that computation to a GLSL shader. The GPU can process millions of pixels in parallel, making it orders of magnitude faster than CPU-based pixel loops. Chapters on GLSL cover this in detail.

As a rule of thumb: if you are iterating over every pixel on a 400x400 canvas, that is 160,000 iterations per frame. The GPU handles this effortlessly; JavaScript will struggle.

✍ Knowledge Check

Which optimization technique avoids the expensive sqrt() operation when comparing distances?

Since $\sqrt{a} < \sqrt{b}$ if and only if $a < b$ (for non-negative values), you can compare $dx^2 + dy^2$ directly against $threshold^2$ without ever computing the square root. This is a standard optimization in game development and computational geometry, saving significant time when performed thousands of times per frame.

11.8.6 Licensing Your Work

When you share code publicly, consider adding a license. Without a license, others cannot legally reuse your code (even if it is public). Common choices for creative coding:

11.10 Working with LLMs

Large Language Models (ChatGPT, Claude, Copilot, Gemini, etc.) have become powerful tools for creative coding. They can explain concepts, debug errors, suggest approaches, and generate code. Using them is permitted and even encouraged in this course — but their use must be documented, and you should use them wisely.

11.10.1 What LLMs Are Good At

  • Explaining concepts: "What is Perlin noise and how does it differ from random()?"
  • Debugging: Paste an error message and your code, and the LLM can often identify the problem
  • API reference: "What arguments does p5.js noise() take?"
  • Code translation: "Convert this Processing Java code to p5.js"
  • Boilerplate: "Create a basic p5.js sketch with a Particle class that has position, velocity, and update/display methods"
  • Exploring approaches: "What are three different ways to create a trail effect in p5.js?"

11.10.2 What LLMs Are Bad At

  • Aesthetic judgment: An LLM cannot tell you if your sketch is beautiful or interesting
  • Artistic vision: It cannot decide what your project should be
  • Cutting-edge APIs: LLMs may not know the latest p5.js features or may confuse versions
  • Complex debugging: Subtle timing bugs, shader issues, and performance problems are hard for LLMs to diagnose without seeing the running sketch
  • Guaranteed correctness: LLMs can produce plausible-looking code that contains subtle errors
⚠ Common Mistake

Copy-paste without understanding. If you paste LLM-generated code into your sketch without understanding it, you create a fragile foundation. When something goes wrong (and it will), you will not know how to fix it because you do not understand how it works. Always read through AI-generated code line by line and make sure you can explain what each part does.

11.10.3 Prompt Engineering for Creative Code

The quality of an LLM's response depends heavily on your prompt. Here are strategies for getting better results:

  1. Provide context: "I am using p5.js (not Processing Java)" is essential. Include your existing code.
  2. Be specific about what you want: "Write a function" is better than "help me with my project"
  3. Specify constraints: "The function should not use any p5.js functions, only plain JavaScript" or "Keep it under 20 lines"
  4. Ask for explanation: "Explain each line" or "Why did you use atan2 instead of atan?"
  5. Iterate: If the first response is not right, refine your prompt: "That is close, but I need it to work with an array of particles, not just one"

11.10.5 Levels of AI Involvement

It helps to categorize how you used AI:

  1. Conceptual: Asked the AI to explain a concept ("What is Hooke's Law?"). You wrote all the code yourself.
  2. Debugging: Showed the AI an error message and asked for help. You had already written the code.
  3. Code suggestion: Asked the AI to write a specific function. You integrated it into your project.
  4. Substantial generation: Asked the AI to write a large portion of your sketch. You modified the output.

None of these levels is inherently wrong, but they represent increasing reliance on the AI. Your documentation should accurately reflect which level applies. Level 1 and 2 usage demonstrates strong independent engagement; level 4 usage requires particularly thoughtful documentation to demonstrate that learning occurred.

💡 Key Concept

The goal of using AI in this course is to learn faster, not to avoid learning. If you use an LLM to generate code, your responsibility is to understand that code well enough to explain it, modify it, and debug it. The AI is a tutor, not a ghostwriter.

11.11 Summary

This chapter covered the practices that support the creative coding workflow — the meta-skills that make the difference between a frustrating experience and a productive one.

Key takeaways:

  1. Plan lightly but deliberately. Define a Minimum Viable Sketch, break your project into layers, and maintain a work log documenting every session.
  2. Organize your code. Use descriptive names, extract constants, separate concerns into functions, and comment the why, not the what.
  3. Save your iterations. Whether through Git commits or duplicated sketches, always have a way to return to a working version before experimenting.
  4. Debug systematically. Read error messages carefully, use console.log() strategically, isolate problems, and try binary search debugging for large codebases.
  5. Review your code. Use the self-review checklist before submitting. When reviewing others' code, be specific, constructive, and respectful of artistic intent.
  6. Optimize only when needed. Measure with frameRate() first. Focus on reducing draw calls, avoiding expensive math in tight loops, and pre-computing static values.
  7. Document and share. Write header comments, create README files, capture your work in video, and connect with the creative coding community.
  8. Use AI tools honestly and effectively. Document your usage, understand the code you incorporate, and remember that the AI is a tutor, not a substitute for learning.

These practices are not bureaucratic overhead. They are the infrastructure that allows you to spend more time creating and less time fighting your tools, your code, and your own memory.

Test your understanding of project workflow and code review concepts.

✍ Question 1

What is a "Minimum Viable Sketch" (MVS)?

An MVS is the simplest runnable version of your creative coding idea that demonstrates its core concept. It should take no more than one work session to build. The purpose is to test your basic architecture and evaluate the core idea before investing time in complexity, polish, or additional features.

✍ Question 2

Why is Array.splice() a performance concern when removing particles from an array inside a loop?

When you call splice(i, 1) to remove an element, JavaScript must shift every element after index i down by one position. If the array has $n$ elements and you remove $k$ elements, the total work is $O(n \cdot k)$, which in the worst case is $O(n^2)$. The swap-remove or compact-in-place pattern avoids this by overwriting dead elements and truncating the array at the end.

✍ Question 3

When reviewing a peer's creative code, which approach is most constructive?

Constructive code review starts with acknowledging what works, then provides specific and actionable feedback. Saying "the variable d on line 42 could be renamed to distToCenter" is far more helpful than "your naming is bad." In creative coding, it is also important to respect the artist's aesthetic intent — the goal is to help them achieve their vision, not impose yours.
