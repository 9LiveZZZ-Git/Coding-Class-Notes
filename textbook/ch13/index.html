13.1 Algorithmic Thinking

At the heart of every creative coding project lies a fundamental skill that transcends any particular programming language or framework: algorithmic thinking. Whether you are writing a particle system, translating a shader from one platform to another, or designing an interactive installation, you are engaged in the process of breaking problems into steps that a computer can execute. This section explores what algorithmic thinking means and why it matters for artists working with code.

📖 Definition

Algorithm — A finite, unambiguous sequence of well-defined instructions for solving a class of problems or performing a computation. An algorithm takes some input, processes it through a series of steps, and produces an output.

The word "algorithm" comes from the name of the 9th-century Persian mathematician Muhammad ibn Musa al-Khwarizmi, whose work on arithmetic procedures laid the foundation for systematic problem-solving in mathematics. His book Al-Kitab al-Mukhtasar fi Hisab al-Jabr wal-Muqabala (circa 820 CE) also gave us the word "algebra."

Today, algorithms are everywhere: sorting a list of names, finding the shortest route on a map, compressing an image, or generating a fractal pattern on screen. Every time you write a for loop, define a recursive function, or chain together conditional statements, you are implementing an algorithm.

What makes an algorithm different from just "some code"? Three properties:

  1. Finiteness: It must terminate after a finite number of steps. An infinite loop is not an algorithm (though the draw() loop in p5.js deliberately runs indefinitely -- each individual frame computation is finite).
  2. Definiteness: Each step must be precisely defined. "Make it look cool" is not an algorithmic instruction; "multiply each pixel's red channel by 0.8" is.
  3. Effectiveness: Each step must be something a computer can actually execute. "Find the meaning of life" is not effective; "compute the sine of x" is.

13.1.1 The Four Pillars of Computational Thinking

Computer scientists identify four core practices that make up computational thinking, a framework introduced by Jeannette Wing in 2006. These are not just for programmers; they are thinking strategies applicable to any domain, including art and design.

1. Decomposition — Breaking a complex problem into smaller, manageable sub-problems. A large creative coding project such as "build an interactive audio visualizer" becomes tractable when decomposed into: (a) capture audio input, (b) perform FFT analysis, (c) map frequency data to visual parameters, (d) render shapes on screen, (e) handle user interaction.

2. Pattern Recognition — Identifying similarities, trends, or regularities in data or processes. When you notice that many ShaderToy shaders begin with the same coordinate normalization step, you recognize a pattern that can be abstracted and reused.

3. Abstraction — Filtering out unnecessary details to focus on what matters. When you write a function called fadePixel(color, amount), you are abstracting the mathematical operation of linear interpolation into a named concept. Users of that function do not need to know the formula; they just need to know what it does.

4. Algorithm Design — Creating a step-by-step solution once you understand the sub-problems, patterns, and abstractions. This is where you decide the order of operations, choose data structures, and plan for edge cases.

💡 Key Concept

Algorithmic thinking is not about writing code. It is a way of approaching problems that can be expressed in pseudocode, flowcharts, diagrams, or natural language before any code is written. The best programmers spend more time thinking than typing.

13.1.2 Expressing Algorithms

Before writing code, algorithms can be expressed in several ways:

Natural Language: Describe the steps in plain English. "For each pixel in the image, compute its distance from the center. If the distance is less than the radius, color it red; otherwise, color it black." This is easy to understand but can be ambiguous.

Pseudocode: A structured, code-like notation that avoids the syntax details of any particular language:

FOR each pixel (x, y) in the image:
    distance = sqrt((x - centerX)^2 + (y - centerY)^2)
    IF distance < radius:
        setColor(x, y, RED)
    ELSE:
        setColor(x, y, BLACK)

Flowcharts: Visual diagrams showing decision points and flow of control. Useful for complex branching logic.

Mathematical notation: For algorithms rooted in mathematics, formulas can be the most precise expression:

$$\text{color}(x, y) = \begin{cases} \text{red} & \text{if } \sqrt{(x - c_x)^2 + (y - c_y)^2} < r \\ \text{black} & \text{otherwise} \end{cases}$$

📜 Historical Note

The idea that any computable function can be expressed as a mechanical procedure was formalized independently by Alan Turing (Turing machines, 1936) and Alonzo Church (lambda calculus, 1936). The Church-Turing thesis states that these two formulations, and all equivalent ones, capture exactly the same class of computable functions. Every algorithm you write in JavaScript, GLSL, or any other language is a specific instance of this universal idea.

13.1.3 Algorithms in Creative Coding

In the arts, algorithms are not just utilitarian tools; they are expressive media. Consider these examples:

  • Sorting algorithms visualized as animations reveal the hidden beauty of computational logic. Different sorting methods produce distinct visual rhythms and patterns.
  • Pathfinding algorithms like A* can drive the behavior of agents in generative art, creating organic-looking movement patterns.
  • Recursive algorithms naturally produce fractal structures, from branching trees to Sierpinski triangles.
  • Cellular automata like Conway's Game of Life demonstrate how simple algorithmic rules generate complex emergent behavior.

13.1.4 The Role of Constraints

In art, constraints are often generative rather than restrictive. Algorithmic constraints -- the rules you set for your system -- determine the aesthetic space your code can explore:

  • Rule-based drawing: "Every line must connect to the nearest unconnected point." This single rule can generate surprisingly complex and beautiful networks.
  • Parameter spaces: A shader with five tunable parameters creates a five-dimensional space of possible images. Exploring this space algorithmically (e.g., random sampling, gradient descent, or systematic sweeps) can reveal unexpected visual territories.
  • Emergent complexity: Simple rules applied iteratively often produce complex results. This is the central insight of cellular automata, L-systems, and agent-based models.

The artist Sol LeWitt famously created works defined entirely by written instructions (algorithms) that others would execute. His wall drawings demonstrate that the algorithm itself is the artwork, and the physical rendering is one of many possible executions. In his own words: "The idea becomes a machine that makes the art."

Other artists working algorithmically include Vera Molnar, one of the earliest computer art pioneers, who used plotters to explore geometric variations; Manfred Mohr, who has explored hypercube geometry through algorithms since the 1970s; and Casey Reas, co-creator of Processing, whose work demonstrates how simple algorithmic rules generate complex visual systems.

In your own work, think of algorithms not as constraints but as collaborators. The algorithm brings computational power, infinite patience, and mathematical precision. You bring aesthetic judgment, conceptual framing, and the ability to select meaningful parameters from an infinite space of possibilities.

13.1.5 Pseudocode as a Creative Tool

Before diving into code, writing pseudocode helps you think through the structure of your algorithm without worrying about syntax. Here is an example for a simple particle system:

CREATE empty list of particles

EVERY FRAME:
    FOR each particle in list:
        UPDATE position using velocity
        UPDATE velocity using acceleration (gravity)
        DECREASE lifetime by 1
        IF lifetime <= 0: REMOVE particle from list

    IF mouse is pressed:
        CREATE new particle at mouse position
            with random velocity
            and lifetime = 120 frames

    FOR each particle in list:
        SET opacity = lifetime / maxLifetime * 255
        DRAW circle at particle position with opacity

This pseudocode clearly communicates the algorithm without any JavaScript or GLSL syntax. Translating it to code becomes a mechanical process once the logic is clear.

13.2 Problem-Solving Strategies

Professional programmers and creative coders draw on a repertoire of general-purpose strategies when approaching new problems. These strategies are not specific to any language; they are thinking tools that guide you from "I have no idea how to do this" to a working solution.

13.2.1 Divide and Conquer

The divide-and-conquer strategy splits a problem into independent sub-problems, solves each one separately, and combines the results. This is the logic behind merge sort, the fast Fourier transform (FFT), and many graphics algorithms.

Creative coding example: Suppose you want to create a mosaic effect on an image. Divide the canvas into a grid of tiles. For each tile, compute the average color of the pixels within it. Fill the tile with that color. Each tile is an independent sub-problem, making the algorithm easy to parallelize.

$$T(n) = 2T\left(\frac{n}{2}\right) + O(n) \implies T(n) = O(n \log n)$$

This recurrence relation describes the time complexity of merge sort and other divide-and-conquer algorithms. The $O(n \log n)$ result is dramatically faster than $O(n^2)$ for large inputs.

13.2.2 Greedy Approach

A greedy algorithm makes the locally optimal choice at each step, hoping this leads to a globally optimal solution. Greedy algorithms are fast and simple but do not always produce the best answer.

Creative coding example: Packing circles into a canvas. A greedy approach places each new circle at the position where it fits best (e.g., largest possible radius without overlapping existing circles). This gives a reasonable result quickly, even though it may not be the mathematically optimal packing.

Another example: when choosing colors for a generative palette, a greedy algorithm might select the next color to maximize the minimum distance from all previously chosen colors in color space. This tends to produce well-distributed, visually distinct palettes.

13.2.3 Brute Force

The brute force strategy tries all possible solutions and picks the best one. It is the simplest approach and always correct, but often too slow for large inputs.

Creative coding example: Finding the closest pair of points among $n$ agents. The brute force approach checks all $\binom{n}{2} = \frac{n(n-1)}{2}$ pairs, which is $O(n^2)$. For 100 agents, that is 4,950 comparisons per frame -- manageable. For 10,000 agents, it is nearly 50 million -- too slow for real-time rendering at 60 fps.

Despite its inefficiency, brute force has an important role in creative coding:

  • It serves as a baseline implementation that you can verify is correct before optimizing.
  • For small inputs, it is often fast enough and much simpler to debug.
  • It helps you understand the problem structure before applying more sophisticated techniques.
  • In shader code, brute force loops are sometimes the only option because shaders lack dynamic data structures like hash maps.

13.2.4 Working Backwards

Working backwards starts from the desired output and reasons about what inputs or steps would produce it. This is especially useful when debugging shaders.

Creative coding example: You see a black screen when running a shader. Working backwards: the output is gl_FragColor. Is it being set? Yes, to vec4(col, 1.0). Is col always zero? Check how col is computed. Trace back through the math to find where the logic fails.

Working backwards is also useful for design: "I want the final image to show a mandala pattern. What does that require? Rotational symmetry. How do I achieve that? Polar coordinates with mod() on the angle. What pattern fills each segment? A combination of sine waves in the radial direction." Starting from the desired result and working backwards to the implementation often reveals the clearest path forward.

13.2.5 Analogical Reasoning

Analogical reasoning applies a solution from a known problem to a new, structurally similar problem. Much of shader porting relies on this: "I know how to normalize coordinates in ShaderToy, so I can adapt that logic for p5.js."

Creative coding example: You want to create a ripple effect on a 2D canvas. You recall that the wave equation produces circular waves in physics. By analogy, you compute the distance from each pixel to the wave source and use a sine function with that distance as input: sin(dist * freq - time * speed). The physics analogy gives you both the approach and the intuition for how the parameters affect the result.

Analogical reasoning is particularly powerful in creative coding because visual patterns often recur across different contexts. A flocking algorithm is analogous to magnetic field lines. A reaction-diffusion system produces patterns analogous to animal markings. A spring-mass simulation behaves analogously to musical vibration. Recognizing these connections lets you transfer solutions across domains.

13.2.6 Incremental Development

Incremental development builds a solution in small, testable steps rather than trying to write the complete solution at once. This is especially valuable in creative coding where visual feedback is immediate.

Example workflow for building a shader:

  1. Start with a flat color to confirm the shader compiles and renders.
  2. Add UV coordinate visualization (red = x, green = y) to verify coordinate space.
  3. Add a single geometric element (e.g., a circle using length(uv - center)).
  4. Add animation by incorporating u_time.
  5. Add color using sine-based palettes.
  6. Add complexity: more shapes, domain repetition, noise.
  7. Polish: anti-aliasing, vignette, color grading.

At each step, you can see the result immediately and catch errors before they compound.

13.2.7 The Egg Drop Puzzle: A Case Study

To illustrate algorithmic thinking in action, consider a classic puzzle: You have 2 eggs and a 100-story building. You want to find the highest safe floor -- the highest floor from which you can drop an egg without it breaking. How do you minimize the worst-case number of drops?

Brute force approach (1 egg): Start at floor 1, test each floor sequentially. Worst case: 100 drops. This is $O(n)$.

Divide and conquer (2 eggs): With 2 eggs, you can be smarter. Drop the first egg from floor 50. If it breaks, test floors 1-49 with the second egg (linear search). If it survives, test floors 51-100 with the same strategy. Worst case: 50 drops.

Optimal strategy: Use decreasing intervals. Drop the first egg from floor 14, then 27, then 39, etc. If it breaks, use the second egg to search the interval. With this strategy, the worst case is 14 drops.

The optimal solution uses dynamic programming. Let $dp[n][e]$ be the minimum worst-case drops for $n$ floors with $e$ eggs:

$$dp[n][e] = 1 + \min_{k=1}^{n} \max(dp[k-1][e-1], \; dp[n-k][e])$$

The key insight: when you drop from floor $k$, either the egg breaks (search $k-1$ floors below with $e-1$ eggs) or it survives (search $n-k$ floors above with $e$ eggs). You take the max (worst case) and minimize over all choices of $k$.

💡 Key Concept

No single strategy works for every problem. Expert programmers unconsciously switch between strategies as they work. Start with the simplest approach (often brute force), verify it works, then optimize if needed.

13.3 Algorithm Complexity

When writing code that runs in real time -- as all creative coding does -- performance matters. Algorithm complexity gives us a language for describing how an algorithm's running time or memory usage grows as the input size increases.

13.3.1 Big-O Notation (Intuitive)

Big-O notation describes the worst-case growth rate of an algorithm's resource usage. It answers the question: "As my input doubles in size, how much more work does my algorithm do?" This is a simplification that ignores constant factors and lower-order terms, focusing only on the dominant behavior for large inputs.

Think of it like measuring travel time: we say "it is a 3-hour drive" rather than "it is 2 hours, 47 minutes, and 23 seconds." The exact number depends on traffic, speed, and route, but the order of magnitude -- hours, not minutes or days -- is what matters for planning.

📖 Definition

Big-O Notation — We write $f(n) = O(g(n))$ if there exist constants $c > 0$ and $n_0$ such that $f(n) \le c \cdot g(n)$ for all $n \ge n_0$. Informally, $f$ grows no faster than $g$ for large inputs.

13.3.2 Why Complexity Matters for Real-Time Graphics

At 60 frames per second, your draw() function has approximately 16.7 milliseconds to complete all its work. On a modern machine, JavaScript can execute roughly 100 million simple operations per second. That gives you about 1.5 million operations per frame.

Consider a flocking simulation with $n$ boids where each boid checks every other boid for neighbors:

  • $n = 100$: $100^2 = 10{,}000$ checks per frame. No problem.
  • $n = 1{,}000$: $1{,}000^2 = 1{,}000{,}000$ checks per frame. Getting tight.
  • $n = 10{,}000$: $10{,}000^2 = 100{,}000{,}000$ checks per frame. Far too slow.

13.3.3 Spatial Data Structures

The solution to the quadratic neighbor-search problem is to use spatial data structures that organize objects by their position, allowing you to skip distant objects entirely.

Uniform Grid: Divide the canvas into a grid of cells. Each object registers in the cell containing its position. To find neighbors of an object, only check objects in the same cell and adjacent cells. If the cell size matches the neighbor radius, this reduces the check from $O(n^2)$ to roughly $O(n)$ for uniformly distributed objects.

// Simple spatial grid in JavaScript
function buildGrid(particles, cellSize) {
    let grid = {};
    for (let p of particles) {
        let key = Math.floor(p.x / cellSize)
                + "," + Math.floor(p.y / cellSize);
        if (!grid[key]) grid[key] = [];
        grid[key].push(p);
    }
    return grid;
}

function getNeighbors(grid, x, y, cellSize) {
    let neighbors = [];
    let cx = Math.floor(x / cellSize);
    let cy = Math.floor(y / cellSize);
    // Check 3x3 neighborhood of cells
    for (let dx = -1; dx <= 1; dx++) {
        for (let dy = -1; dy <= 1; dy++) {
            let key = (cx + dx) + "," + (cy + dy);
            if (grid[key]) {
                neighbors.push(...grid[key]);
            }
        }
    }
    return neighbors;
}

Quadtree: A tree data structure that recursively subdivides 2D space into four quadrants. When a region contains too many objects, it splits. Neighbor queries walk the tree and only visit relevant branches. Average complexity: $O(n \log n)$.

The choice between a uniform grid and a quadtree depends on how your objects are distributed:

  • Uniform grid: Best when objects are roughly evenly distributed. Simple to implement. Cell size should match the interaction radius.
  • Quadtree: Best when objects are unevenly distributed (e.g., clustered in some regions). Adapts automatically to local density. More complex to implement.

13.3.4 Amortized Analysis and Frame Budgets

Not all frames need to do the same amount of work. Amortized analysis considers the average cost over a sequence of operations. In creative coding, you can spread expensive computations across multiple frames:

// Instead of rebuilding the spatial grid every frame:
let frameCount = 0;

function draw() {
    if (frameCount % 3 === 0) {
        rebuildSpatialGrid(); // expensive, but only every 3rd frame
    }
    updateParticles();  // uses the grid from the last rebuild
    renderParticles();
    frameCount++;
}

This technique is called temporal amortization. The grid is slightly stale on 2 out of 3 frames, but for most visual applications, this is imperceptible. Other examples of temporal amortization in creative coding:

  • Rebuilding a quadtree only every few frames
  • Performing collision detection on a subset of objects each frame, cycling through all objects over several frames
  • Updating distant particles at a lower rate than nearby ones (level-of-detail for computation)
  • Rendering complex backgrounds to a texture once and reusing the texture for many frames

13.3.5 Space Complexity

In addition to time complexity, space complexity describes how much memory an algorithm uses. This matters for creative coding because:

  • A full-screen framebuffer at 1920x1080 with 4 color channels (RGBA) uses $1920 \times 1080 \times 4 = 8{,}294{,}400$ bytes, or about 8 MB.
  • Storing 10,000 particle objects with 10 properties each requires $10{,}000 \times 10 \times 8 = 800{,}000$ bytes (about 800 KB for 64-bit floats).
  • A quadtree for those particles uses additional memory for the tree structure itself.

On modern hardware, memory is rarely the bottleneck for creative coding projects, but it can matter on mobile devices or when working with very large datasets (e.g., high-resolution video textures, large point clouds, or volumetric data). When memory is constrained, consider using lower-precision data types (e.g., Uint8Array instead of Float64Array) or streaming data rather than loading everything at once.

✍ Knowledge Check

A function loops through an array of $n$ elements, and for each element, performs a binary search on a sorted array of $n$ elements. What is the overall time complexity?

The outer loop runs $n$ times. Each iteration performs a binary search, which is $O(\log n)$. Multiplying gives $O(n \log n)$. This is the same complexity class as efficient sorting algorithms like merge sort.

13.4 Understanding ShaderToy

ShaderToy is an online platform created by Inigo Quilez and Pol Jeremias that hosts thousands of GLSL fragment shaders. It provides a simplified environment where you write only the fragment shader; the platform handles the vertex shader, full-screen quad setup, uniform passing, and rendering loop automatically.

📜 Historical Note

ShaderToy launched in 2013 and quickly became the de facto platform for shader experimentation and sharing. Inigo Quilez, one of its creators, is also known for pioneering signed distance function (SDF) techniques used extensively in ray marching. His website iquilezles.org is an essential resource for shader artists.

13.4.1 The mainImage Signature

Every ShaderToy shader defines a single entry function:

void mainImage(out vec4 fragColor, in vec2 fragCoord) {
    // your shader code here
    fragColor = vec4(1.0, 0.0, 0.0, 1.0); // output: red
}

The two parameters are:

  • fragCoord (input) — The pixel coordinate of the current fragment. Origin is at the bottom-left, with $x$ increasing rightward and $y$ increasing upward. Units are in pixels (e.g., for a 800x600 canvas, $x$ ranges from 0.5 to 799.5).
  • fragColor (output) — The RGBA color you assign to this pixel. Each component ranges from 0.0 to 1.0.

13.4.3 Common ShaderToy Patterns

Nearly every ShaderToy shader begins with coordinate normalization:

// Normalize to [0, 1]
vec2 uv = fragCoord / iResolution.xy;

// Or normalize to [-1, 1] with aspect ratio correction
vec2 uv = (2.0 * fragCoord - iResolution.xy) / iResolution.y;

The second form centers the origin, accounts for non-square canvases, and is preferred for effects that should look the same regardless of window size.

13.4.4 Understanding iMouse

The iMouse uniform deserves special attention because its behavior is more complex than the other uniforms:

  • iMouse.xy — Current mouse position (in pixels) while the mouse button is held down. When the button is released, these values freeze at the last click position.
  • iMouse.zw — The position where the mouse button was last pressed down. The sign of z indicates whether the button is currently held (positive) or released (negative).

This means iMouse is not a continuous mouse tracker. It only updates during click-drag interactions. Many shaders use a pattern like:

// Only use mouse if button has been clicked
if (iMouse.z > 0.0) {
    vec2 mouseUV = iMouse.xy / iResolution.xy;
    // use mouseUV for interaction
}

13.4.5 Texture Inputs (iChannel)

ShaderToy supports four texture input channels (iChannel0 through iChannel3), each of which can be:

  • An image: A static texture loaded from ShaderToy's built-in library (noise textures, organic textures, abstract patterns).
  • A video: A webcam feed or video file.
  • Audio: Microphone input or a music track, represented as a texture where the x-axis is frequency and the y-axis encodes the waveform (row 0) and FFT spectrum (row 1).
  • A buffer: The output of another render pass (Buffer A, B, C, or D), enabling feedback and multi-pass effects.
  • A cubemap: An environment map for reflection effects.

When porting to p5.js, each texture channel becomes a uniform sampler that you pass via setUniform().

13.4.6 ShaderToy vs. GLSL Standards

It is important to understand that ShaderToy uses a slightly non-standard GLSL environment:

  • ShaderToy wraps your code inside its own main() function. You never write void main() or set gl_FragColor directly -- the platform handles that using the mainImage signature.
  • ShaderToy supports GLSL ES 3.0 features (like texture() instead of texture2D()) that may not work in WebGL 1.0 environments.
  • Some ShaderToy shaders use features specific to the platform (like buffer self-feedback) that require careful translation.

13.5 Porting ShaderToy to p5.js

Translating a ShaderToy shader to run in p5.js is a practical skill that gives you access to thousands of visual effects for your projects. The process is systematic: once you learn the translation rules, you can port most single-pass shaders in minutes.

13.5.1 Step-by-Step Translation Guide

Step 1: Set up the p5.js vertex shader. Every p5.js WEBGL shader needs a vertex shader. Use this standard boilerplate for full-screen effects:

attribute vec3 aPosition;
attribute vec2 aTexCoord;
varying vec2 vTexCoord;

void main() {
    vTexCoord = aTexCoord;
    vec4 positionVec4 = vec4(aPosition, 1.0);
    positionVec4.xy = positionVec4.xy * 2.0 - 1.0;
    gl_Position = positionVec4;
}

Step 2: Create the fragment shader header. Declare your own uniforms to replace ShaderToy's built-ins:

precision mediump float;
varying vec2 vTexCoord;
uniform vec2 u_resolution;
uniform float u_time;
uniform vec2 u_mouse;
// Add more uniforms as needed

Step 3: Reconstruct fragCoord. ShaderToy provides fragCoord as a parameter. In p5.js, compute it from the varying texture coordinate:

void main() {
    vec2 fragCoord = vTexCoord * u_resolution;
    fragCoord.y = u_resolution.y - fragCoord.y; // flip Y axis
    // ... rest of shader code ...
}

13.5.4 Common Pitfalls

⚠ Common Mistake

Forgetting to flip Y. ShaderToy's origin is bottom-left; p5.js/canvas origin is top-left. If your ported shader appears upside down, add fragCoord.y = u_resolution.y - fragCoord.y; after computing fragCoord.

Other common issues when porting:

  • iResolution.xy vs u_resolution: ShaderToy's iResolution is a vec3. If you declared u_resolution as vec2, remove any .xy swizzles on it. Similarly, iResolution.x becomes u_resolution.x.
  • texture vs texture2D: WebGL 1 (used by p5.js) requires texture2D(). WebGL 2 and ShaderToy use texture(). This is one of the most common compilation errors when porting.
  • Multipass shaders: ShaderToy's Buffer A/B/C/D tabs require createFramebuffer() in p5.js. Each buffer becomes a separate render target with its own shader.
  • Missing precision statement: Always include precision mediump float; at the top of your fragment shader. Desktop browsers may tolerate its absence, but mobile GPUs require it.
  • Integer division: GLSL has no implicit type conversion. Use 2.0 instead of 2 when dividing floats. The expression 1/2 evaluates to 0 in GLSL, not 0.5.
  • Uninitialized variables: GLSL does not guarantee default values. Always initialize your variables, especially vec3 col = vec3(0.0); before accumulating into it.
  • Loop limits: Some WebGL implementations require loop bounds to be compile-time constants. If a ShaderToy shader uses a uniform as a loop limit, you may need to use a constant instead and break early.
  • Out parameters: ShaderToy's mainImage uses an out parameter for the output color. In the p5.js main() function, you write to gl_FragColor instead. There is no equivalent of out parameters in the main function.
📜 Historical Note

The difference between texture() and texture2D() stems from the evolution of the GLSL specification. In GLSL ES 1.0 (WebGL 1), there were separate functions for each texture type: texture2D(), textureCube(), etc. GLSL ES 3.0 (WebGL 2) unified these into a single overloaded texture() function. ShaderToy uses the newer convention.

13.5.5 Multipass Shaders

ShaderToy supports multiple render passes through its Buffer system. Each buffer (A through D) has its own shader that writes to an offscreen texture. The "Image" tab is the final pass that composites everything to screen.

In p5.js, you replicate this pattern with createFramebuffer():

let bufferA, shaderA, shaderFinal;

function setup() {
    createCanvas(800, 600, WEBGL);
    bufferA = createFramebuffer({ format: FLOAT });
    shaderA = createShader(vertSrc, fragSrcA);
    shaderFinal = createShader(vertSrc, fragSrcFinal);
}

function draw() {
    // Pass 1: render to buffer A
    bufferA.begin();
    shader(shaderA);
    shaderA.setUniform("u_prev", bufferA); // self-feedback
    shaderA.setUniform("u_resolution", [width, height]);
    shaderA.setUniform("u_time", millis() / 1000.0);
    rect(0, 0, width, height);
    bufferA.end();

    // Final pass: render to screen
    shader(shaderFinal);
    shaderFinal.setUniform("u_buffer", bufferA);
    shaderFinal.setUniform("u_resolution", [width, height]);
    rect(0, 0, width, height);
}

13.5.6 Debugging Ported Shaders

When a ported shader does not work as expected, use these systematic debugging techniques:

1. Color debugging: Output intermediate values as colors to visualize what the shader is computing:

// Debug: visualize UV coordinates
gl_FragColor = vec4(uv, 0.0, 1.0); // red = x, green = y

// Debug: visualize a distance field
gl_FragColor = vec4(vec3(d), 1.0); // grayscale = distance

// Debug: visualize normals
gl_FragColor = vec4(normal * 0.5 + 0.5, 1.0); // map [-1,1] to [0,1]

2. Isolate the problem: Comment out most of the shader and add features back one at a time until the error appears.

3. Check coordinate ranges: Print or visualize the range of your UV coordinates. Are they in [0,1]? [-1,1]? Something unexpected?

4. Verify uniforms: Add a visual indicator that uniforms are being received. For example, draw a circle at the mouse position to confirm u_mouse is being passed correctly.

5. Compare side by side: Keep the original ShaderToy shader running in one browser tab and your ported version in another. Spot differences frame by frame.

💡 Key Concept

GLSL does not have console.log(). The only way to inspect values in a shader is to output them as colors. This is a fundamental debugging technique that you will use constantly. Red channel for x-values, green channel for y-values, and blue channel for whatever else you want to inspect.

13.6 Fading and Blending Techniques

Fading and blending are fundamental techniques in creative coding. They enable motion trails, ghosting effects, smooth transitions, painting tools, and audio visualizers. This section covers the major approaches and the math behind them.

13.6.1 Alpha Fading

The simplest fading technique: draw a semi-transparent rectangle over the entire canvas each frame. Old content gradually disappears as new layers of semi-transparent background accumulate.

$$\text{brightness after } n \text{ frames} = (1 - \alpha)^n$$

where $\alpha$ is the opacity of the background overlay (e.g., 0.05 for a slow fade). After $n$ frames, content that has not been redrawn has been multiplied by $(1 - \alpha)^n$. For $\alpha = 0.05$:

  • After 10 frames: $(0.95)^{10} \approx 0.60$ (60% brightness)
  • After 50 frames: $(0.95)^{50} \approx 0.08$ (8% brightness)
  • After 100 frames: $(0.95)^{100} \approx 0.006$ (essentially gone)
p5.js Fading Techniques Comparison

13.6.2 Blend Modes

Blend modes define how a new color $S$ (source) combines with an existing color $D$ (destination) on the canvas. Here are the most important ones for creative coding:

Normal (Alpha) Blending:

$$C_{\text{out}} = \alpha_S \cdot C_S + (1 - \alpha_S) \cdot C_D$$

Additive Blending:

$$C_{\text{out}} = C_S + C_D$$

Colors accumulate and become brighter. Used for glowing effects, fire, light beams. Clamps at 1.0 (white).

Multiplicative Blending:

$$C_{\text{out}} = C_S \times C_D$$

Colors darken. White is the identity (does nothing). Used for shadows, darkening filters.

Screen Blending:

$$C_{\text{out}} = 1 - (1 - C_S)(1 - C_D)$$

The complement of multiply. Colors lighten. Black is the identity. Used for highlights and soft light effects.

13.6.3 GPU Fading with Framebuffer Feedback

The most powerful fading technique uses a framebuffer (also called an FBO or render target) that reads its own previous contents each frame. The algorithm is:

  1. Bind the framebuffer as the render target.
  2. Draw the framebuffer's previous contents back into itself with reduced opacity (e.g., 96%).
  3. Draw new content on top at full opacity.
  4. Unbind the framebuffer and display it to the screen.

Over time, old content fades exponentially. After $n$ frames at opacity $\alpha$:

$$\text{brightness}(n) = \alpha^n$$

At $\alpha = 0.96$: content is half-faded after about 17 frames ($0.96^{17} \approx 0.50$) and essentially invisible after 100 frames ($0.96^{100} \approx 0.017$).

In p5.js WEBGL mode, the framebuffer approach looks like:

let fbo;

function setup() {
    createCanvas(600, 600, WEBGL);
    fbo = createFramebuffer();
}

function draw() {
    background(0);
    fbo.begin();

    // Draw previous frame with slight fade
    push();
    tint(255, 245); // 245/255 ≈ 0.96 opacity
    texture(fbo);
    noStroke();
    plane(width, height);
    pop();

    // Draw new content at full opacity
    if (mouseIsPressed) {
        fill(255, 100, 50);
        noStroke();
        circle(mouseX - width/2, mouseY - height/2, 30);
    }

    fbo.end();

    // Display the framebuffer
    texture(fbo);
    noStroke();
    plane(width, height);
}

13.6.4 Custom Shader Fading

For fading toward a specific color (not just black), use a shader with mix():

precision mediump float;
varying vec2 vTexCoord;
uniform sampler2D u_prev;
uniform vec3 u_bgColor;
uniform float u_fadeAmount;

void main() {
    vec4 prev = texture2D(u_prev, vTexCoord);
    vec3 faded = mix(prev.rgb, u_bgColor, u_fadeAmount);
    gl_FragColor = vec4(faded, 1.0);
}

Each frame, every pixel moves u_fadeAmount (e.g., 0.02 or 2%) closer to the background color. This allows fading to any color, not just black.

13.6.5 CPU-Based Fading with Aging Data Structures

The CPU approach stores each visual element as an object with a lifetime. This gives you per-element control over fade rate, color, and behavior:

let strokes = [];

function draw() {
    background(20);

    if (mouseIsPressed) {
        strokes.push({
            x: mouseX, y: mouseY,
            size: random(10, 40),
            r: random(100, 255), g: random(50, 200), b: random(150, 255),
            age: 0, maxAge: 120
        });
    }

    for (let i = strokes.length - 1; i >= 0; i--) {
        let s = strokes[i];
        s.age++;
        let life = 1.0 - s.age / s.maxAge;
        fill(s.r, s.g, s.b, life * 255);
        noStroke();
        circle(s.x, s.y, s.size * life);
        if (s.age >= s.maxAge) strokes.splice(i, 1);
    }
}

Advantages of CPU fading: Works in 2D mode. Each element can have its own fade rate, color, and behavior. Easy to understand and debug.

Disadvantages: Performance degrades with many objects. Each object takes memory. Removing from the middle of an array is $O(n)$.

13.6.6 Lerp-Based Fading

Linear interpolation (lerp) is one of the most useful functions in creative coding. The formula is:

$$\text{lerp}(a, b, t) = a \cdot (1 - t) + b \cdot t = a + t \cdot (b - a)$$

where $t \in [0, 1]$ controls the blend: $t = 0$ gives $a$, $t = 1$ gives $b$, and values in between give a weighted average. In GLSL, this function is called mix().

For fading, lerp can be applied per-pixel each frame to move colors toward a target:

// GLSL: fade each pixel toward background color
vec3 faded = mix(currentColor, backgroundColor, 0.02);
// Each frame, the pixel moves 2% closer to the background

Lerp is also fundamental to many other operations:

  • Color transitions: mix(colorA, colorB, t) where $t$ varies over time.
  • Smooth movement: position = lerp(position, target, 0.1) creates an easing effect where the object moves quickly at first and slows as it approaches the target.
  • Cross-fading: Blend between two textures or effects by varying $t$ from 0 to 1.
  • Terrain blending: Mix between grass and rock textures based on slope angle.
📐 Mathematical Property

Repeatedly applying x = lerp(x, target, t) each frame produces exponential easing. After $n$ frames: $x_n = \text{target} - (\text{target} - x_0)(1-t)^n$. The value approaches the target asymptotically -- quickly at first, then ever more slowly. This is the same math behind the frame accumulation fade.

13.6.7 Smoothstep: Non-Linear Interpolation

While lerp()/mix() creates linear transitions, smoothstep() creates smooth, ease-in-ease-out transitions:

$$\text{smoothstep}(a, b, t) = \begin{cases} 0 & t \le a \\ 3s^2 - 2s^3 & a < t < b, \; s = \frac{t-a}{b-a} \\ 1 & t \ge b \end{cases}$$

The cubic polynomial $3s^2 - 2s^3$ (called the Hermite interpolant) has zero derivatives at both endpoints, creating smooth transitions. In shaders, smoothstep is used for:

  • Anti-aliased edges on shapes
  • Smooth color transitions
  • Vignette effects (darkening edges of the frame)
  • Fog and distance fading
// Vignette effect
vec2 uv = gl_FragCoord.xy / u_resolution.xy;
float vignette = smoothstep(0.8, 0.3, length(uv - 0.5));
col *= vignette;

13.6.8 Frame Accumulation / Motion Blur

Motion blur is achieved by blending the current frame with previous frames. The framebuffer feedback technique draws the previous frame's contents back with slight transparency, then draws new content on top. Over multiple frames, moving objects leave smooth trails.

The mathematical model: if $F_n$ is the framebuffer at frame $n$ and $C_n$ is the new content drawn at frame $n$:

$$F_n = \alpha \cdot F_{n-1} + C_n$$

where $\alpha \in [0, 1)$ is the decay factor (e.g., 0.96). This is an exponential moving average, giving more weight to recent frames.

Expanding the recurrence shows why this is called exponential decay:

$$F_n = C_n + \alpha C_{n-1} + \alpha^2 C_{n-2} + \alpha^3 C_{n-3} + \ldots$$

Each previous frame's contribution is weighted by a decreasing power of $\alpha$. The most recent frame has weight 1, the frame before has weight $\alpha$, two frames ago has weight $\alpha^2$, and so on.

13.6.9 Choosing Fade Parameters

The fade rate $\alpha$ controls the visual character of the trail:

0.96Long (~100 frames)Smooth trails, light echoesAudio visualizers, particle trails

0.80Short (~15 frames)Subtle afterimageResponsive UI effects
0.50Very short (~3 frames)Brief flashHit effects, notifications

You can calculate the number of frames for content to fade to a given threshold:

$$n = \frac{\ln(\text{threshold})}{\ln(\alpha)}$$

For example, to find when content fades to 1% brightness with $\alpha = 0.96$: $n = \frac{\ln(0.01)}{\ln(0.96)} = \frac{-4.605}{-0.0408} \approx 113$ frames, or about 1.9 seconds at 60 fps.

In additive blending, what happens when you draw a red circle (1, 0, 0) on top of a green circle (0, 1, 0)?

Additive blending computes $C_{\text{out}} = C_S + C_D = (1,0,0) + (0,1,0) = (1,1,0)$, which is yellow. This is why additive blending creates bright, glowing effects where colors overlap -- they accumulate toward white.

13.7.1 Ray Marching Basics

Ray marching is a technique for rendering 3D scenes defined by mathematical functions rather than polygon meshes. Instead of tracing rays to find triangle intersections (as in traditional ray tracing), ray marching steps along each ray in small increments, querying a signed distance function (SDF) at each step.

An SDF returns the distance from a point to the nearest surface. If the distance is positive, the point is outside all objects. If negative, it is inside an object. If zero, it is on the surface.

The basic algorithm:

  1. For each pixel, cast a ray from the camera through the pixel into the scene.
  2. Start at the ray origin. Query the SDF to get the distance $d$ to the nearest surface.
  3. Advance along the ray by $d$ (this is safe because nothing is closer than $d$).
  4. Repeat until $d$ is very small (hit) or the ray has traveled very far (miss).
📖 Definition

Signed Distance Function (SDF) — A function $f: \mathbb{R}^3 \to \mathbb{R}$ that returns the shortest distance from a point $p$ to the nearest surface. The sign indicates inside (negative) vs. outside (positive). The zero-level set $\{p : f(p) = 0\}$ defines the surface.

The key insight of ray marching is called sphere tracing: at each step, the SDF tells you the maximum safe distance to travel. Since nothing is closer than $d$, advancing by exactly $d$ guarantees you will not pass through any surface. This makes the algorithm adaptive -- it takes large steps in empty space and tiny steps near surfaces.

Common SDF primitives:

// Sphere centered at origin with radius r
float sdSphere(vec3 p, float r) {
    return length(p) - r;
}

// Box centered at origin with half-size b
float sdBox(vec3 p, vec3 b) {
    vec3 q = abs(p) - b;
    return length(max(q, 0.0)) + min(max(q.x, max(q.y, q.z)), 0.0);
}

// Plane at y = 0
float sdPlane(vec3 p) {
    return p.y;
}

// Torus (ring) with major radius R and minor radius r
float sdTorus(vec3 p, vec2 t) {
    vec2 q = vec2(length(p.xz) - t.x, p.y);
    return length(q) - t.y;
}

// Cylinder along the Y axis
float sdCylinder(vec3 p, float r, float h) {
    vec2 d = abs(vec2(length(p.xz), p.y)) - vec2(r, h);
    return min(max(d.x, d.y), 0.0) + length(max(d, 0.0));
}

Computing surface normals for lighting requires evaluating the SDF at six nearby points (the central difference gradient):

$$\vec{n}(p) = \text{normalize}\left(\begin{bmatrix} f(p + \epsilon\hat{x}) - f(p - \epsilon\hat{x}) \\ f(p + \epsilon\hat{y}) - f(p - \epsilon\hat{y}) \\ f(p + \epsilon\hat{z}) - f(p - \epsilon\hat{z}) \end{bmatrix}\right)$$

where $\epsilon$ is a small value (typically 0.001). This is expensive (6 SDF evaluations), but it works for any SDF without needing to derive the gradient analytically.

13.7.2 Noise in Shaders

Noise functions add organic randomness to shaders. Unlike random() which gives uncorrelated values, noise functions produce smooth, continuous randomness where nearby inputs give similar outputs. This property, called coherence, is what makes noise look natural rather than static-like.

The most common types in shaders:

  • Value noise: Random values at integer grid points, interpolated between them. Simple but can show grid-aligned artifacts.
  • Gradient noise (Perlin): Random gradients at grid points, producing smoother results than value noise. Invented by Ken Perlin in 1983 for the movie Tron, for which he won an Academy Award for Technical Achievement.
  • Simplex noise: A more efficient variant of Perlin noise with fewer directional artifacts. Also invented by Perlin (2001). Uses a simplex grid (triangles in 2D, tetrahedra in 3D) instead of a square/cubic grid.
  • Worley/Cellular noise: Computes distance to randomly placed feature points. Creates cell-like, organic patterns. Used for water caustics, stone textures, and biological patterns.
  • Fractal Brownian Motion (fBm): Multiple octaves of noise layered together at different frequencies and amplitudes. Each octave doubles the frequency and halves the amplitude: $$\text{fBm}(p) = \sum_{i=0}^{n-1} \frac{\text{noise}(p \cdot 2^i)}{2^i}$$ The result has detail at multiple scales, similar to natural textures like clouds, terrain, and wood grain.

Noise can be used for many effects in shaders:

  • Terrain generation: fBm noise as a heightfield
  • Cloud rendering: Volume rendering with noise density fields
  • Texture variation: Break up uniform surfaces with subtle noise
  • Animation: Use time as an additional noise dimension for smooth, organic motion
  • Distortion: Add noise to UV coordinates before sampling a pattern

A simple hash-based noise function commonly used in shaders:

// Hash function for pseudo-random values
float hash(vec2 p) {
    return fract(sin(dot(p, vec2(127.1, 311.7))) * 43758.5453);
}

// 2D value noise
float noise(vec2 p) {
    vec2 i = floor(p);
    vec2 f = fract(p);
    f = f * f * (3.0 - 2.0 * f); // smooth interpolation

    float a = hash(i);
    float b = hash(i + vec2(1.0, 0.0));
    float c = hash(i + vec2(0.0, 1.0));
    float d = hash(i + vec2(1.0, 1.0));

    return mix(mix(a, b, f.x), mix(c, d, f.x), f.y);
}

// Fractal Brownian Motion
float fbm(vec2 p) {
    float sum = 0.0;
    float amp = 0.5;
    for (int i = 0; i < 5; i++) {
        sum += amp * noise(p);
        p *= 2.0;
        amp *= 0.5;
    }
    return sum;
}
⚠ Common Mistake

The hash function fract(sin(dot(p, vec2(127.1, 311.7))) * 43758.5453) is widely used but not mathematically rigorous. It produces acceptable randomness for visual effects but should not be used for cryptography or scientific simulation. On some GPUs, precision issues can cause visible patterns at extreme coordinates.

13.7.3 Domain Repetition

Domain repetition uses the modulo operation to tile a pattern infinitely. By applying mod() to the input coordinates before evaluating an SDF or pattern, a single shape is repeated across space:

// Repeat space with period s
vec3 rep(vec3 p, float s) {
    return mod(p + s * 0.5, s) - s * 0.5;
}

This transforms a single sphere into an infinite grid of spheres with just one line of code. The mod() function wraps the coordinates so that every point in space maps back to a single "cell," and the subtraction centers the origin within each cell.

Domain repetition is computationally free -- it does not create more geometry. The SDF is still evaluated once per ray march step, but the coordinate space wraps around. You can also repeat in only one or two dimensions:

// Repeat only in X and Z (creates columns)
p.xz = mod(p.xz + s * 0.5, s) - s * 0.5;

// Repeat only in X (creates walls)
p.x = mod(p.x + s * 0.5, s) - s * 0.5;

Combining domain repetition with animated offsets creates mesmerizing patterns of infinite movement.

13.7.4 Polar Coordinates in Shaders

Converting Cartesian coordinates $(x, y)$ to polar coordinates $(r, \theta)$ enables radial and rotational effects:

$$r = \sqrt{x^2 + y^2}, \quad \theta = \text{atan2}(y, x)$$

vec2 toPolar(vec2 uv) {
    return vec2(length(uv), atan(uv.y, uv.x));
}

Polar coordinates are the foundation of many shader effects:

  • Radial gradients: Color based on $r$ (distance from center).
  • Spiral patterns: Use $r + \theta$ or $r + k\theta$ as input to a periodic function.
  • Rotation: Add a time-varying offset to $\theta$ to rotate the entire pattern.
  • Flower patterns: Use $r = a + b \cdot \cos(n\theta)$ to define petal-like shapes.
  • Kaleidoscope: Mirror the angle within repeated segments (see below).

A common technique converts back from polar to Cartesian after modifying the coordinates:

// Convert to polar, modify, convert back
float r = length(uv);
float a = atan(uv.y, uv.x);
a += u_time; // rotate the pattern
vec2 modifiedUV = r * vec2(cos(a), sin(a));

13.7.5 Kaleidoscope Effect

A kaleidoscope shader divides the angular coordinate into segments and mirrors alternating ones, creating symmetric patterns. The real-world kaleidoscope works by using angled mirrors to create multiple reflections. We can simulate this digitally using modular arithmetic on the angle coordinate.

The key operation uses mod() on the angle and abs() for the mirror:

float segments = 6.0;
float segAngle = 3.14159 * 2.0 / segments;
angle = mod(angle, segAngle);
angle = abs(angle - segAngle * 0.5);
GLSL Kaleidoscope Shader

Try changing the number of segments in the shader above. Powers of 2 (4, 8, 16) and multiples of 3 (6, 9, 12) produce particularly pleasing symmetries. You can also try adding time to the angle before the kaleidoscope operation to create a rotating pattern, or multiplying the angle after mirroring to create nested repetitions.

💡 Key Concept

The kaleidoscope technique is a special case of a more general principle: symmetry through coordinate manipulation. By modifying the UV coordinates before computing the pattern, you can create any symmetry group. Mirroring on one axis gives bilateral symmetry. Modular arithmetic on the angle gives rotational symmetry. Combining both gives dihedral symmetry (the symmetry of regular polygons).

13.7.6 Color Palettes in Shaders

Creating beautiful colors in shaders is an art in itself. The most common approach uses cosine-based palettes, invented by Inigo Quilez:

$$\text{color}(t) = a + b \cdot \cos(2\pi(c \cdot t + d))$$

where $a$, $b$, $c$, and $d$ are vec3 parameters that control the palette, and $t$ is a scalar (usually derived from distance, angle, or time).

vec3 palette(float t) {
    vec3 a = vec3(0.5, 0.5, 0.5);
    vec3 b = vec3(0.5, 0.5, 0.5);
    vec3 c = vec3(1.0, 1.0, 1.0);
    vec3 d = vec3(0.263, 0.416, 0.557);
    return a + b * cos(6.28318 * (c * t + d));
}

By changing the four parameter vectors, you can create an enormous variety of color schemes. Quilez provides an interactive palette generator at iquilezles.org/articles/palettes/.

13.7.8 Anti-Aliasing in Shaders

Hard edges in shaders appear jagged (aliased). The smoothstep() function is the standard tool for anti-aliasing:

// Hard edge (aliased)
float circle = step(0.5, length(uv));

// Soft edge (anti-aliased)
float pixelSize = 1.0 / u_resolution.y;
float circle = smoothstep(0.5, 0.5 - pixelSize * 2.0, length(uv));

The width of the transition zone should be proportional to the pixel size. Using fwidth() (available in some GLSL versions) provides an even more robust approach by computing the screen-space rate of change of the distance field.

13.8 Code Attribution and Ethics

As artists who work with code, we stand on the shoulders of an enormous open-source community. ShaderToy alone hosts over 80,000 public shaders, each representing someone's creative and technical effort. Using this shared knowledge responsibly requires understanding attribution, licenses, and academic honesty.

13.8.1 Why Attribution Matters

Attribution serves multiple purposes:

  • Ethical: Recognizing the labor and creativity of others is a moral obligation. Code is creative expression, and using it without credit is a form of plagiarism.
  • Legal: Most code has a license that specifies conditions for use. Ignoring the license can have legal consequences.
  • Professional: The creative coding community is small. Your reputation depends on being known as someone who gives credit generously.
  • Educational: Attribution creates a trail that helps others (and your future self) understand where code came from and how it was modified.

13.8.2 Academic Honesty in Computing Arts

In an academic setting, the stakes for attribution are especially high. Course policies on code sharing vary, but the general principles are universal:

  • Individual work is individual: If an assignment requires individual submission, the code you write must be your own. You may discuss concepts and approaches with classmates, but the implementation must be independently written.
  • Collaboration must be declared: If you worked with a partner or received significant help, say so explicitly in your submission comments. This includes casual conversations where a classmate explained an approach or helped you debug a problem.
  • External code must be attributed: Any code from tutorials, Stack Overflow, GitHub, ShaderToy, or LLMs must be clearly marked with its source.
  • Understanding is required: Submitting code you do not understand, regardless of its source, is academically dishonest. If you use external code, be prepared to explain how it works.

The MAT 200C course syllabus states: "You may not share your code publicly... If you use code from an external source, you must cite it explicitly." This applies to ShaderToy shaders, tutorial code, and LLM-generated code equally.

⚠ Common Mistake

Students sometimes believe that modifying code "enough" makes it their own. This is not true. If your work is derived from someone else's code, you must credit the source regardless of how much you changed. The test is not "how different is my code?" but rather "did I start from scratch or build on existing work?"

13.8.3 How to Attribute Code

A good attribution comment includes:

// Based on "Warp Speed" by Dave Hoskins
// Original: https://www.shadertoy.com/view/Msl3WH
// License: CC BY-NC-SA 3.0
// Modified: changed color palette, added mouse interaction
// Date accessed: March 2026

For code from tutorials, Stack Overflow, or documentation:

// Perlin noise implementation adapted from
// The Book of Shaders, Ch. 11 by Patricio Gonzalez Vivo
// https://thebookofshaders.com/11/
// Modified to accept 3D input

For ShaderToy shaders specifically, always include:

  1. The shader name and author
  2. The ShaderToy URL
  3. The license (shown on the shader's page)
  4. A brief description of what you modified
  5. The date you accessed the shader

13.8.5 Software Licenses

Software libraries and frameworks use different licenses than creative works:

  • MIT License: The most permissive. Do anything with the code; just keep the copyright notice. Used by p5.js (via LGPL for the library itself) and many npm packages.
  • Apache 2.0: Similar to MIT, but includes explicit patent protection. Used by TensorFlow and many Google projects.
  • LGPL: You can use the library in any project, but modifications to the library itself must be shared. p5.js uses this license.
  • GPL: Strong copyleft. If you use GPL code, your entire project must also be GPL. Used by Linux and WordPress.
  • No License: All rights reserved by default. You cannot legally use, copy, or modify the code without explicit permission.
⚠ Common Mistake

Code on GitHub without a LICENSE file is not open source. "Public" and "open source" are not the same thing. If you see code on GitHub or ShaderToy with no license, you should contact the author for permission before using it.

13.8.6 Crediting LLM-Generated Code

Large Language Models (LLMs) such as ChatGPT, Claude, and Copilot are increasingly used in creative coding. Per the MAT 200C course policy, LLM-generated code must be clearly marked:

// --- BEGIN LLM-GENERATED CODE ---

// Prompt: "Write a GLSL function that creates a 2D rotation matrix"

    float c = cos(angle);

// --- END LLM-GENERATED CODE ---

Key principles for using LLM-generated code:

  1. Mark the code clearly with begin/end comments.
  2. Name the LLM and version used.
  3. Include your prompts so others can understand the generation context.
  4. Understand every line. You are responsible for the code, including bugs.
  5. Do not submit LLM-generated code as your own original work.

13.8.7 The Golden Rule of Attribution

💡 Key Concept

When in doubt, over-attribute. No one has ever been penalized for giving too much credit. If you are unsure whether something counts as borrowing, cite it anyway. Attribution is free, and it builds trust and community.

✍ Knowledge Check

The "NC" (NonCommercial) restriction prohibits commercial use. You cannot sell a product that incorporates CC BY-NC-SA code. All other options are permitted: academic use with credit, modification with same license (SA), and portfolio display with attribution (BY).

13.9 Building a Shader Portfolio

As a creative coder, your portfolio is your most important professional asset. A well-curated collection of shader work demonstrates both technical skill and artistic vision.

13.9.1 Documenting Your Process

The creative process is as valuable as the final result. Document your work as you go:

  • Screenshots at each major stage: Show the evolution from early experiments to polished output.
  • Code comments: Explain not just what the code does but why you made each design choice.
  • Version control: Use Git to track changes. Your commit history tells the story of your creative process.
  • Sketches and notes: Scan or photograph hand-drawn diagrams, math derivations, and conceptual sketches.

13.9.2 Capturing Shader Output

Screenshots and video recordings are essential for sharing shader work:

  • Screenshots: Use browser dev tools or a screenshot tool. Capture at high resolution (at least 1920x1080).
  • Video: Use OBS Studio or screen recording tools. Record at 60 fps to preserve the smoothness of animation.
  • GIFs: Short loops work well for social media. Tools like FFmpeg can convert video to optimized GIFs.
  • Live demos: Host interactive versions on GitHub Pages, Netlify, or the p5.js web editor.

When capturing shader output, remember that compression artifacts (especially in JPEG and low-bitrate video) can destroy the subtle gradients and fine details that make shaders beautiful. Use PNG for screenshots and high-bitrate settings (or lossless codecs) for video recording.

13.9.3 Writing Artist Statements

An artist statement for a computational artwork should address:

  • Concept: What idea or question drove the work?
  • Process: What tools, techniques, and algorithms did you use?
  • Context: How does this work relate to existing art, research, or cultural themes?
  • Technical details: Mention specific techniques (ray marching, fBm noise, domain repetition) that will interest technically-minded viewers.

13.9.4 Online Presence

Platforms for sharing shader and creative coding work:

  • ShaderToy — The primary community for GLSL shaders.
  • p5.js Web Editor — Share complete p5.js sketches with one link.
  • OpenProcessing — Community gallery for Processing and p5.js sketches.

13.9.5 Presenting Technical Work to Non-Technical Audiences

When showing shader work to curators, galleries, or general audiences, focus on the experience rather than the implementation:

  • Lead with the visual or interactive experience. Let the artwork speak first.
  • Describe the concept in accessible terms: "This piece uses mathematical functions to simulate infinite space" rather than "This uses domain repetition with SDFs."
  • Connect to broader themes: nature, perception, infinity, emergence, order and chaos.
  • Save technical details for an appendix, GitHub README, or technical blog post.

  • Mentor others: Answer questions in forums, participate in creative coding communities (Reddit r/creativecoding, creative coding Discord servers, Processing forums).

Your portfolio is not just a collection of finished works. It is a living document that shows your growth, your thinking process, and your engagement with the creative coding community. Quality and thoughtfulness matter more than quantity.

✍ Knowledge Check

A p5.js sketch needs to find the nearest neighbor for each of 5,000 particles every frame. Which approach would best maintain real-time performance?

With 5,000 particles, brute force requires $5000^2 = 25{,}000{,}000$ comparisons per frame, which is too slow for 60 fps. A spatial data structure (grid or quadtree) reduces this to roughly $O(n)$ for uniformly distributed particles, making real-time performance achievable. Reducing frame rate is a workaround, not a solution.

13.10 Summary

This chapter brought together several important threads that connect algorithmic thinking to creative practice. Let us review the key takeaways from each section:

  • Algorithmic thinking (Section 13.1) is a structured approach to problem-solving built on four pillars: decomposition, pattern recognition, abstraction, and algorithm design. These skills transcend any specific programming language and are equally applicable to visual art, music, and interactive design.
  • Problem-solving strategies (Section 13.2) -- divide and conquer, greedy, brute force, working backwards, analogical reasoning, and incremental development -- give you a versatile toolbox for tackling unfamiliar coding challenges. The egg drop puzzle illustrated how different strategies yield dramatically different solutions to the same problem.
  • Algorithm complexity (Section 13.3) and Big-O notation help you predict whether your code will run fast enough for real-time graphics at 60 fps. Spatial data structures like uniform grids and quadtrees can turn $O(n^2)$ problems into manageable $O(n)$ or $O(n \log n)$ ones. Remember: profile first, optimize second.
  • ShaderToy (Section 13.4) provides a simplified GLSL environment with built-in uniforms (iResolution, iTime, iMouse, iChannel0-3) and the mainImage entry point. Understanding these conventions, including the bottom-left origin and the iMouse behavior, is the first step to porting shaders.
  • Porting shaders from ShaderToy to p5.js (Section 13.5) follows a systematic seven-step recipe: add declarations, change the entry point, compute fragCoord, flip Y, rename uniforms, change the output variable, and replace texture functions. Debugging techniques like color visualization are essential when things go wrong.
  • Fading and blending techniques (Section 13.6) include alpha fading, additive/multiplicative/screen blending, lerp-based fading, and framebuffer feedback. GPU approaches handle full-screen effects efficiently; CPU approaches with aging data structures give per-element control. Combining both provides maximum flexibility.
  • Advanced shader techniques (Section 13.7) -- ray marching, noise (value, Perlin, fBm), domain repetition, polar coordinates, kaleidoscope effects, cosine color palettes, SDF operations, and anti-aliasing -- are building blocks that can be combined to create complex, beautiful visual experiences.
  • Code attribution and ethics (Section 13.8) are professional obligations. Always credit sources, understand licenses (MIT, GPL, CC BY, CC BY-NC-SA), and clearly mark LLM-generated code with begin/end markers, the model name, and your prompts.
  • Building a shader portfolio (Section 13.9) requires documenting your process, capturing high-quality screenshots and video, writing thoughtful artist statements, and engaging with the creative coding community.
💡 Key Concept

The topics in this chapter are deeply interconnected. Algorithmic thinking guides how you decompose a shader porting task. Complexity analysis tells you whether your fading technique will run in real time. Attribution ethics ensure the community that produces ShaderToy shaders continues to thrive. And your portfolio documents the intersection of all these skills.

Test your understanding of the concepts covered in this chapter.

✍ Question 1

Which of the four pillars of computational thinking involves filtering out unnecessary details to focus on what is essential?

Abstraction is the process of ignoring irrelevant details to focus on what matters. When you create a function like fadePixel(), you abstract away the math so users can think at a higher level. Decomposition breaks problems into parts; pattern recognition finds regularities; algorithm design creates step-by-step solutions.

✍ Question 2

When porting a ShaderToy shader to p5.js, what must you do to account for the different coordinate system origins?

ShaderToy places the origin at the bottom-left (Y increases upward), while p5.js/canvas places the origin at the top-left (Y increases downward). Subtracting the y coordinate from the resolution height flips it to match the expected orientation.

✍ Question 3

What is the GLSL equivalent of the JavaScript lerp(a, b, t) function?

In GLSL, mix(a, b, t) computes $a \cdot (1-t) + b \cdot t$, which is linear interpolation. smoothstep() performs a smooth (Hermite) interpolation with clamping, which is a different operation. The other options are not built-in GLSL functions.

✍ Question 4

In a ray marching shader, what does the signed distance function (SDF) return?

An SDF returns the distance from a point in space to the nearest surface. If positive, the point is outside all objects. If negative, it is inside. If zero, it is on the surface. This distance tells the ray marcher how far it can safely step along the ray without missing any geometry.
