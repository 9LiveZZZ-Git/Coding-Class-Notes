4.1 Introduction

In the previous chapter, we built a Mandelbrot set renderer in GLSL. The iteration rule was straightforward:

$$z_{n+1} = z_n^2 + c$$

with $z_0 = 0$ and $c$ determined by each pixel's position in the complex plane. For each pixel, we iterated until $|z|$ exceeded an escape radius or we hit a maximum iteration count. Escaped pixels were colored by iteration count; non-escaping pixels were colored black.

This gave us the iconic Mandelbrot image. But we can push much further in two directions:

  1. Different formulas: What happens if we replace $z^2$ with $z^3$, or $\sin(z)$, or $e^z$? Each function has its own geometry, its own symmetries, and produces its own fractal landscape.
  2. Better coloring: Instead of just counting iterations (which produces visible banding), we can extract rich statistics from the iteration process and map them to color in sophisticated ways.

This chapter explores both directions. We will also introduce Newton fractals as a fundamentally different kind of fractal iteration, cover the technique of frame feedback in GLSL for cellular automata and trail effects, and connect our computational work to the history of early abstract animation.

💡 Key Concept

Fractal exploration is computational alchemy. We do not have a complete theory that predicts what each formula will look like. We must compute it and see. The artist's role is to explore parameter spaces, discover visual phenomena, and develop aesthetic judgment about coloring and composition.

By the end of this chapter, you will be able to render fractals from arbitrary complex functions, color them with smooth gradients and perceptually uniform palettes, implement Newton's method as a fractal generator, and use frame feedback for cellular automata and visual effects.

📜 Historical Note

The idea of iterating complex functions and studying the resulting geometry dates to the early 20th century. Pierre Fatou (1878--1929) and Gaston Julia (1893--1978) independently studied the iteration of rational functions in the complex plane around 1918--1920. However, without computers, they could only prove theorems about the structure -- they never saw the images. It was not until Benoit Mandelbrot used IBM computers in 1978--1980 to render these sets that their extraordinary visual beauty was revealed.

4.1.1 A Map of the Territory

The landscape we are about to explore can be organized along two axes. The formula axis determines the shape of the fractal: $z^2 + c$ gives the Mandelbrot, $z^3 + c$ gives the cubic Multibrot, $\sin(z) + c$ gives a periodic tree-like structure, and so on. The coloring axis determines how we render the fractal: raw iteration count, smooth coloring, orbit statistics, cosine palettes, HSV mapping, or CIELAB perceptual gradients. Any formula can be combined with any coloring method, giving us a vast creative space to explore.

We will also encounter two entirely different types of fractal iteration:

  • Escape-time fractals (Mandelbrot, Julia sets): We iterate until $|z|$ exceeds a threshold, then color by how long it took to escape.
  • Convergence fractals (Newton fractals): We iterate until $z$ converges to a fixed point, then color by which fixed point it reached.

Both types produce intricate fractal boundaries, but for fundamentally different reasons. Understanding both gives us a much richer toolkit for creative computation.

4.2 Complex Arithmetic in GLSL

GLSL has no native complex number type. This is because GLSL was designed for 3D graphics, where complex numbers are rarely needed. However, we can represent a complex number $a + bi$ as a vec2 where .x is the real part $a$ and .y is the imaginary part $b$. We then build complex operations as functions, creating a small but powerful library.

4.2.1 Basic Operations

Addition and subtraction work automatically because GLSL's vec2 addition is component-wise:

vec2 z1 = vec2(1.0, 2.0); // 1 + 2i
vec2 z2 = vec2(3.0, 4.0); // 3 + 4i
vec2 sum = z1 + z2;        // 4 + 6i  (correct!)

Scalar multiplication also works automatically:

vec2 scaled = z1 * 3.0; // 3 + 6i (correct!)
⚠ Common Mistake

Complex multiplication does NOT work with *, because vec2 * vec2 in GLSL performs component-wise multiplication, not complex multiplication. z1 * z2 gives vec2(3, 8), but the correct result of $(1+2i)(3+4i) = -5 + 10i$ is vec2(-5, 10).

We implement complex multiplication from the formula $(a+bi)(c+di) = (ac-bd) + (ad+bc)i$:

vec2 cx_mul(vec2 a, vec2 b) {
  return vec2(
    a.x * b.x - a.y * b.y,  // real: ac - bd
    a.x * b.y + a.y * b.x   // imag: ad + bc
  );
}

4.2.2 Division and Conjugate

Complex division uses the conjugate trick. To divide $\frac{a+bi}{c+di}$, we multiply numerator and denominator by the conjugate $c - di$:

$$\frac{a+bi}{c+di} = \frac{(a+bi)(c-di)}{(c+di)(c-di)} = \frac{(ac+bd) + (bc-ad)i}{c^2+d^2}$$

vec2 cx_div(vec2 a, vec2 b) {
  float denom = dot(b, b); // b.x^2 + b.y^2
  return vec2(
    a.x*b.x + a.y*b.y,
    a.y*b.x - a.x*b.y
  ) / denom;
}

Note the use of dot(b, b) instead of b.x*b.x + b.y*b.y. GLSL's built-in dot() function computes the dot product, which for a vec2 is exactly $x^2 + y^2$ -- the squared modulus of the complex number. This is a recurring pattern: we use dot(z, z) whenever we need $|z|^2$.

The conjugate $\overline{a+bi} = a - bi$ simply negates the imaginary part:

vec2 cx_conj(vec2 z) { return vec2(z.x, -z.y); }

4.2.3 Powers and Transcendental Functions

Complex squaring is a special case of multiplication, slightly optimized:

vec2 cx_sqr(vec2 z) {
  return vec2(z.x*z.x - z.y*z.y, 2.0*z.x*z.y);
}

Complex power uses the polar form $z = re^{i\theta}$ where $r = |z|$ and $\theta = \arg(z)$. Then $z^n = r^n e^{in\theta} = r^n(\cos(n\theta) + i\sin(n\theta))$:

vec2 cx_pow(vec2 z, float n) {
  float r = length(z);
  float theta = atan(z.y, z.x);
  return pow(r, n) * vec2(cos(n*theta), sin(n*theta));
}

Complex exponential follows from Euler's formula $e^{a+bi} = e^a(\cos b + i\sin b)$:

vec2 cx_exp(vec2 z) {
  return exp(z.x) * vec2(cos(z.y), sin(z.y));
}

Complex sine: Using the identity $\sin(a+bi) = \sin a \cosh b + i\cos a \sinh b$:

float cosh_f(float x) { return (exp(x) + exp(-x)) * 0.5; }
float sinh_f(float x) { return (exp(x) - exp(-x)) * 0.5; }

vec2 cx_sin(vec2 z) {
  return vec2(sin(z.x)*cosh_f(z.y), cos(z.x)*sinh_f(z.y));
}

Complex cosine: $\cos(a+bi) = \cos a \cosh b - i\sin a \sinh b$:

vec2 cx_cos(vec2 z) {
  return vec2(cos(z.x)*cosh_f(z.y), -sin(z.x)*sinh_f(z.y));
}

Complex logarithm (principal branch): $\ln(z) = \ln|z| + i\arg(z)$:

vec2 cx_log(vec2 z) {
  return vec2(log(length(z)), atan(z.y, z.x));
}
📖 Definition

Complex Number Library -- The complete set of functions (cx_mul, cx_div, cx_sqr, cx_pow, cx_exp, cx_sin, cx_cos, cx_log, cx_conj) that you paste at the top of your GLSL fragment shader to enable complex arithmetic. Every fractal shader in this chapter uses this library.

4.2.4 Why Build Our Own Library?

You might wonder why GLSL does not simply include complex number support. The answer is that GLSL is designed for 3D graphics -- transforming vertices, computing lighting, and texturing surfaces. Complex numbers are not part of that world. But because GLSL provides fast vec2 operations, it turns out to be an excellent platform for complex number computation. Our library adds the missing operations.

The key insight is that complex arithmetic maps perfectly onto 2D vector operations with just a few modifications. Addition is identical. Multiplication requires the "FOIL" cross terms. Division requires the conjugate trick. And transcendental functions use Euler's formula to bridge the gap between exponential and trigonometric representations.

💡 Key Concept

Euler's Formula is the bridge between exponential and trigonometric representations of complex numbers: $e^{i\theta} = \cos\theta + i\sin\theta$. This single identity underlies our implementations of cx_exp, cx_pow, cx_sin, and cx_cos. It is arguably the most important formula in all of mathematics for connecting algebra, geometry, and analysis.

4.2.5 The Modulus and Argument

Every complex number $z = a + bi$ can also be written in polar form as $z = re^{i\theta}$, where:

  • $r = |z| = \sqrt{a^2 + b^2}$ is the modulus (distance from the origin)
  • $\theta = \arg(z) = \text{atan2}(b, a)$ is the argument (angle from the positive real axis)

In GLSL, the modulus is length(z) and the argument is atan(z.y, z.x). The squared modulus dot(z, z) (which equals $a^2 + b^2$) is what we use for escape testing, because it avoids the expensive square root.

GLSL's * operator on vec2 performs component-wise multiplication: vec2(a,b) * vec2(c,d) = vec2(a*c, b*d). But complex multiplication requires the cross terms: $(a+bi)(c+di) = (ac - bd) + (ad + bc)i$. We must implement this manually with cx_mul().

4.3 Alternative Fractal Formulas

The Mandelbrot iteration $z_{n+1} = z^2 + c$ is just one of infinitely many possible iteration rules. By changing the formula, we explore entirely different fractal landscapes. Each function's geometry -- its symmetries, periodicities, and growth rates -- shapes the fractal it produces.

4.3.1 The Multibrot Family: $z^n + c$

Replacing the exponent 2 with higher powers gives us the Multibrot family. The general formula is $z_{n+1} = z_n^p + c$, where $p$ is any real number greater than 1. The name "Multibrot" is a portmanteau of "multi" and "Mandelbrot."

The mathematics behind the symmetry is elegant. In polar form, if $z = re^{i\theta}$, then $z^n = r^n e^{in\theta}$. The function $z^n$ maps the complex plane by multiplying angles by $n$ and raising magnitudes to the $n$-th power. This angular multiplication creates $n$ identical "petals" around the origin. The Multibrot set inherits $(n-1)$-fold rotational symmetry from the critical point structure of the iteration.

$z^3 + c$ (Cubic Mandelbrot): This produces a fractal with 2-fold rotational symmetry (the standard Mandelbrot has only mirror symmetry). The main body has fewer, larger bulbs, and the filaments branch in threes.

// Cubic Mandelbrot iteration
z = cx_mul(cx_sqr(z), z) + c;

$z^4 + c$ (Quartic Mandelbrot): Produces 3-fold rotational symmetry with even more elaborate branching patterns. The general rule is that $z^n + c$ produces $(n-1)$-fold symmetry.

// Quartic Mandelbrot iteration
vec2 z2 = cx_sqr(z);
z = cx_sqr(z2) + c;

4.3.2 Transcendental Fractals

Moving beyond polynomial iteration formulas opens up a vast new territory. Transcendental functions -- those involving $e^z$, $\sin(z)$, $\cos(z)$, and their compositions -- produce fractals with qualitatively different character. While polynomial fractals have a finite number of symmetry axes, transcendental fractals can have infinite periodicity.

$e^z + c$ (Exponential): The exponential function has period $2\pi i$ in the imaginary direction (since $e^{z + 2\pi i} = e^z$), creating ribbon-like, vertically periodic structures that look like flowing curtains or jellyfish tentacles.

z = cx_exp(z) + c;
// Scale: c = vPosition * 3.0 - vec2(1.0, 0.0)
// Escape: dot(z,z) > 100.0

$\sin(z) + c$ (Sine): The sine function has period $2\pi$ in the real direction, producing horizontally periodic, tree-like branching structures. The hyperbolic sine in the imaginary component causes rapid growth in the imaginary direction, making the fractal expand vertically much faster than horizontally.

z = cx_sin(z) + c;
// Scale: c = vPosition * 4.0
// Escape: dot(z,z) > 50.0

The sine fractal is particularly beautiful because it combines the organic, branching quality of tree-like structures with the mathematical precision of periodic repetition. The horizontal periodicity means that the same pattern repeats every $2\pi$ units along the real axis, like an infinite wallpaper.

$\cos(z) + c$ (Cosine): Very similar to the sine fractal, but shifted by $\pi/2$. The cosine fractal has a different center of symmetry and produces slightly different (but equally beautiful) structures.

z = cx_cos(z) + c;
// Scale: c = vPosition * 4.0
// Escape: dot(z,z) > 50.0

4.3.3 The Burning Ship Fractal

The Burning Ship fractal uses a simple but powerful modification to the Mandelbrot formula:

$$z_{n+1} = (|\text{Re}(z_n)| + i|\text{Im}(z_n)|)^2 + c$$

In code, this means taking abs(z) before squaring. This seemingly small change -- reflecting each component to be positive before the squaring step -- has dramatic consequences. The absolute value operation is not holomorphic (not complex-differentiable), which breaks the smooth, flowing character of the standard Mandelbrot. The result is jagged, angular structures with sharp edges and asymmetric features.

When viewed with the imaginary axis flipped (upside down), the main body of the fractal resembles a ship engulfed in flames, giving it its evocative name. The "mast" and "rigging" structures are particularly striking.

In GLSL:

// Burning Ship iteration
z = abs(z); // take absolute value of both components
z = vec2(z.x*z.x - z.y*z.y, 2.0*z.x*z.y) + c; // then square and add c
📜 Historical Note

The Burning Ship fractal was first described by Michael Michelitsch and Otto E. Rossler in 1992. It is sometimes called the "Mandelbrot's evil twin" because it shares the same basic structure but with a more aggressive, asymmetric appearance.

4.3.4 More Exotic Formulas

$i^z + c$: Using $i^z = e^{z \cdot \ln(i)} = e^{z \cdot i\pi/2}$:

vec2 lni = vec2(0.0, 3.14159265 / 2.0); // ln(i) = i*pi/2
z = cx_exp(cx_mul(z, lni)) + c;

This formula creates spiraling, galaxy-like structures because multiplying by $i\pi/2$ rotates and scales the complex plane before exponentiation.

$\sin(z)^2 + c$: Combines the periodicity of sine with the doubling behavior of squaring:

vec2 sv = cx_sin(z);
z = cx_mul(sv, sv) + c;

$c^z + i$: Here the base varies per pixel rather than the exponent:

z = cx_exp(cx_mul(z, cx_log(c))) + vec2(0.0, 1.0);

Each of these formulas produces a unique fractal landscape. The creative process involves trying different formulas, adjusting viewing parameters, and developing an intuition for which mathematical properties (periodicity, symmetry, growth rate) produce which visual effects.

💡 Key Concept

Mandelbrot vs. Julia sets: For any iteration formula $z_{n+1} = f(z_n, c)$, there are two different fractals you can render. In the Mandelbrot view, $z_0 = 0$ is fixed and $c$ varies per pixel. In the Julia view, $c$ is fixed and $z_0$ varies per pixel. Every point in the Mandelbrot set corresponds to a connected Julia set; every point outside corresponds to a disconnected (dust-like) Julia set.

4.4 Coloring Fractals: Beyond Iteration Count

4.4.1 The Banding Problem

With just the integer iteration count, we get a limited palette. Points inside the set all get the same color (black). Points outside get one of $N$ possible colors, where $N$ is the maximum iteration count. This produces visible banding -- sharp concentric boundaries where the iteration count changes by 1.

These bands follow the equipotential lines of the Mandelbrot set -- curves where every point has the same escape time. While mathematically interesting (they reveal the structure of the escape-time function), they are aesthetically undesirable because they create jarring visual artifacts that distract from the fractal's underlying geometry.

The rest of this section presents three approaches to eliminating banding: smooth coloring (the most common), histogram equalization (for maximum dynamic range), and distance estimation (for boundary-focused rendering).

4.4.2 Smooth Coloring (Normalized Iteration Count)

The key insight is that when $z$ escapes, its magnitude $|z|$ contains information about how "close" it was to escaping on the previous iteration. We can use this to compute a fractional iteration count.

📐 Theorem: Smooth Iteration Count

For the iteration $z_{n+1} = z_n^2 + c$, when $z$ escapes after $n$ iterations with $|z_n| = R$, the smooth iteration count is:

$$n_{\text{smooth}} = n + 1 - \frac{\log(\log|z_n|)}{\log 2}$$

This formula produces a continuous value that varies smoothly from pixel to pixel, eliminating banding.

Derivation: After $n$ iterations, $|z_n| \approx R^{2^{n-n_0}}$ for some reference iteration $n_0$. Taking logs twice:

$$\log\log|z_n| \approx (n - n_0)\log 2 + \log\log R$$

Solving for the fractional part gives us the correction term $1 - \frac{\log(\log|z_n|)}{\log 2}$. For this to work well, use a larger escape radius (e.g., $|z|^2 > 256$ instead of $|z|^2 > 4$), which gives the formula more "room" to be accurate.

More concretely: if you use escape radius $R = 2$, then $\log|z_n|$ might only be slightly above $\log 2 \approx 0.693$, and the double logarithm is very sensitive to this value, producing noisy corrections. With $R = 16$ (i.e., $|z|^2 > 256$), $\log|z_n|$ is at least $\log 16 \approx 2.77$, and the correction is much smoother.

For the general power $z^p + c$, the smooth coloring formula becomes:

$$n_{\text{smooth}} = n + 1 - \frac{\log(\log|z_n|)}{\log p}$$

// GLSL smooth iteration count
float smoothIter = float(iterations);
if (iterations < MAX_ITER) {
  float logZn = log(dot(z, z)) / 2.0; // log|z|
  float nu = log(logZn / log(2.0)) / log(2.0);
  smoothIter = float(iterations) + 1.0 - nu;
}

4.4.3 Why the Smooth Coloring Formula Works

To understand the smooth coloring formula more deeply, consider what happens as $z$ escapes. After many iterations, the growth of $|z|$ is dominated by the squaring operation. If $|z_n|$ is large, then $|z_{n+1}| \approx |z_n|^2$ (the $+c$ term becomes negligible).

Starting from some iteration $n_0$ where $|z_{n_0}| = R_0$ (some large value), we have:

$$|z_{n_0 + k}| \approx R_0^{2^k}$$

Taking the logarithm twice:

$$\log\log|z_{n_0+k}| \approx k \cdot \log 2 + \log\log R_0$$

This shows that $\log\log|z|$ increases linearly with iteration count (with slope $\log 2$), which is exactly what we need for smooth interpolation. Solving for the "fractional iteration count" gives us:

$$k_{\text{frac}} = \frac{\log\log|z_n| - \log\log R_0}{\log 2}$$

The formula $n + 1 - \frac{\log\log|z_n|}{\log 2}$ is a simplified version that absorbs the constant terms. The $+1$ accounts for the fact that we detect escape one iteration after the "true" escape point.

4.4.4 Histogram Coloring

Histogram coloring (also called histogram equalization) assigns colors so that equal areas of the image get equal ranges of the color palette. Without this technique, most of the palette might be "spent" on a narrow band of iteration counts that covers a large area, while the interesting boundary region gets only a few colors.

The algorithm is a two-pass technique:

  1. First pass: Compute the iteration count for every pixel and build a histogram $H[k]$ counting how many pixels have iteration count $k$.
  2. Compute cumulative distribution: For each iteration count $k$, compute $C[k] = \sum_{j=0}^{k} H[j] / N_{\text{total}}$, where $N_{\text{total}}$ is the total number of escaped pixels. This gives the fraction of pixels with iteration count $\leq k$.
  3. Second pass: For each pixel with iteration count $k$, its normalized color value is $t = C[k]$. This spreads colors evenly across the image area.

Mathematically, if we define the probability distribution $p(k) = H[k] / N_{\text{total}}$, then the cumulative distribution function $C(k) = \sum_{j=0}^{k} p(j)$ maps iteration counts to a uniform distribution over $[0, 1]$. This is the discrete analogue of the probability integral transform.

This technique is harder to implement in a single-pass GLSL shader (it requires computing the histogram over all pixels, which needs texture readback or compute shaders), but produces excellent results in CPU-based renderers or multi-pass GPU implementations. It ensures that no color is "wasted" on iteration count ranges that cover very few pixels, maximizing the visual information content of the image.

4.4.5 Distance Estimation Coloring

The distance estimator computes an approximation of the distance from a point $c$ to the boundary of the Mandelbrot set. During iteration, we also track the derivative $z'_n$ of the orbit:

$$z'_{n+1} = 2z_n z'_n + 1 \quad \text{with } z'_0 = 0$$

The distance estimate is then:

$$d(c) \approx \frac{|z_n| \cdot \log|z_n|}{|z'_n|}$$

In GLSL, tracking the derivative alongside the main iteration:

vec2 z = vec2(0.0);
vec2 dz = vec2(0.0); // derivative
const int MAX_ITER = 300;
int iterations = 0;

for (int i = 0; i < MAX_ITER; i++) {
  // Update derivative first: dz = 2*z*dz + 1
  dz = 2.0 * cx_mul(z, dz) + vec2(1.0, 0.0);
  // Then update z
  z = cx_sqr(z) + c;
  if (dot(z, z) > 1e6) break;
  iterations++;
}

// Distance estimate
float r = length(z);
float dr = length(dz);
float dist = 0.5 * r * log(r) / dr;

This can be used to color the exterior of the set by distance to the boundary, producing smooth shading that resembles a 3D rendering of the fractal. The distance estimator is also useful for adaptive rendering: areas far from the boundary can be rendered at lower resolution, while areas near the boundary (where the interesting detail is) get more computational attention.

4.4.5 Orbit Statistics

During iteration, the sequence $z_0, z_1, z_2, \ldots$ traces a path through the complex plane. This path -- called the orbit -- contains rich information that we can extract and map to color. Here are the most useful orbit statistics:

Total path length: The sum of distances between consecutive orbit points.

$$L = \sum_{i=0}^{N-1} |z_{i+1} - z_i|$$

float pathLength = 0.0;
for (int i = 0; i < MAX_ITER; i++) {
  vec2 newZ = cx_sqr(z) + c;
  pathLength += distance(newZ, z);
  z = newZ;
  if (dot(z, z) > 256.0) break;
  iterations++;
}

Average orbit distance: The mean distance from the origin across all orbit points.

$$\bar{r} = \frac{1}{N} \sum_{i=0}^{N} |z_i|$$

float sumR = 0.0;
// ... inside the loop:
sumR += length(z);
// ... after the loop:
float avgOrbit = sumR / max(float(iterations), 1.0);

Orbit trap: The minimum distance from any orbit point to some geometric feature -- a point, a line, a circle, or any shape you choose. This technique was popularized by Clifford Pickover.

// Point orbit trap at the origin
float minDist = 1e10;
// ... inside the loop:
minDist = min(minDist, length(z));

// Circle orbit trap (center=(0.5, 0), radius=0.3)
float trapDist = abs(length(z - vec2(0.5, 0.0)) - 0.3);
minDist = min(minDist, trapDist);

// Cross orbit trap (distance to nearest axis)
float crossDist = min(abs(z.x), abs(z.y));
minDist = min(minDist, crossDist);

Tortuosity: The ratio of path length to displacement. A value of 1.0 means the orbit is a straight line; higher values indicate winding, chaotic orbits.

float displacement = length(z); // z_0 was (0,0)
float tortuosity = pathLength / max(displacement, 0.001);

4.4.6 Combining Statistics into Color

The art of fractal coloring lies in how we map these statistics to visual properties. There is no single "correct" mapping -- this is where artistic judgment matters most. Here are some strategies:

Map statistics to HSV channels: Use one statistic for hue, another for saturation, another for brightness.

float hue = fract(smoothIter / 25.0);
float sat = clamp(avgSegment * 0.5, 0.3, 1.0);
float val = clamp(1.0 - tortuosity * 0.05, 0.2, 1.0);
vec3 color = hsv2rgb(vec3(hue, sat, val));

Map to RGB channels directly: Each statistic modulates one color channel.

float r = sin(smoothIter * 0.1) * 0.5 + 0.5;
float g = sin(pathLength * 0.05 + 2.0) * 0.5 + 0.5;
float b = sin(avgOrbit * 0.2 + 4.0) * 0.5 + 0.5;
vec3 color = vec3(r, g, b);

Use orbit trap as alpha/blend: Blend between two color schemes based on how close the orbit came to a trap shape.

float t = smoothIter / float(MAX_ITER);
vec3 baseColor = palette(t, ...);
float trapFactor = clamp(1.0 - minDist * 5.0, 0.0, 1.0);
vec3 trapColor = vec3(1.0, 0.9, 0.7); // warm highlight
vec3 color = mix(baseColor, trapColor, trapFactor);
⚠ Common Mistake

When using multiple orbit statistics, make sure they are on comparable scales before combining them. Path length can range from 0 to thousands; tortuosity from 1 to hundreds; orbit trap from 0 to a few. Normalize each statistic (e.g., using clamp, fract, or division by expected range) before mapping to color.

4.5 Cosine Color Palettes

One of the most elegant and flexible coloring techniques for fractals comes from Inigo Quilez, a pioneer in real-time shader graphics. His cosine palette formula generates smooth, controllable color gradients from just four parameter vectors:

$$\text{color}(t) = \mathbf{a} + \mathbf{b} \cdot \cos\left(2\pi(\mathbf{c} \cdot t + \mathbf{d})\right)$$

where $t \in [0, 1]$ is the input parameter (normalized iteration count, for fractals), and $\mathbf{a}$, $\mathbf{b}$, $\mathbf{c}$, $\mathbf{d}$ are vec3 values (one component per RGB channel).

💡 Key Concept

The four parameters control the palette as follows:

  • a = offset (shifts the overall brightness of each channel)
  • b = amplitude (controls the range/contrast of each channel)
  • c = frequency (how many times the cosine cycles across $[0,1]$)
  • d = phase (shifts where each channel's cosine starts)

A common starting point is $\mathbf{a} = \mathbf{b} = (0.5, 0.5, 0.5)$, $\mathbf{c} = (1, 1, 1)$, and varying $\mathbf{d}$ to control the palette's hue distribution.

The beauty of this formula is its simplicity: one line of code, four parameters, infinite variety. Let us see how it works in practice.

In GLSL:

vec3 palette(float t, vec3 a, vec3 b, vec3 c, vec3 d) {
  return a + b * cos(6.28318 * (c * t + d));
}

// Example: rainbow palette
vec3 color = palette(t,
  vec3(0.5), vec3(0.5),
  vec3(1.0), vec3(0.0, 0.33, 0.67));

4.5.1 Understanding the Parameters

Let us break down each parameter to build intuition:

$\mathbf{a}$ (offset): This shifts the entire palette up or down in brightness. Setting $\mathbf{a} = (0.5, 0.5, 0.5)$ centers the cosine oscillation so colors range from 0 to 1. Setting $\mathbf{a} = (0.3, 0.3, 0.3)$ produces a darker palette.

$\mathbf{b}$ (amplitude): This controls how much each channel oscillates. With $\mathbf{b} = (0.5, 0.5, 0.5)$ and $\mathbf{a} = (0.5, 0.5, 0.5)$, colors span the full $[0, 1]$ range. Reducing $\mathbf{b}$ narrows the color range (more pastel); increasing it beyond 0.5 requires clamping.

$\mathbf{c}$ (frequency): This determines how many times the cosine completes a full cycle as $t$ goes from 0 to 1. With $\mathbf{c} = (1, 1, 1)$, each channel cycles once. With $\mathbf{c} = (2, 1, 0)$, the red channel cycles twice, green once, and blue stays constant -- producing warm-to-cool transitions.

$\mathbf{d}$ (phase): This is the most artistically powerful parameter. It controls where each channel starts its cosine cycle. Setting $\mathbf{d} = (0, 0.33, 0.67)$ staggers the channels by one-third of a cycle each, producing a rainbow. Setting $\mathbf{d} = (0, 0.1, 0.2)$ keeps the channels close together, producing warm sunset-like gradients.

4.5.3 Applying Cosine Palettes to Fractals

To use a cosine palette with a fractal, normalize your coloring value to $[0, 1]$ (or any range -- the cosine will wrap cyclically) and pass it through the palette function:

// In a Mandelbrot shader:
float t = smoothIter / float(MAX_ITER);

// Multiply t to control cycling speed
vec3 color = palette(t * 5.0,
  vec3(0.5, 0.5, 0.5),  // a: centered brightness
  vec3(0.5, 0.5, 0.5),  // b: full range
  vec3(1.0, 1.0, 1.0),  // c: one cycle each
  vec3(0.0, 0.33, 0.67)  // d: rainbow phase
);

if (iterations == MAX_ITER) color = vec3(0.0); // inside = black

The multiplier on $t$ controls how many times the palette cycles across the fractal. A multiplier of 1.0 maps the entire palette once across the full iteration range. A multiplier of 5.0 creates five bands of color. Adding u_time to the phase creates smoothly animated color cycling.

📐 Theorem: Cosine Palette Bounds

For a cosine palette with parameters $\mathbf{a}$ and $\mathbf{b}$, the output for each channel is bounded by $[a_i - |b_i|, \; a_i + |b_i|]$. To ensure all values stay in $[0, 1]$, we need $a_i - |b_i| \geq 0$ and $a_i + |b_i| \leq 1$. The standard choice $a_i = b_i = 0.5$ is optimal: it uses the full $[0, 1]$ range without clamping.

4.6 Color Spaces: HSV and CIELAB

4.6.1 RGB: The Default

RGB is the native color space of screens. Each color is a mix of Red, Green, and Blue intensities. GLSL's gl_FragColor output is in RGBA format.

Problem: RGB is not intuitive for artistic color design. "Make this color warmer" does not map to a simple RGB change. Also, RGB is not perceptually uniform -- equal numerical distances in RGB do not correspond to equal perceived color differences.

4.6.2 HSV (Hue, Saturation, Value)

HSV rearranges color into more intuitive dimensions:

  • Hue ($H$): The "color" -- position on the rainbow. Cyclic: $0 = 360 = \text{red}$.
  • Saturation ($S$): Vividness. $0 = \text{gray}$, $1 = \text{fully vivid}$.
  • Value ($V$): Brightness. $0 = \text{black}$, $1 = \text{maximum brightness}$.

HSV is excellent for fractals because we can cycle through all hues by mapping iteration count to $H$, while independently controlling vividness and brightness with $S$ and $V$. The cyclic nature of hue (wrapping from 1.0 back to 0.0) is a natural match for fractal coloring, where we often want the colors to cycle smoothly through the rainbow as iteration count increases.

A common technique is to set hue proportional to the smooth iteration count, saturation to a constant (say 0.8 for vivid colors), and value based on whether the point escaped:

// HSV fractal coloring pattern
float hue = fract(smoothIter / 30.0); // cycle every 30 iterations
float sat = 0.8;
float val = (iterations < MAX_ITER) ? 0.9 : 0.0; // black for interior
vec3 color = hsv2rgb(vec3(hue, sat, val));

The divisor in fract(smoothIter / 30.0) controls how many iterations span a full color cycle. Smaller values create rapid cycling with many color bands; larger values create gradual transitions with fewer bands. This is a key artistic parameter to experiment with.

4.6.3 The HSV to RGB Algorithm

The standard HSV to RGB conversion divides the hue circle into six sectors of 60 degrees each. In each sector, one RGB channel is at maximum, one is at minimum, and one transitions linearly between them. The compact GLSL implementation below achieves this without explicit branching:

HSV to RGB conversion in GLSL uses a compact formula that computes three piecewise-linear ramps, phase-shifted by 120 degrees. The mod(..., 6.0) - 3.0 creates a sawtooth wave that, after taking absolute value and subtracting 1, produces the characteristic triangular ramps for each channel:

vec3 hsv2rgb(vec3 c) {
  // c.x = hue [0,1], c.y = saturation [0,1], c.z = value [0,1]
  vec3 rgb = clamp(
    abs(mod(c.x * 6.0 + vec3(0.0, 4.0, 2.0), 6.0) - 3.0) - 1.0,
    0.0, 1.0
  );
  return c.z * mix(vec3(1.0), rgb, c.y);
}

4.6.4 CIELAB (L*a*b*)

CIELAB is a color space designed by the International Commission on Illumination (CIE) to be perceptually uniform. This means a distance of 1 unit in CIELAB looks like the same "amount of color change" everywhere in the space.

The three channels are:

  • L* (0 to 100): Lightness. Perceptually linear -- $L^* = 50$ looks "halfway between black and white."
  • a* (roughly -128 to +127): Green (negative) to Red (positive).
  • b* (roughly -128 to +127): Blue (negative) to Yellow (positive).

Why does perceptual uniformity matter for fractals? Consider a gradient from dark blue to light yellow. In RGB, a linear interpolation produces muddy, grayish intermediate colors. In CIELAB, the same interpolation produces a smooth, natural-looking transition. When coloring fractals, we often map a continuous value (like smooth iteration count) to a color gradient, and perceptual uniformity ensures that equal changes in the value produce equal visual changes in the output.

4.6.5 CIELAB to RGB Conversion

The conversion from CIELAB to RGB involves passing through the CIE XYZ color space, then applying the sRGB color matrix and gamma correction. The XYZ space is a device-independent space that represents all visible colors; CIELAB is a nonlinear transformation of XYZ designed for perceptual uniformity.

The conversion from CIELAB to RGB involves passing through the XYZ color space with gamma correction:

vec3 lab2rgb(vec3 lab) {
  float fy = (lab.x + 16.0) / 116.0;
  float fx = lab.y / 500.0 + fy;
  float fz = fy - lab.z / 200.0;
  float delta = 6.0 / 29.0;
  float d2 = delta * delta;
  float x = (fx > delta) ? fx*fx*fx : (fx - 16.0/116.0) * 3.0 * d2;
  float y = (fy > delta) ? fy*fy*fy : (fy - 16.0/116.0) * 3.0 * d2;
  float z = (fz > delta) ? fz*fz*fz : (fz - 16.0/116.0) * 3.0 * d2;
  x *= 0.95047; z *= 1.08883; // D65 white point
  float r =  3.2406*x - 1.5372*y - 0.4986*z;
  float g = -0.9689*x + 1.8758*y + 0.0415*z;
  float b =  0.0557*x - 0.2040*y + 1.0570*z;
  // Gamma correction
  r = r > 0.0031308 ? 1.055*pow(r,1.0/2.4)-0.055 : 12.92*r;
  g = g > 0.0031308 ? 1.055*pow(g,1.0/2.4)-0.055 : 12.92*g;
  b = b > 0.0031308 ? 1.055*pow(b,1.0/2.4)-0.055 : 12.92*b;
  return clamp(vec3(r, g, b), 0.0, 1.0);
}

4.6.6 Using CIELAB for Fractal Coloring

CIELAB is particularly effective for creating natural-looking fractal gradients. Instead of mapping iteration count to HSV hue (which produces a "neon rainbow" look), we can trace a path through CIELAB space:

// CIELAB fractal coloring
float t = smoothIter / float(MAX_ITER);
float L = 50.0 + 40.0 * cos(t * 6.28318);        // lightness oscillation
float a = 80.0 * cos(t * 6.28318 * 3.0);          // green-red oscillation
float b = 80.0 * sin(t * 6.28318 * 3.0 + 1.0);    // blue-yellow oscillation
vec3 color = lab2rgb(vec3(L, a, b));

The resulting colors are more subdued and "natural" than HSV rainbows -- they look like they could appear in nature or in high-quality print. This makes CIELAB a good choice for fractal art intended for print or exhibition, where the garish quality of RGB/HSV rainbows can be distracting.

What does "perceptually uniform" mean in the context of the CIELAB color space?

4.7 Newton Fractals

4.7.1 Newton's Method in the Complex Plane

Newton's method is a root-finding algorithm that iterates:

$$z_{n+1} = z_n - \frac{f(z_n)}{f'(z_n)}$$

In one dimension (real numbers), Newton's method is a standard technique taught in calculus: start with a guess, compute the tangent line to the function at that point, and follow the tangent line to where it crosses zero. This gives a better approximation, and we repeat. The method converges quadratically (the number of correct digits roughly doubles each iteration) when started near a root.

In the complex plane, the situation becomes dramatically more interesting. A polynomial of degree $n$ has exactly $n$ roots (by the Fundamental Theorem of Algebra), and each starting point $z_0$ converges to one of them. The question "which root does $z_0$ converge to?" partitions the complex plane into $n$ basins of attraction.

💡 Key Concept

The fractal arises from the boundaries between basins of attraction. For each root, there is a region of starting points that converge to it (its "basin of attraction"). Points near the boundary between two basins exhibit chaotic behavior -- tiny changes in starting position lead to convergence to different roots. These boundaries are fractal curves of infinite complexity.

4.7.2 The Standard Example: $f(z) = z^3 - 1$

This function has three roots, equally spaced on the unit circle at angles $0$, $\frac{2\pi}{3}$, and $\frac{4\pi}{3}$:

$$z_1 = 1, \quad z_2 = -\frac{1}{2} + \frac{\sqrt{3}}{2}i, \quad z_3 = -\frac{1}{2} - \frac{\sqrt{3}}{2}i$$

The derivative is $f'(z) = 3z^2$, so Newton's iteration becomes:

$$z_{n+1} = z_n - \frac{z_n^3 - 1}{3z_n^2}$$

4.7.3 Coloring by Root Convergence

For each pixel, we iterate until $z$ is close to one of the known roots (within a small tolerance like $10^{-4}$), then assign a color based on which root it converged to. We can also modulate the brightness based on how many iterations were needed:

// After the iteration loop:
vec2 root1 = vec2(1.0, 0.0);
vec2 root2 = vec2(-0.5, 0.866025);
vec2 root3 = vec2(-0.5, -0.866025);

int whichRoot = -1;
if (distance(z, root1) < 0.001) whichRoot = 0;
if (distance(z, root2) < 0.001) whichRoot = 1;
if (distance(z, root3) < 0.001) whichRoot = 2;

// Shade by iteration count (fewer iterations = brighter)
float shade = 1.0 - float(iterations) / float(MAX_ITER) * 0.7;

vec3 color = vec3(0.0);
if (whichRoot == 0) color = vec3(1.0, 0.2, 0.2) * shade;
if (whichRoot == 1) color = vec3(0.2, 1.0, 0.2) * shade;
if (whichRoot == 2) color = vec3(0.3, 0.3, 1.0) * shade;

The shade variation creates a 3D-like appearance, as if the fractal were a landscape viewed from above, with valleys (many iterations) appearing darker than ridges (few iterations).

4.7.4 Understanding Basin Boundaries

The fractal boundary between basins of attraction has a remarkable property: it is the same boundary for all three (or $n$) basins. In other words, any point on the boundary is simultaneously on the boundary of every basin. This was proven by Fatou and Julia in the early 20th century and is one of the most counterintuitive results in complex dynamics.

A consequence is that if you zoom into any part of the boundary, you will see regions of all three colors interleaved at every scale. There is no "clean" border between any two basins -- the third basin is always present, interweaving with the other two in a fractal pattern.

The speed of convergence also varies dramatically near the boundary. Points far from the boundary converge in just a few iterations (bright in our shading scheme). Points near the boundary take many iterations (dark), and points exactly on the boundary never converge (they form a set of measure zero called the Julia set of the Newton map).

4.7.5 Damped Newton's Method

The generalized Newton iteration introduces a multiplier $m$:

$$z_{n+1} = z_n - m \cdot \frac{f(z_n)}{f'(z_n)}$$

  • $m = 1$: Standard Newton's method
  • $m \neq 1$ (real): Relaxed ($m < 1$) or over-relaxed ($m > 1$) Newton -- produces dramatic fractal variations
  • $m$ complex: Produces spiraling, twisting basin boundaries

As $m$ deviates from 1, the basin boundaries widen, swirl, and develop increasingly chaotic structures. Some values of $m$ produce spiraling vortices; others create kaleidoscopic patterns with multiple symmetry levels.

4.7.6 Other Polynomials

The technique generalizes to any polynomial. For $f(z) = z^n - 1$, the roots are the $n$-th roots of unity, equally spaced on the unit circle at angles $\frac{2\pi k}{n}$ for $k = 0, 1, \ldots, n-1$. The derivative is $f'(z) = nz^{n-1}$.

For $f(z) = z^5 - 1$, we get five basins of attraction with 5-fold rotational symmetry. For $f(z) = z^3 - 2z + 2$, we get three roots that are not equally spaced, producing an asymmetric but equally fractal boundary.

You can even use non-polynomial functions. For example, $f(z) = \sin(z)$ has infinitely many roots at $z = n\pi$, and Newton's method applied to it produces an infinite fractal pattern:

// Newton's method for sin(z)
// f(z) = sin(z), f'(z) = cos(z)
vec2 dz = cx_div(cx_sin(z), cx_cos(z));
z = z - dz;

4.7.7 Artistic Variations

Beyond changing the polynomial, several modifications produce artistically interesting results:

  • Nova fractal: Add a constant $c$ per pixel: $z_{n+1} = z_n - m \cdot \frac{f(z_n)}{f'(z_n)} + c$. This combines Newton iteration with Mandelbrot-style parameter variation.
  • Halley's method: A higher-order root-finding method that uses the second derivative: $z_{n+1} = z_n - \frac{2f(z_n)f'(z_n)}{2[f'(z_n)]^2 - f(z_n)f''(z_n)}$. This produces smoother basins with different fractal structure.
  • Blending convergence and iteration count: Use the iteration count for brightness modulation on top of root-based hue, creating depth and texture in the image.
📜 Historical Note

Newton fractals were first studied computationally by John Hubbard in the 1970s. He was surprised to find that Newton's method -- a well-understood algorithm from 1669 -- produced fractal basin boundaries in the complex plane. The result showed that even simple, deterministic processes can produce arbitrarily complex behavior. The mathematician Arthur Cayley had attempted to analyze the convergence boundaries of Newton's method for cubic polynomials in 1879, but could not resolve the question -- it would take a century and the advent of computer graphics to reveal the fractal answer.

4.8 GLSL Frame Feedback

4.8.1 The Concept

So far, each frame is computed from scratch -- the shader does not know what it drew last frame. Frame feedback changes this: we save the previous frame as a texture and feed it back into the shader as an input.

This enables:

  • Cellular automata (Game of Life) -- each cell's next state depends on its neighbors' current states
  • Reaction-diffusion systems -- chemicals diffusing and reacting across a surface
  • Trail effects -- fading afterimages that create motion trails
  • Iterative refinement -- progressively rendering more detail each frame

4.8.2 The Ping-Pong Pattern

The key technique is double buffering (also called "ping-pong"):

  1. Frame 1: Read from buffer A, write to buffer B
  2. Frame 2: Read from buffer B, write to buffer A
  3. Frame 3: Read from buffer A, write to buffer B ...

We alternate which buffer we read from and which we write to. This avoids the problem of reading from and writing to the same buffer simultaneously (which would give unpredictable results since the GPU processes pixels in parallel).

In p5.js, we can use createGraphics() or createFramebuffer() to make off-screen buffers. Here is the JavaScript pattern:

let bufferA, bufferB;
let currentBuffer = 0;

function setup() {
  createCanvas(512, 512, WEBGL);
  bufferA = createGraphics(512, 512, WEBGL);
  bufferB = createGraphics(512, 512, WEBGL);
  bufferA.pixelDensity(1);
  bufferB.pixelDensity(1);
  // Initialize bufferA with starting state...
}

function draw() {
  let readBuf = (currentBuffer === 0) ? bufferA : bufferB;
  let writeBuf = (currentBuffer === 0) ? bufferB : bufferA;

  // Run shader: read from readBuf, write to writeBuf
  writeBuf.shader(myShader);
  myShader.setUniform("prevFrame", readBuf);
  myShader.setUniform("resolution", [512, 512]);
  writeBuf.quad(-1, -1, 1, -1, 1, 1, -1, 1);

  // Display the result
  image(writeBuf, -width/2, -height/2, width, height);

  // Swap buffers
  currentBuffer = 1 - currentBuffer;
}
⚠ Common Mistake

Forgetting to set pixelDensity(1) on the off-screen buffers is a common source of bugs. On high-DPI displays (Retina), p5.js defaults to 2x pixel density, which means your 512x512 buffer is actually 1024x1024 pixels. This causes texture coordinate mismatches in the shader. Always set pixelDensity(1) for buffers used in frame feedback.

4.8.3 Game of Life in GLSL

The most famous example of a system that benefits from frame feedback is Conway's Game of Life, a cellular automaton invented by mathematician John Conway in 1970. Each cell (pixel) is alive (1) or dead (0), and its next state depends only on its current state and the states of its eight neighbors.

The rules are remarkably simple -- just three conditions:

  • A dead cell with exactly 3 live neighbors becomes alive (birth)
  • A live cell with 2 or 3 live neighbors stays alive (survival)
  • All other live cells die (underpopulation or overpopulation)
💡 Key Concept

The Game of Life demonstrates emergence: complex, seemingly purposeful behavior arising from extremely simple rules. From these three conditions alone, the following phenomena arise:

  • Still lifes: Stable patterns that do not change (blocks, beehives, loaves)
  • Oscillators: Patterns that cycle through a fixed sequence of states (blinkers, pulsars, pentadecathlons)
  • Spaceships: Patterns that translate across the grid over time (gliders, lightweight spaceships)
  • Guns: Patterns that periodically emit spaceships (Gosper's glider gun)
  • Universal computation: The Game of Life is Turing-complete -- it can simulate any computation

This is a central theme in computing arts -- designing systems whose output transcends the simplicity of their rules.

4.8.4 Implementing Game of Life in GLSL

In the GLSL Game of Life, each pixel stores a binary state (alive or dead) in its red channel. The fragment shader reads the previous frame via a texture uniform, counts the eight neighbors by sampling surrounding pixels, and applies the rules:

// GLSL Game of Life fragment shader (core logic)
vec2 pixel = 1.0 / resolution;
float current = texture2D(prevFrame, vTexCoord).r;

// Count live neighbors
float neighbors = 0.0;
for (int dx = -1; dx <= 1; dx++) {
  for (int dy = -1; dy <= 1; dy++) {
    if (dx == 0 && dy == 0) continue;
    vec2 nc = vTexCoord + vec2(float(dx), float(dy)) * pixel;
    neighbors += step(0.5, texture2D(prevFrame, nc).r);
  }
}

// Apply rules
float alive = 0.0;
if (current > 0.5) {
  // Alive: survive with 2 or 3 neighbors
  if (neighbors > 1.5 && neighbors < 3.5) alive = 1.0;
} else {
  // Dead: birth with exactly 3 neighbors
  if (neighbors > 2.5 && neighbors < 3.5) alive = 1.0;
}

gl_FragColor = vec4(vec3(alive), 1.0);

The step(0.5, value) function returns 1.0 if value >= 0.5, and 0.0 otherwise. This converts the continuous texture value into a binary alive/dead decision, handling any floating-point imprecision from the texture sampling.

The JavaScript side manages the ping-pong buffers and initializes the grid with random state. Clicking the canvas adds live cells, seeding new patterns.

4.8.5 Frame Feedback for Visual Effects

Frame feedback is not limited to cellular automata. It opens up a wide range of visual effects:

Trail effect (fading): Blend the previous frame with new content to create motion trails:

// In a GLSL shader with prevFrame texture:
vec4 prev = texture2D(prevFrame, vTexCoord) * 0.95; // fade
vec4 current = /* compute new content */;
gl_FragColor = max(prev, current); // keep brighter

Reaction-diffusion: Each pixel's new value depends on its neighbors' previous values. The Gray-Scott model simulates two chemicals that diffuse and react:

// Gray-Scott reaction-diffusion (simplified)
float a = texture2D(prevFrame, vTexCoord).r;
float b = texture2D(prevFrame, vTexCoord).g;

// Laplacian: average of neighbors minus center
float laplacianA = 0.0, laplacianB = 0.0;
for (int dx = -1; dx <= 1; dx++) {
  for (int dy = -1; dy <= 1; dy++) {
    vec2 nc = vTexCoord + vec2(float(dx), float(dy)) * pixel;
    vec4 n = texture2D(prevFrame, nc);
    float w = (dx == 0 || dy == 0) ? 0.2 : 0.05;
    if (dx == 0 && dy == 0) w = -1.0;
    laplacianA += n.r * w;
    laplacianB += n.g * w;
  }
}

float feed = 0.055, kill = 0.062;
float newA = a + (1.0 * laplacianA - a*b*b + feed*(1.0-a));
float newB = b + (0.5 * laplacianB + a*b*b - (kill+feed)*b);
gl_FragColor = vec4(newA, newB, 0.0, 1.0);

Reaction-diffusion systems produce organic-looking patterns -- spots, stripes, labyrinths, and worm-like structures -- from simple local rules. They model actual biological pattern formation (Alan Turing proposed this mechanism in 1952). We will cover this in depth in a later chapter.

Zoom feedback: Sample the previous frame with a slight zoom or rotation to create recursive visual effects:

// Zoom into center with slight rotation
vec2 center = vec2(0.5, 0.5);
vec2 offset = vTexCoord - center;
float angle = 0.01; // slight rotation per frame
mat2 rot = mat2(cos(angle), sin(angle), -sin(angle), cos(angle));
vec2 samplePos = center + rot * offset * 0.99; // 0.99 = zoom in
vec4 prev = texture2D(prevFrame, samplePos) * 0.98;
📖 Definition

Video feedback is the visual analogue of audio feedback (the screech when a microphone picks up its own speaker output). In video, pointing a camera at its own monitor creates recursive, fractal-like imagery. GLSL frame feedback is the digital version of this technique, with precise mathematical control over the transformation applied at each step.

4.9.2 Animated Parameter Morphing

Sending a time uniform to the shader unlocks animation. The key insight is that fractal shapes change smoothly as their parameters change -- so animating a parameter creates a continuously morphing visual. This is one of the most compelling aspects of real-time fractal rendering.

// JavaScript side
s.setUniform("time", millis() / 1000.0);

Time-based animation creates hypnotic fractal movies:

4.9.3 Mouse Dragging for Pan

For precise control of the view center, accumulate mouse drag movements:

let cx = 0, cy = 0;

function mouseDragged() {
  cx += (mouseX - pmouseX) / width * 2;
  cy -= (mouseY - pmouseY) / height * 2;
  s.setUniform("center", [cx, cy]);
}

4.9.4 Scroll Wheel for Zoom

The mouse wheel provides intuitive zoom control. Each scroll step multiplies the zoom level by a factor:

let zoom = 1.5;

function mouseWheel(event) {
  if (event.delta > 0) {
    zoom *= 1.1; // zoom out
  } else {
    zoom *= 0.9; // zoom in
  }
  s.setUniform("zoom", zoom);
  return false; // prevent page scroll
}

4.10 Early Abstract Animation

The computational approaches we have been studying -- generating images from mathematical functions, creating patterns through iterative processes, mapping data to visual form -- have a rich history that predates computers.

4.10.1 Oskar Fischinger (1900--1967)

Fischinger was a German-American animator who pioneered visual music -- abstract animated films synchronized to music. His work demonstrated that purely abstract visual forms, when carefully synchronized with music, could create deeply emotional experiences without any representational content.

His "Wax Experiments" (1921--1926) created animation by slicing through layered wax blocks, each cross-section becoming a frame. This is essentially 3D volumetric data rendered as a 2D sequence -- anticipating modern techniques like CT scanning, volume rendering, and slicing through 3D fractal data.

📜 Historical Note

Fischinger's technique of encoding time as a spatial dimension (the depth axis of the wax block) is directly analogous to how we use u_time as a parameter in our shaders. Both approaches treat time as just another variable in a mathematical function.

4.10.2 Norman McLaren (1914--1987)

McLaren pioneered direct animation -- drawing and scratching directly on film, bypassing the camera entirely. In Dots (1940), he created both the visuals and soundtrack by hand-marking the film strip -- the visual patterns on one edge of the film were read by the projector's optical sound reader, producing corresponding audio tones. This is perhaps the earliest example of a unified audiovisual generative system.

McLaren's key insight: "Animation is not the art of drawings that move, but the art of movements that are drawn." This resonates directly with shader programming, where we design processes that produce visual output. McLaren worked at the National Film Board of Canada for over 40 years, producing over 60 films and mentoring generations of experimental animators.

4.10.3 Mary Ellen Bute (1906--1983)

Bute used oscilloscopes and electronic signals to generate abstract visual forms, synchronized to music. Tarantella (1940) combined electronic Lissajous figures with hand-animation. Bute's oscilloscope work is a direct predecessor of shader-based visualization: converting mathematical functions to visual patterns in real time. The Lissajous figures she used -- parametric curves of the form $(A\sin(at + \delta), B\sin(bt))$ -- are essentially what we compute when we evaluate trigonometric functions across the complex plane.

Bute was also notable for being one of the first women to gain recognition as an experimental filmmaker, working at a time when the field was overwhelmingly male. Her combination of mathematical sophistication with artistic sensitivity produced some of the most visually striking abstract films of the 1930s and 1940s.

4.10.4 Heider and Simmel (1944)

Fritz Heider and Marianne Simmel created a short animation of geometric shapes (triangles, circles) moving around a rectangle. When shown to test subjects, nearly everyone described the shapes as intentional agents: "the big triangle is chasing the small one," "the circle is hiding."

This demonstrates that humans automatically attribute agency to moving shapes. This phenomenon, known as animacy perception, is deeply hardwired in the human visual system. Infants as young as 6 months show sensitivity to animate motion. For computing artists, this is both a powerful tool and a responsibility: even abstract geometric forms, when animated, will be interpreted as characters with intentions, emotions, and social relationships.

This is directly relevant when we create agent-based simulations (which we will study in later chapters): viewers will see intentions and emotions in our agents' movements even if we coded no such thing. The implication is that motion design is never purely abstract -- it always carries narrative meaning for the viewer, whether intended or not.

4.10.5 Viking Eggeling (1880--1925)

Eggeling was a Swedish artist who created Diagonal Symphony (1924), often considered the first abstract animated film. Working from scroll-like drawings that depicted visual "themes" and their variations, Eggeling treated visual form like musical composition -- with counterpoint, rhythm, and development. His approach of systematic visual transformation parallels our use of parameterized functions and animated uniforms.

Eggeling developed a theory of "Generalbass der Malerei" (a thorough-bass of painting), seeking to discover universal principles of visual composition analogous to the rules of musical harmony. He spent years creating "scroll paintings" -- long horizontal compositions showing the progressive transformation of shapes -- before translating these into the animated film Diagonal Symphony. Tragically, Eggeling died in 1925, just days after the first public screening of his film.

The concept of the "visual scroll" -- a continuous transformation of form -- maps directly to our shader practice. A fragment shader with a time uniform is essentially an infinite visual scroll: the mathematical function describes a continuous transformation parameterized by time, and each frame is a slice through this continuum.

4.10.7 The Continuity

These artists share a set of principles that run directly through computing arts:

  • Abstract form as a complete artistic language (no representation needed)
  • Process over product (design the system, not individual frames)
  • Mathematics as aesthetics (functions and transformations are inherently visual)
  • Cross-modal correspondence (visual-musical synesthesia)
  • Technology as artistic instrument (film, oscilloscopes, GPUs)

When we write a fragment shader that generates a fractal landscape, we are working in the same tradition. The shader is a process -- a mathematical function that the GPU evaluates at every pixel -- and the resulting image emerges from this process. We do not draw the Mandelbrot set; we describe the rules, and the set reveals itself.

This shift from "drawing pictures" to "designing processes" is perhaps the most fundamental conceptual change in moving from traditional to computational art. Fischinger, McLaren, Bute, and Eggeling all understood this decades before computers made it explicit.

"I am interested in the abstract presentation of music, not through illustrations, but through new forms created by the music itself." -- Oskar Fischinger

✍ Knowledge Check

Norman McLaren said: "Animation is not the art of drawings that move, but the art of movements that are drawn." How does this relate to shader programming?

McLaren's insight is that the fundamental unit of animation is not a picture but a transformation -- the change between frames. Similarly, a fragment shader is not a drawing but a function: given a position and time, it computes a color. We design the process (the mathematical rule), and the visual output emerges from it.

4.11 Summary

This chapter covered a wide arc from mathematical foundations to artistic vision:

  • Complex arithmetic in GLSL provides the computational substrate for all our fractal work. We built multiplication, division, exponentiation, sine, cosine, power, and logarithm for complex numbers represented as vec2.
  • Alternative fractal formulas ($z^3 + c$, $\sin(z) + c$, $e^z + c$, Burning Ship, etc.) produce entirely different fractal landscapes. Each function's geometry -- its symmetries, periodicities, and growth rates -- shapes the fractal it produces.
  • Smooth coloring eliminates the banding artifacts of integer iteration counts. The formula $n_{\text{smooth}} = n + 1 - \frac{\log(\log|z|)}{\log 2}$ gives a continuous coloring value. Histogram coloring and distance estimation offer additional approaches.
  • Cosine palettes ($\text{color}(t) = a + b\cos(2\pi(ct+d))$) provide compact, highly flexible color mapping with just four parameter vectors.
  • Color spaces -- HSV for intuitive artistic control, CIELAB for perceptual uniformity -- give us tools to think about color beyond raw RGB values.
  • Newton fractals arise from applying Newton's root-finding method to complex polynomials. They are colored by which root each starting point converges to, revealing fractal basin boundaries. The damping parameter $m$ creates dramatic variations.
  • Frame feedback enables cellular automata, reaction-diffusion, and trail effects by feeding the previous frame back as a texture input using the ping-pong double-buffering pattern.
  • Interactivity through mouse uniforms, time-based animation, and keyboard controls transforms static fractal renderers into real-time exploration tools.
  • Early abstract animation -- Fischinger, McLaren, Bute, Heider and Simmel, Eggeling -- establishes that abstract moving forms carry meaning, that process can be art, and that mathematics and aesthetics are deeply intertwined.

4.11.2 Key Takeaways for Practice

Here are the most practically useful techniques from this chapter, summarized for quick reference:

  1. Always use the complex number library. Paste the complete set of cx_* functions at the top of every fractal shader. It costs nothing in performance for unused functions (GLSL compilers eliminate dead code).
  2. Use smooth coloring by default. There is almost never a reason to use raw integer iteration counts. The smooth formula adds one line of code and dramatically improves the output.
  3. Start with a cosine palette. The Quilez cosine palette formula is the best balance of flexibility, compactness, and visual quality for shader-based fractal coloring.
  4. Match escape radius to formula. For $z^n + c$, an escape radius of 4 is sufficient but 256 is better for smooth coloring. For $e^z + c$, use 100 or more. For $\sin(z) + c$, use 50.
  5. Explore parameter space systematically. When trying a new formula, start zoomed out (large scale), identify interesting regions, then zoom in. Many fractals have their most interesting features near specific parameter values that require exploration to find.
  6. Use the mouse for real-time exploration. Mapping mouse position to fractal parameters (Julia set $c$, Newton multiplier $m$, cosine palette phase) turns your shader into an interactive instrument.

The fractal formula determines the geometry (where are the boundaries, what symmetries exist), while the coloring method determines the aesthetic (how that geometry is revealed to the viewer). Mastering both gives you an enormous creative space to explore.

Chapter 4 Review Quiz

Test your understanding of creative fractals, coloring techniques, and frame feedback.

✍ Question 1

What symmetry does the Multibrot set $z^4 + c$ exhibit?

The Multibrot set $z^n + c$ exhibits $(n-1)$-fold rotational symmetry. For $z^4 + c$, that is $4 - 1 = 3$-fold rotational symmetry.

✍ Question 2

In the cosine palette formula $\text{color}(t) = a + b \cdot \cos(2\pi(ct + d))$, what does the parameter $\mathbf{d}$ control?

The $\mathbf{d}$ vector controls the phase offset for each RGB channel. By shifting the phases, you change which colors appear at which $t$ values. The parameters are: $\mathbf{a}$ = brightness offset, $\mathbf{b}$ = amplitude, $\mathbf{c}$ = frequency, $\mathbf{d}$ = phase.

✍ Question 3

Why does the "ping-pong" double-buffering technique use two buffers instead of one?

The GPU processes many pixels simultaneously. If a shader reads from the same buffer it is writing to, some pixels would read old values and others would read newly written values, depending on the order of execution. Two buffers solve this: we read from one (complete, stable data) and write to the other (fresh output), then swap for the next frame.

✍ Question 4

What is the Burning Ship fractal's key modification to the standard Mandelbrot iteration?

The Burning Ship uses $z_{n+1} = (|\text{Re}(z_n)| + i|\text{Im}(z_n)|)^2 + c$. Taking the absolute value of each component before squaring breaks the smooth symmetry of the standard Mandelbrot, creating the jagged, asymmetric structures characteristic of the Burning Ship.

✍ Question 5

How many roots does the polynomial $z^5 - 1$ have in the complex plane, and what is the angular spacing between them?

By the Fundamental Theorem of Algebra, $z^5 - 1$ has exactly 5 roots. These are the 5th roots of unity, equally spaced on the unit circle. The angular spacing is $\frac{360°}{5} = 72°$. The roots are $e^{2\pi i k / 5}$ for $k = 0, 1, 2, 3, 4$.
