2.1 JavaScript and p5.js Revisited

Before we dive into the deeper topics of this chapter, let us briefly revisit the tools we are working with and understand where they come from. p5.js is a JavaScript library for creative coding that runs in any modern web browser. It descends from a rich lineage of artist-oriented programming environments.

2.1.1 The Processing Lineage

In 2001, Casey Reas and Ben Fry, then graduate students at the MIT Media Lab working under John Maeda, created Processing. Their goal was radical: make programming accessible to visual artists and designers who had no formal computer science training. Processing used a simplified Java syntax and provided immediate visual feedback — type a few lines of code, press play, and see shapes appear on screen.

📜 Historical Note

John Maeda, who directed the Aesthetics + Computation Group at MIT Media Lab, had already explored the intersection of code and design through his Design By Numbers project (1999). Processing built on this foundation, creating a tool that would eventually be used by millions of artists, designers, and educators worldwide.

Processing grew into an ecosystem. Ports appeared for Python (Processing.py), Android, and eventually the web. In 2013, Lauren Lee McCarthy created p5.js, a complete reimagining of Processing for the JavaScript ecosystem. Rather than simply translating Processing's Java API to JavaScript, McCarthy redesigned the library around web-native concepts: the DOM, event handling, and the HTML5 Canvas API.

2.1.2 Why JavaScript Matters for Artists

JavaScript is the language of the web browser, and the web browser is the most universal creative platform ever built. A p5.js sketch runs on any device with a browser — no installation, no compilation, no app store. You share your art by sharing a URL. This is a profound shift from the early days of computer art, when creating and sharing work required access to expensive mainframe computers and specialized plotting hardware.

The web platform also provides access to capabilities that Processing never had: real-time audio input, webcam access, WebGL-accelerated 3D graphics, WebSocket networking, and the entire ecosystem of JavaScript libraries. p5.js embraces these through an addon system: p5.sound for audio, ml5.js for machine learning, and many community-contributed libraries.

2.1.3 The p5.js Lifecycle

Every p5.js sketch revolves around two core functions that define its behavior over time:

function setup() {
  // Runs ONCE when the program starts
  createCanvas(400, 400);
}

function draw() {
  // Runs REPEATEDLY (~60 times per second)
  background(220);
  ellipse(mouseX, mouseY, 50, 50);
}

The setup() function initializes state — canvas size, background color, any variables that need to exist before animation begins. The draw() function is the animation loop, called approximately 60 times per second by default. Between them, p5.js manages a rendering context — a data structure that tracks the current fill color, stroke weight, transformation matrix, blending mode, and other drawing state. Understanding and controlling this state is the key to mastering transformations, which is the primary focus of this chapter.

A useful mental model: think of the rendering context as an invisible assistant who remembers your instructions. When you say fill(255, 0, 0), the assistant notes “red fill” and applies it to every subsequent shape until you change it. When you say translate(100, 0), the assistant shifts its grid paper 100 pixels to the right, and all future coordinates are measured from the shifted origin. push() tells the assistant to take a snapshot of everything; pop() tells it to restore to that snapshot.

2.2 CPU vs GPU Rendering

Your computer contains two fundamentally different processors that can produce graphics. Understanding the distinction is essential as you progress toward shaders later in this course.

2.2.1 The CPU: Serial Processing

The CPU (Central Processing Unit) is a general-purpose processor optimized for sequential tasks. A modern CPU has 4 to 16 powerful cores, each capable of executing billions of instructions per second. It excels at complex decision-making, branching logic, and tasks where each step depends on the result of the previous step.

CPUs are designed for latency — completing any single task as fast as possible. They have large caches, sophisticated branch predictors, and out-of-order execution units. A single CPU core can do almost anything, but it can only do one thing at a time (per core).

When p5.js draws with the default 2D renderer, the CPU processes each shape one at a time: calculate the vertices of this rectangle, fill its pixels, then move to the next rectangle. For a grid of 264 squares (like Schotter), the CPU performs 264 sequential drawing operations.

2.2.2 The GPU: Parallel Processing

The GPU (Graphics Processing Unit) takes a radically different approach. Instead of a few powerful cores, a GPU has thousands of small cores designed to execute the same operation on many data points simultaneously. A modern GPU might have 4,000+ cores. GPUs are designed for throughput — completing the maximum total amount of work per second, even if each individual task takes slightly longer.

💡 Key Concept

SIMD (Single Instruction, Multiple Data) — The GPU's programming model. One program (the shader) runs simultaneously on thousands of cores, each processing a different pixel. Every pixel is computed independently and in parallel.

Consider coloring a 400×400 canvas (160,000 pixels). The CPU must visit each pixel sequentially — 160,000 steps. The GPU assigns each pixel to a separate core and computes them all at once — effectively one step (in the ideal case).

2.2.4 A Mental Model: The Factory Analogy

Think of the CPU as a master craftsperson: highly skilled, able to handle any task, but working alone (or with a few apprentices). They can make complex decisions, adapt on the fly, and solve novel problems.

Think of the GPU as a factory floor with thousands of workers, each trained to do one specific task extremely fast. Give them all the same job (color this pixel), and they finish in a flash. But ask each worker to do something different, and the system grinds to a halt.

This analogy explains why GPUs excel at shaders (every pixel runs the same program) but struggle with complex branching logic (each pixel might need different treatment). It also explains why we will use the CPU for most of this chapter’s exercises and save the GPU for later chapters on GLSL and ray marching.

📖 Definition

Shader — A small program that runs on the GPU, typically once per pixel (fragment shader) or once per vertex (vertex shader). Shaders execute in parallel across thousands of GPU cores, enabling real-time pixel-level effects on high-resolution displays.

2.3 Geometric Transformations

Geometric transformations are operations that change the position, orientation, or size of shapes. In p5.js, instead of moving individual shapes, we move the entire coordinate system. This seemingly indirect approach turns out to be enormously powerful, enabling hierarchical compositions, animation, and the kind of structured randomness that defines algorithmic art.

The idea of transforming coordinate systems rather than individual points dates back to the 19th century and the development of linear algebra. In computer graphics, this approach was formalized in the 1960s and remains the standard in every graphics API from OpenGL to p5.js. Once you understand transformations, you can compose complex scenes from simple primitives, animate articulated figures, and create the recursive structures that produce fractals.

2.3.1 Translation

translate(tx, ty) moves the origin of the coordinate system by tx pixels horizontally and ty pixels vertically. After calling translate(100, 50), drawing at position (0, 0) actually places the shape at what was previously (100, 50).

📖 Definition

Translation — A geometric transformation that shifts every point by the same displacement vector $(t_x, t_y)$. For any point $(x, y)$, the translated point is $(x + t_x, \; y + t_y)$.

Translations are cumulative. Calling translate(100, 0) followed by translate(50, 0) is equivalent to translate(150, 0).

2.3.2 Rotation

rotate(angle) rotates the coordinate system around the current origin by the given angle (in radians). Positive angles rotate clockwise in p5.js (because the y-axis points downward).

📖 Definition

Rotation — A transformation that turns every point around a fixed center by angle $\theta$. A point $(x, y)$ becomes:

$$(x \cos\theta - y \sin\theta, \quad x \sin\theta + y \cos\theta)$$

⚠ Common Mistake

Rotation always happens around the current origin. If the origin is at the top-left corner (the default), the entire drawing swings around that corner. To rotate a shape around its own center, you must translate() to that center first, then rotate().

2.3.3 Scaling

scale(sx, sy) multiplies all coordinates by the given factors. scale(2) makes everything twice as large. scale(0.5) makes everything half-sized. You can scale differently along each axis: scale(2, 0.5) stretches horizontally and squashes vertically.

📖 Definition

Scaling — A transformation that multiplies coordinates by scale factors. Point $(x, y)$ becomes $(s_x \cdot x, \; s_y \cdot y)$.

2.3.4 Order of Transformations Matters

A critical subtlety: the order in which you apply transformations changes the result. Consider these two sequences:

// Sequence A: translate then rotate
translate(200, 0);
rotate(PI / 4);
rect(0, 0, 50, 50);

// Sequence B: rotate then translate
rotate(PI / 4);
translate(200, 0);
rect(0, 0, 50, 50);

In Sequence A, we first move 200 pixels right, then rotate around that new position. The rectangle ends up 200 pixels to the right, tilted 45 degrees.

In Sequence B, we first rotate the entire coordinate system 45 degrees, then move 200 pixels along the rotated x-axis (which now points diagonally). The rectangle ends up in a completely different position.

The general rule: later transformations operate in the coordinate system established by earlier ones. Think of it as giving instructions to a person facing a direction: “turn right, then walk forward” produces a different path than “walk forward, then turn right.”

2.3.5 The Transformation Matrix

Internally, p5.js represents all transformations as a single 3×3 matrix (using homogeneous coordinates). Every call to translate(), rotate(), or scale() multiplies the current matrix by a new transformation matrix.

The general 2D affine transformation matrix is:

$$\mathbf{M} = \begin{pmatrix} a & c & t_x \\ b & d & t_y \\ 0 & 0 & 1 \end{pmatrix}$$

A point $(x, y)$ is transformed by computing:

$$\begin{pmatrix} x' \\ y' \\ 1 \end{pmatrix} = \begin{pmatrix} a & c & t_x \\ b & d & t_y \\ 0 & 0 & 1 \end{pmatrix} \begin{pmatrix} x \\ y \\ 1 \end{pmatrix}$$

The specific matrices for each transformation are:

Translation by $(t_x, t_y)$:

$$\mathbf{T} = \begin{pmatrix} 1 & 0 & t_x \\ 0 & 1 & t_y \\ 0 & 0 & 1 \end{pmatrix}$$

Rotation by $\theta$:

$$\mathbf{R} = \begin{pmatrix} \cos\theta & -\sin\theta & 0 \\ \sin\theta & \cos\theta & 0 \\ 0 & 0 & 1 \end{pmatrix}$$

Scaling by $(s_x, s_y)$:

$$\mathbf{S} = \begin{pmatrix} s_x & 0 & 0 \\ 0 & s_y & 0 \\ 0 & 0 & 1 \end{pmatrix}$$

💡 Key Concept

Matrix composition — Multiple transformations combine by matrix multiplication. The order matters because matrix multiplication is not commutative: rotating then translating produces a different result than translating then rotating. In p5.js, transformations are applied in the order you write them, and each modifies the current coordinate system.

2.4 The Matrix Stack: push() and pop()

Every call to translate(), rotate(), or scale() modifies the current transformation matrix. Without a way to undo these changes, transformations would accumulate uncontrollably. This is where push() and pop() come in.

📖 Definition

push() saves a copy of the current drawing state (transformation matrix, fill, stroke, etc.) onto a stack. pop() restores the most recently saved state. They always come in matched pairs.

2.4.1 Why State Saving Matters

Consider drawing two squares, each rotated independently:

// WITHOUT push/pop -- BROKEN
translate(100, 100);
rotate(0.3);
rect(0, 0, 40, 40);  // Square A

translate(200, 0);    // This 200 is relative to the ROTATED system!
rotate(0.5);          // This adds to the previous 0.3 rotation!
rect(0, 0, 40, 40);  // Square B -- in the wrong place and angle
// WITH push/pop -- CORRECT
push();
translate(100, 100);
rotate(0.3);
rect(0, 0, 40, 40);  // Square A
pop();                // Coordinate system restored

push();
translate(300, 100);
rotate(0.5);
rect(0, 0, 40, 40);  // Square B -- independent
pop();

2.4.2 The Stack Data Structure

A stack is a last-in-first-out (LIFO) data structure, like a stack of plates. push() places a plate on top; pop() removes the top plate. The key insight is that you always restore the most recently saved state, which makes nested transformations natural.

2.4.3 Nested Transformations and Hierarchical Coordinate Systems

By nesting push()/pop() blocks, you create a hierarchy of coordinate systems. Each child coordinate system is defined relative to its parent. When the parent moves, the child moves with it.

This is exactly how articulated structures work: a robotic arm, a human skeleton, or a solar system. The shoulder moves the upper arm, which carries the forearm, which carries the hand. Each joint adds its own rotation on top of all the rotations above it in the hierarchy.

2.4.4 What State Does push() Save?

In p5.js, push() saves all of the current drawing state, not just the transformation matrix. This includes:

  • Transformation matrix (translate, rotate, scale)
  • Fill color and fill enabled/disabled state
  • Stroke color, weight, cap, and join
  • rectMode, ellipseMode, imageMode
  • Text properties (font, size, alignment)
  • Tint settings

This is extremely useful: you can set up a completely different “drawing style” inside a push()/pop() block, and it will not affect anything outside the block.

// Global style: thin black strokes, white fill
stroke(0); strokeWeight(1); fill(255);

push();
  // Local style: thick red strokes, no fill
  stroke(255, 0, 0); strokeWeight(5); noFill();
  translate(200, 200);
  rotate(PI / 4);
  rect(0, 0, 80, 80);
pop();

// Back to thin black strokes and white fill
rect(50, 50, 80, 80);

2.4.5 Common Transformation Patterns

Several transformation patterns appear repeatedly in creative coding:

Pattern 1: Draw at center. Position a shape at a specific location and rotate around its center.

push();
translate(x, y);
rotate(angle);
rectMode(CENTER);
rect(0, 0, w, h);
pop();

Pattern 2: Orbital motion. Move a shape in a circular path around a point.

push();
translate(centerX, centerY);
rotate(orbitAngle);
translate(orbitRadius, 0);  // Move out from center
ellipse(0, 0, size, size);
pop();

Pattern 3: Scaled group. Draw a group of shapes at a different scale.

push();
translate(x, y);
scale(0.5);  // Everything inside is half-sized
drawComplexShape();
pop();
✍ Knowledge Check

What happens if you call translate(100, 0) twice in a row without using push() and pop()?

Transformations in p5.js are cumulative. Each translate() modifies the current coordinate system, so two calls of translate(100, 0) result in a total displacement of (200, 0). This is why push()/pop() are essential for isolating transformations.

2.5 Georg Nees and Algorithmic Art

The art we are learning to create with p5.js has deep roots. The first computer-generated artworks appeared in the early 1960s, when access to computers was limited to universities, military installations, and corporate research labs. A small number of pioneers recognized that the computer could be an artistic tool.

2.5.1 The Stuttgart Exhibition of 1965

In February 1965, Georg Nees (1926–2016), a German mathematician and Siemens engineer, exhibited computer-generated drawings at the Studiengalerie of the Technische Hochschule Stuttgart. This is widely considered one of the first exhibitions of computer art in the world.

Nees wrote programs in ALGOL on a Siemens mainframe. The output was rendered on a Zuse Z64 plotter — a mechanical device that moved a pen across paper according to computed coordinates. There were no screens, no real-time feedback. You wrote code, waited for the plotter to finish, and then saw the result.

📜 Historical Note

The Stuttgart exhibition was organized by philosopher Max Bense, who had developed an “information aesthetics” — a theory that beauty could be analyzed mathematically in terms of order and complexity. Bense saw Nees’ work as empirical evidence for this theory. The exhibition was met with both fascination and hostility: some viewers questioned whether a machine could create “real” art.

2.5.2 The Pioneers

Three names dominate the early history of computer art:

Georg Nees — Created Schotter (1968), Locken, and other works exploring the transition from order to disorder. His PhD thesis (1969), supervised by Max Bense, was the first doctoral dissertation on computer art.

Frieder Nake (b. 1938) — A German mathematician who independently began creating computer art in 1963. His work Hommage à Paul Klee (1965) demonstrates how algorithmic rules can produce compositions reminiscent of human painters. Nake later became a prominent theorist of computational aesthetics.

A. Michael Noll (b. 1939) — Working at Bell Telephone Laboratories in New Jersey, Noll created computer-generated patterns inspired by Piet Mondrian and Bridget Riley. His 1966 study asked viewers to distinguish between a Mondrian painting and a computer-generated imitation — most preferred the computer version, though they thought it was the Mondrian.

2.5.3 Harold Cohen and AARON

Harold Cohen (1928–2016), a British-born painter, spent decades developing AARON, an autonomous program that generated original artworks. Unlike the algorithmic artists who created rules for abstract forms, Cohen attempted to encode artistic knowledge — how to compose a figure, how to balance color, how to create depth. AARON evolved from producing simple abstract drawings in the 1970s to creating full-color figurative paintings by the 1990s.

💡 Key Concept

The early computer artists established a fundamental principle: the artist writes the rules; the computer executes them. The artwork is not any single output — it is the algorithm itself. This idea prefigures generative art, NFTs, and AI art by half a century.

2.5.4 Vera Molnár

Vera Molnár (1924–2023), a Hungarian-French artist, was another crucial pioneer. She began using computers for art in 1968 and spent decades exploring the boundary between geometric order and randomness — a theme directly related to Nees’ Schotter. Her series (Des)Ordres (1974) systematically varies the degree of disorder in geometric grids, and her work 1% de désordre demonstrates how even a tiny perturbation of a regular grid catches the eye.

Molnár’s approach was distinctive: she described her process as “machine imaginaire” — imagining what a computer would do before she had access to one. She would execute algorithms by hand, following strict rules, anticipating the computational art that would come later.

2.5.5 Schotter: Order to Disorder

Schotter (German for “gravel” or “rubble”) is Nees’ most famous work, created in 1968. It consists of a grid of approximately 12 columns and 22 rows of squares. At the top, the squares are perfectly aligned. Moving downward, each row introduces slightly more positional displacement and rotational perturbation, until the bottom rows dissolve into apparent chaos.

The piece is a visual metaphor for entropy — the second law of thermodynamics made visible. It demonstrates how a simple rule (increase randomness proportionally to row number) can produce a visually compelling gradient from order to disorder.

Nees described his approach as “generative aesthetics” — defining a set of rules and constraints, then letting the computer explore the resulting space of possibilities. The artist’s role shifts from creating individual works to designing the system that generates works. This idea would later become the foundation of generative art, computational design, and even aspects of artificial intelligence research.

📖 Definition

Generative art — Art created by a system (algorithm, set of rules, or autonomous process) that operates with some degree of autonomy, producing results that are not fully predetermined by the artist. The artist designs the system; the system creates the individual works.

2.6.5 Analyzing the Structure

Let us break down exactly what makes Schotter work as art:

  • The grid provides the order. Without the underlying grid, the displaced squares would just be random shapes on a canvas. The viewer perceives the grid as the “expected” state, making each deviation from it meaningful.
  • The gradient provides the narrative. The eye naturally scans from top to bottom, experiencing a story: order dissolves into chaos. This temporal metaphor gives the static image a sense of process and time.
  • The randomness provides the texture. No two squares in the disordered region are displaced identically. This organic variation makes the piece feel alive rather than mechanically repetitive.
💡 Key Concept

Schmidhuber’s beauty thesis: Juergen Schmidhuber proposed that humans find beauty in things that are compressible but not trivially simple. We enjoy discovering patterns — the “aha!” moment when apparent complexity resolves into an elegant rule. Schotter lives in this sweet spot: the underlying rule is simple, but the visual output is rich and varied.

2.6.6 Variations on Schotter

The Schotter algorithm is a template you can modify endlessly:

  • Radial Schotter: Arrange squares in concentric rings. Disorder increases from center outward.
  • Color Schotter: Map the displacement factor to a color gradient (blue at top to red at bottom).
  • Interactive Schotter: Remove noLoop() and tie the displacement factor to mouseY, letting the viewer control the chaos.
  • 3D Schotter: Use WEBGL mode and add z-axis displacement to create depth.
  • Temporal Schotter: Animate the displacement factor increasing over time, so the grid visibly dissolves.

2.7 Information Theory: Entropy and Compression

To understand why Schotter is interesting — why the gradient from order to disorder is compelling — we need the language of information theory, the mathematical framework invented by Claude Shannon in 1948.

2.7.1 Claude Shannon and the Birth of Information Theory

📜 Historical Note

Claude Shannon (1916–2001) published “A Mathematical Theory of Communication” in the Bell System Technical Journal in 1948. This single paper founded the field of information theory, introduced the concept of the “bit” as a unit of information, and laid the mathematical groundwork for the entire digital age. Shannon was also an accomplished juggler, unicyclist, and builder of mechanical toys.

Shannon’s key question: How much information does a message contain? He realized that information is fundamentally about surprise. A message that tells you something you already knew (or expected) carries little information. A message that tells you something unexpected carries a lot.

2.7.2 Surprisal

The surprisal (or self-information) of an event with probability $p$ is:

$$I(x) = -\log_2(p)$$

Measured in bits. Intuition:

  • A certain event ($p = 1$): surprisal $= -\log_2(1) = 0$ bits. No surprise, no information.
  • A coin flip ($p = 0.5$): surprisal $= -\log_2(0.5) = 1$ bit. One bit of information.
  • Rolling a six ($p = 1/6$): surprisal $= -\log_2(1/6) \approx 2.58$ bits.
  • A one-in-a-million event ($p = 10^{-6}$): surprisal $\approx 19.9$ bits.

2.7.3 Shannon Entropy

Entropy is the average surprisal across all possible events. For a random variable $X$ with possible outcomes $x_1, x_2, \ldots, x_n$ having probabilities $p_1, p_2, \ldots, p_n$:

📐 Shannon Entropy

$$H(X) = -\sum_{i=1}^{n} p_i \log_2(p_i)$$

Entropy $H(X)$ is measured in bits. It represents the minimum average number of yes/no questions needed to determine the outcome, or equivalently, the minimum average number of bits needed to encode the outcome.

2.7.4 Worked Examples

Example 1: Fair coin. Two outcomes, each with probability $0.5$.

$$H = -(0.5 \log_2 0.5 + 0.5 \log_2 0.5) = -(0.5 \times (-1) + 0.5 \times (-1)) = 1 \text{ bit}$$

Example 2: Biased coin (90% heads, 10% tails).

$$H = -(0.9 \log_2 0.9 + 0.1 \log_2 0.1) \approx -(0.9 \times (-0.152) + 0.1 \times (-3.322)) \approx 0.469 \text{ bits}$$

The biased coin has lower entropy because the outcome is more predictable.

Example 3: Fair six-sided die.

$$H = -\sum_{i=1}^{6} \frac{1}{6} \log_2 \frac{1}{6} = \log_2 6 \approx 2.585 \text{ bits}$$

2.7.5 Entropy and Art

Consider Schotter in information-theoretic terms:

  • Top row: Every square is in its expected position. Low surprisal, low entropy.
  • Bottom row: Each square could be anywhere within its displacement range. High surprisal, high entropy.
  • The gradient: Entropy increases monotonically from top to bottom. This is a visual representation of the second law of thermodynamics.

Schmidhuber’s thesis suggests that we find Schotter beautiful precisely because it lives in the “sweet spot”: complex enough to engage, structured enough to comprehend.

2.7.6 Properties of Entropy

Shannon entropy has several important mathematical properties:

  • Non-negativity: $H(X) \geq 0$. You cannot have negative information.
  • Maximum: For $n$ outcomes, $H(X) \leq \log_2 n$, achieved when all outcomes are equally likely (maximum uncertainty).
  • Minimum: $H(X) = 0$ when one outcome has probability 1 (complete certainty).
  • Additivity: For independent random variables, $H(X, Y) = H(X) + H(Y)$.

2.7.7 Entropy as a Lower Bound on Compression

Shannon’s source coding theorem (also called the noiseless coding theorem) establishes that entropy is the fundamental limit of lossless compression:

📐 Theorem (Shannon’s Source Coding Theorem)

The average number of bits per symbol in any lossless encoding of a source cannot be less than the source’s entropy $H$. Furthermore, there exist encodings (like Huffman coding) that approach this limit arbitrarily closely.

This means entropy tells you the absolute best you can do with compression. If the entropy is 2.5 bits per symbol, you cannot compress below 2.5 bits per symbol on average, no matter how clever your algorithm is.

2.7.8 Joint Entropy and Mutual Information

When we have two random variables (for example, the color of pixel $(x, y)$ and the color of its neighbor $(x+1, y)$), we can ask how much information they share. This leads to:

  • Joint entropy $H(X, Y)$: the total uncertainty of both variables together.
  • Conditional entropy $H(Y|X)$: the remaining uncertainty in $Y$ after observing $X$.
  • Mutual information $I(X; Y) = H(X) + H(Y) - H(X, Y)$: how much knowing $X$ tells you about $Y$.

Images are highly compressible precisely because neighboring pixels have high mutual information: knowing one pixel strongly predicts its neighbors. Random noise has zero mutual information between pixels, making it incompressible.

✍ Knowledge Check

A random variable has 4 equally likely outcomes. What is its Shannon entropy?

For $n$ equally likely outcomes, entropy is $H = \log_2 n$. With 4 outcomes: $H = \log_2 4 = 2$ bits. You need exactly 2 yes/no questions to identify which of the 4 outcomes occurred.

2.8 Kolmogorov Complexity

While Shannon entropy measures the average information in a random source, Kolmogorov complexity measures the information content of a specific individual object.

2.8.1 Formal Definition

📖 Definition

The Kolmogorov complexity $K(x)$ of a string $x$ is the length of the shortest program (in some fixed universal programming language) that produces $x$ as output and then halts.

Consider two strings of length 1000:

  • String A: 010101010101...01 (alternating 0s and 1s)
  • String B: 110010111010...01 (appears random)

String A can be generated by a very short program: “print ‘01’ 500 times.” Its Kolmogorov complexity is roughly $\log_2 500 + C$ bits, where $C$ is the overhead of the programming language. String B, if truly random, requires a program that essentially contains the entire string — its complexity is close to 1000 bits.

2.8.2 Uncomputability

📐 Theorem

Kolmogorov complexity is uncomputable. There is no algorithm that, given a string $x$, outputs $K(x)$. This follows from the halting problem: to find the shortest program, you would need to enumerate all possible programs and determine which ones halt and produce $x$, but the halting problem tells us this is impossible in general.

2.8.3 Berry's Paradox

The uncomputability of Kolmogorov complexity is related to an ancient logical paradox. Berry’s paradox asks: consider “the smallest positive integer not definable in fewer than twelve words.” This sentence has eleven words, so it defines a number in eleven words that supposedly requires twelve — a contradiction.

In Kolmogorov’s framework: suppose a program $P$ of length $n$ outputs “the first string whose Kolmogorov complexity exceeds $n$.” But $P$ itself is a program of length $n$ that produces that string, so the string’s complexity is at most $n$ — contradicting the claim that it exceeds $n$. This contradiction proves that no such program can exist, which implies that Kolmogorov complexity is uncomputable.

For generative art, Kolmogorov complexity provides a precise way to think about the relationship between a program and its output:

2.8.5 Kolmogorov Complexity and Randomness

A string is called Kolmogorov random (or algorithmically random) if its Kolmogorov complexity is approximately equal to its length: $K(x) \geq |x| - c$ for some small constant $c$. In other words, there is no shorter description of the string than the string itself. It is incompressible.

This gives us a formal definition of randomness that does not depend on probability theory. A string is random if it has no patterns, no structure, no regularity that could be exploited to produce a shorter description. This is a remarkably clean definition: randomness is simply incompressibility.

An important consequence: you can never prove that a specific string is Kolmogorov random, because you can never be certain that no shorter program exists. But you can prove that a string is not random by exhibiting a short program that generates it. This asymmetry mirrors our experience with art: it is easy to show that something has structure (by describing the pattern), but difficult to prove that something is truly “random” or structureless.

💡 Key Concept

Most strings are random. For strings of length $n$, there are $2^n$ possible strings but fewer than $2^k$ programs of length less than $k$ (for any $k < n$). Therefore, only a vanishingly small fraction of strings can have low Kolmogorov complexity. The vast majority of strings are incompressible.

2.8.6 The Invariance Theorem

One might worry that Kolmogorov complexity depends on the choice of programming language. The invariance theorem resolves this: for any two universal programming languages $L_1$ and $L_2$, there exists a constant $c$ (depending only on the languages, not on the string) such that:

$$|K_{L_1}(x) - K_{L_2}(x)| \leq c$$

The constant $c$ is essentially the size of a compiler/interpreter from one language to the other. For long strings, this constant becomes negligible, so Kolmogorov complexity is essentially language-independent.

2.8.7 Practical Implications for Artists

While Kolmogorov complexity is uncomputable in theory, as an artist you can use the concept productively:

  • Source code length is an upper bound on complexity. A 500-byte program that produces a visually rich image is achieving a high ratio of “visual complexity” to “descriptive complexity.”
  • Compression ratio of the output approximates its complexity. Save your output as PNG; the file size estimates the information content.
  • Code golf (writing the shortest possible program) is a creative constraint that directly engages with Kolmogorov complexity. What is the shortest draw() function that produces something beautiful?

2.9 Determinism and Pseudorandom Numbers

Computers are deterministic machines: given the same inputs, they produce the same outputs. Yet art thrives on variation, surprise, and organic irregularity. The bridge between these worlds is the pseudorandom number generator (PRNG).

2.9.1 What Is a PRNG?

📖 Definition

A pseudorandom number generator (PRNG) is a deterministic algorithm that, given an initial seed, produces a sequence of numbers that passes statistical tests for randomness. The same seed always produces the same sequence.

2.9.2 Linear Congruential Generator (LCG)

The oldest widely-used PRNG formula, dating to 1951 (Lehmer):

$$x_{n+1} = (a \cdot x_n + c) \mod m$$

where $a$, $c$, and $m$ are carefully chosen constants and $x_0$ is the seed. The classic C standard library uses $a = 1103515245$, $c = 12345$, $m = 2^{31}$.

// Linear Congruential Generator
let state = seed;

function nextRandom() {
  state = (state * 1103515245 + 12345) & 0x7FFFFFFF;
  return state / 0x7FFFFFFF; // Normalize to [0, 1)
}

LCGs are fast but have known weaknesses: the low-order bits cycle with short periods, and sequential outputs can fall on a small number of hyperplanes in higher dimensions.

2.9.3 Xorshift

A more modern approach, invented by George Marsaglia in 2003:

function xorshift32(state) {
  state ^= state << 13;  // XOR with left-shifted self
  state ^= state >> 17;  // XOR with right-shifted self
  state ^= state << 5;   // XOR with left-shifted self
  return state >>> 0;     // Force unsigned 32-bit integer
}

Each XOR-shift operation mixes the bits. After three operations, every output bit depends on many input bits (the “avalanche effect”), making the output appear random. Xorshift is faster than LCG and produces better statistical distributions.

2.9.4 Seeds in p5.js

randomSeed(42);          // Fix the sequence
let a = random(100);     // Always the same for seed 42
let b = random(100);     // Always the same next value

// Without randomSeed(), p5.js derives the seed
// from system entropy (time, hardware state),
// so each run differs.

2.9.5 Why Determinism Matters for Art

  • Reproducibility: Record the seed to recreate the exact composition later.
  • Exploration: Systematically try seeds to find the best output.
  • Sharing: Share code + seed; anyone can verify the result.
  • Generative NFTs: The token ID is the seed. Each token produces a unique, permanent artwork.

2.9.6 Noise Functions: Structured Randomness

Pure random numbers (random()) produce harsh, jagged variation. For many artistic applications, we want smooth randomness — values that change gradually rather than jumping erratically. p5.js provides noise(), an implementation of Perlin noise, invented by Ken Perlin in 1983, inspired by his work on the visual effects for the film Tron (1982) at MAGI/Synthavision.

// random() -- harsh, uncorrelated values
let y1 = random(height);  // Each call is independent

// noise() -- smooth, correlated values
let y2 = noise(x * 0.01) * height;  // Nearby x values give nearby y values

Perlin noise has a fractal-like quality: you can layer multiple “octaves” of noise at different frequencies and amplitudes to produce organic-looking textures for terrain, clouds, fire, and water.

2.9.7 True Randomness vs Pseudorandomness

For most artistic applications, pseudorandom numbers are indistinguishable from true randomness. But it is worth knowing the difference:

  • True random numbers come from physical processes: radioactive decay, atmospheric noise, thermal fluctuations. Services like random.org provide them via atmospheric noise.
  • Pseudorandom numbers are deterministic sequences that appear random. They are periodic (eventually the sequence repeats), but the period can be astronomical (the Mersenne Twister has a period of $2^{19937} - 1$).
  • Cryptographically secure PRNGs (CSPRNGs) are pseudorandom generators designed so that even knowing part of the sequence does not help predict the rest. Used for encryption, passwords, and security-critical applications. Slower than regular PRNGs.

For creative coding, a standard PRNG is perfectly adequate. The key property you need is reproducibility (same seed, same output), not cryptographic security.

✍ Knowledge Check

If you call randomSeed(7) at the start of draw() and the sketch runs for 1000 frames, what happens?

Calling randomSeed(7) at the start of each draw() call resets the PRNG to the same initial state every frame. Every call to random() within that frame produces the same value it produced in every other frame. The visual output appears frozen — identical each frame.

2.10 Recursion: Theory and Practice

Recursion is one of the most powerful ideas in computer science. A recursive function is one that calls itself, breaking a problem into smaller instances of the same problem until reaching a trivial base case.

2.10.1 Anatomy of a Recursive Function

Every recursive function has exactly two essential components:

  1. Base case (termination condition): The simplest instance of the problem that can be solved directly, without making any further recursive calls. This is what prevents infinite recursion. Without a base case, the function would call itself forever.
  2. Recursive case (reduction step): Reduces the current problem to one or more strictly smaller instances of the same problem and combines their results to produce the answer. The key requirement is that each recursive call must make progress toward the base case — typically by operating on a smaller number, a shorter list, or a shallower tree.
💡 Key Concept

The leap of faith: When writing recursive functions, trust that the recursive call correctly solves the smaller subproblem. Focus on (1) defining the base case, (2) showing that the recursive case reduces to a smaller version, and (3) combining the sub-result correctly. This “recursive leap of faith” is the key mental skill for thinking recursively.

function factorial(n) {
  if (n === 0) return 1;        // Base case
  return n * factorial(n - 1);  // Recursive case
}
// factorial(5) = 5 * 4 * 3 * 2 * 1 * 1 = 120

2.10.2 The Call Stack

When a function calls itself, the computer must remember where it was in the previous call. It does this using a call stack — a region of memory organized as a stack (LIFO). Each function call creates a new stack frame containing the function’s local variables, parameters, and return address.

For factorial(4), the call stack grows like this:

factorial(4)  -- waits for factorial(3)
  factorial(3)  -- waits for factorial(2)
    factorial(2)  -- waits for factorial(1)
      factorial(1)  -- waits for factorial(0)
        factorial(0)  -- returns 1 (base case!)
      returns 1 * 1 = 1
    returns 2 * 1 = 2
  returns 3 * 2 = 6
returns 4 * 6 = 24
⚠ Common Mistake

Stack overflow: If the base case is missing or never reached, the function calls itself forever, filling the call stack until it exceeds the memory limit. In JavaScript, this produces a “Maximum call stack size exceeded” error. Always ensure your recursive case makes progress toward the base case.

2.10.3 Linear vs Tree Recursion

Linear recursion: Each call makes at most one recursive call. Example: factorial. The call chain is a straight line. Time complexity: $O(n)$.

Tree recursion: Each call makes two or more recursive calls. Example: Fibonacci. The calls branch like a tree.

function fib(n) {
  if (n <= 1) return n;             // Base cases
  return fib(n - 1) + fib(n - 2);  // TWO recursive calls
}

The number of calls grows exponentially: $O(2^n)$ for naive Fibonacci. fib(30) makes over a million calls. fib(50) would take years. This is why memoization or iterative solutions are important for tree-recursive problems — but the naive recursive form is valuable for understanding the concept.

Any recursive algorithm can be rewritten as an iterative one (using loops and an explicit stack), and vice versa. So why use recursion?

For visual art, recursion is often the natural choice because the structures we want to draw (trees, fractals, nested patterns) are inherently recursive. The code mirrors the visual structure, making it easier to understand and modify.

2.10.6 Tail Recursion

A recursive function is tail recursive if the recursive call is the very last operation in the function — nothing else happens after the recursive call returns. Tail-recursive functions can be optimized by the compiler/interpreter into loops (called tail call optimization or TCO), eliminating the stack overhead.

// NOT tail recursive: multiplication happens AFTER the recursive call
function factorial(n) {
  if (n === 0) return 1;
  return n * factorial(n - 1);  // Must multiply after return
}

// Tail recursive version: uses an accumulator
function factorialTail(n, acc = 1) {
  if (n === 0) return acc;
  return factorialTail(n - 1, n * acc);  // Last operation is the call
}
⚠ Common Mistake

JavaScript engines are not required to implement tail call optimization (only Safari does as of 2024). In practice, even tail-recursive functions in JavaScript can cause stack overflows for large inputs. For deep recursion (thousands of levels), consider an iterative approach or increase the recursion limit.

✍ Knowledge Check

What happens if you write a recursive function without a base case?

Without a base case, the function calls itself endlessly, each call adding a new frame to the call stack. The stack has a finite size (typically a few thousand frames in JavaScript), so eventually the runtime throws a “Maximum call stack size exceeded” error. This is called a stack overflow.

2.11 Recursive Art: Trees, Fractals, L-Systems

Recursion is not just a programming technique — it is the language of self-similarity, the property that a part of a structure resembles the whole. Nature is filled with self-similar structures: the branching of trees mirrors the branching of their individual limbs; a fern frond is composed of smaller fronds that are miniature copies of the whole; coastlines look equally jagged whether viewed from a satellite or a cliff top; blood vessels branch and re-branch at every scale from the aorta to the capillaries. Recursive algorithms express these forms directly, making them the natural tool for computational artists working with organic and fractal forms.

2.11.1 Fractal Trees

A fractal tree is drawn by a recursive function that: (1) draws a branch, (2) moves to its tip, (3) calls itself twice (or more) at different angles with a shorter length. The base case stops when the branch is too short.

2.11.2 The Sierpinski Triangle

The Sierpinski triangle (or Sierpinski gasket) is a fractal named after the Polish mathematician Wacław Sierpiński. It is constructed by repeatedly removing the central triangle from an equilateral triangle:

  1. Start with a filled equilateral triangle.
  2. Find the midpoints of the three sides. Connect them to form an inner triangle. Remove the inner triangle.
  3. Repeat step 2 for each of the three remaining smaller triangles.
  4. Continue to desired depth.

2.11.3 The Koch Curve

The Koch curve (Helge von Koch, 1904) is one of the earliest mathematically described fractals, published decades before the word “fractal” existed. It is constructed by repeatedly replacing the middle third of each line segment with two sides of an equilateral triangle:

  1. Start with a straight line segment.
  2. Divide it into three equal parts.
  3. Replace the middle third with two sides of an equilateral triangle (pointing outward).
  4. Repeat for every line segment.

At each iteration, the number of line segments multiplies by 4, but each segment is 1/3 the previous length. After $n$ iterations, there are $4^n$ segments, each of length $(1/3)^n$ times the original.

After infinite iterations, the Koch curve has infinite length (since $4/3 > 1$, the total length grows without bound) but encloses a finite area (bounded by the original triangle). Its fractal dimension is:

$$D = \frac{\log 4}{\log 3} \approx 1.2619$$

This non-integer dimension captures the intuition that the Koch curve is “more than a line but less than a plane.” It fills more space than a 1D curve but less than a 2D region.

📜 Historical Note

The word fractal was coined by Benoit Mandelbrot in 1975, from the Latin fractus (broken, fragmented). Mandelbrot’s 1982 book The Fractal Geometry of Nature brought fractals to wide attention and demonstrated their ubiquity in natural phenomena: coastlines, clouds, mountains, blood vessels, and galaxy distributions.

2.11.4 L-Systems (Lindenmayer Systems)

In 1968, the Hungarian biologist Aristid Lindenmayer invented L-systems as a mathematical formalism for modeling plant growth. An L-system consists of:

  • An alphabet of symbols (e.g., F, +, -, [, ])
  • An axiom — the starting string
  • Production rules — replacement rules applied in parallel
💡 Key Concept

L-system turtle interpretation: F = move forward and draw, + = turn right, - = turn left, [ = push state (position + angle), ] = pop state. By choosing different rules and angles, L-systems can produce an extraordinary variety of plant-like forms.

Example: Koch curve as an L-system.

  • Axiom: F
  • Rule: F → F+F--F+F
  • Angle: 60 degrees

After 0 iterations: F
After 1 iteration: F+F--F+F
After 2 iterations: F+F--F+F+F+F--F+F--F+F--F+F+F+F--F+F

Each iteration replaces every F with the full rule, producing exponentially longer strings that encode increasingly detailed curves.

2.11.5 L-System Plant Example

Here is a more elaborate L-system that produces a plant-like structure. The key additions are the bracket symbols [ and ], which save and restore the turtle’s state (position and heading), creating branches.

Fractal Plant L-system:

  • Axiom: X
  • Rules: X → F+[[X]-X]-F[-FX]+X, F → FF
  • Angle: 25 degrees

After just 5 iterations, this system produces a remarkably realistic branching structure from these two simple rules.

2.11.6 Why Self-Similarity Appears in Nature

The prevalence of self-similar structures in nature is not a coincidence. There are deep reasons:

  • Efficiency: Branching structures (trees, lungs, blood vessels) maximize surface area within a volume. A recursive branching pattern achieves this efficiently.
  • Growth by local rules: Biological organisms grow by cells following local rules. An L-system models this: each symbol (cell) is replaced according to its own rule, in parallel, without global coordination.
  • Fractal dimension: Natural coastlines, mountain ranges, and cloud boundaries have fractal dimensions between 1 and 2, meaning they are rougher than a line but smoother than a plane. This intermediate complexity is what makes natural scenes visually rich.
💡 Key Concept

Fractal dimension quantifies the “roughness” of a fractal. A straight line has dimension 1, a filled square has dimension 2, and the Koch curve has dimension $\log 4 / \log 3 \approx 1.26$. The Sierpinski triangle has dimension $\log 3 / \log 2 \approx 1.585$. These non-integer dimensions capture the idea that fractals occupy “more space” than simple curves but less than filled regions.

2.11.7 Recursion Depth and Computational Cost

The fractal tree example demonstrates an important relationship between recursion depth and computational cost. Starting from a branch length of 120 pixels and shrinking by 30% (multiplying by 0.7) each level:

Level 0: 120 px   (1 branch)
Level 1: 84 px    (2 branches)
Level 2: 58.8 px  (4 branches)
Level 3: 41.2 px  (8 branches)
...
Level 9: 4.8 px   (512 branches)
Level 10: 3.4 px  (1024 branches -- base case)

Total branches drawn: $1 + 2 + 4 + \ldots + 512 = 2^{10} - 1 = 1023$ branches plus 1024 leaves. The tree looks enormously complex, but the program that generates it is only about 20 lines. This is Kolmogorov complexity in action: a short program producing high visual complexity.

✍ Knowledge Check

In a recursive fractal tree that splits into 2 branches at each level, how many total branches are drawn at depth 8?

At each level $k$, we draw $2^k$ branches. Total is $\sum_{k=0}^{7} 2^k = 2^8 - 1 = 255$. This is a geometric series where doubling at each level leads to exponential growth, which is why recursive fractals must have a base case to stop.

2.12 Bitwise Operations and Enumeration

At the lowest level, computers store and manipulate data as sequences of bits — binary digits that are either 0 or 1. Everything in a computer — numbers, text, images, programs — is ultimately represented as patterns of bits. Bitwise operations manipulate these bits directly, giving us a powerful and efficient toolkit for enumeration, masking, and combinatorial problems.

Understanding bitwise operations connects you to the fundamental nature of computation. When Nees programmed his early works on the Siemens mainframe, he was much closer to the bit level than we are today. Modern high-level languages like JavaScript abstract away the bits, but they remain accessible through bitwise operators — and they enable elegant solutions that would be cumbersome with arithmetic alone.

2.12.2 Checking a Specific Bit

The fundamental technique for enumeration: check whether bit $k$ is set in number $n$:

let isSet = (n & (1 << k)) !== 0;

1 << k creates a mask with only bit $k$ set. The AND operation isolates that bit.

2.12.3 Common Bitwise Patterns

Several bitwise patterns appear frequently in creative coding and algorithms:

// Check if bit k is set in n
let isSet = (n & (1 << k)) !== 0;

// Set bit k in n
n = n | (1 << k);

// Clear bit k in n
n = n & ~(1 << k);

// Toggle bit k in n
n = n ^ (1 << k);

// Check if n is a power of 2
let isPow2 = (n & (n - 1)) === 0 && n > 0;

// Count set bits (population count)
function popcount(n) {
  let count = 0;
  while (n) { count += n & 1; n >>>= 1; }
  return count;
}

2.12.4 Enumeration: All Subsets

If you have $n$ elements, each either included or excluded, there are $2^n$ possible subsets. You represent each subset as a number from 0 to $2^n - 1$, where bit $k$ indicates whether element $k$ is included.

For example, with 4 edges of a square:

  • 0 = 0000 — no edges drawn
  • 1 = 0001 — top edge only
  • 5 = 0101 — top and bottom edges
  • 15 = 1111 — all four edges

This is an incredibly compact representation. A single 32-bit integer can represent any subset of up to 32 elements. Operations like “is element $k$ in the subset?” take a single CPU instruction.

2.12.5 Sol LeWitt's Incomplete Open Cubes

📜 Historical Note

Sol LeWitt (1928–2007) was an American artist widely regarded as a founder of Conceptual Art and Minimalism. His 1974 work Incomplete Open Cubes systematically enumerates all ways to remove edges from a wireframe cube while keeping the structure connected and three-dimensional. A cube has 12 edges, giving $2^{12} = 4096$ possible subsets. LeWitt identified 122 valid configurations. The work exemplifies art as exhaustive enumeration — an algorithm executed by hand.

LeWitt’s process is essentially a computational algorithm, which we can express precisely:

  1. For each of the 4096 subsets of 12 edges:
    1. Check if the remaining edges form a connected graph.
    2. Check if the structure spans all three dimensions (not flat).
    3. If both conditions are met, include it in the collection.
  2. Remove configurations that are rotational duplicates.
  3. Display all valid, unique configurations.

A computer can perform this enumeration in milliseconds. LeWitt did it by hand — which is itself a kind of performance art. The work prefigures computational thinking: defining precise rules and exhaustively exploring the resulting space.

2.13 Data Compression: RLE and Huffman

Data compression turns theoretical complexity into practical technology. We now examine two foundational compression algorithms in detail.

2.13.1 Run-Length Encoding (RLE)

The simplest compression algorithm. Replace runs of identical values with a count and the value:

AAAAABBBCC  -->  5A3B2C    (10 chars --> 6 chars, 60%)
ABCDEFGH    -->  1A1B1C1D1E1F1G1H  (8 chars --> 16 chars, 200% -- WORSE!)

RLE can make data larger if there are no runs. Always check whether the compressed version is actually smaller than the original. This is why real-world formats combine multiple techniques.

2.13.2 Huffman Coding

David Huffman invented this algorithm as a graduate student at MIT in 1952 (it was a homework assignment). Huffman coding assigns variable-length binary codes to symbols: frequent symbols get short codes, rare symbols get long codes.

The Algorithm:

  1. Count the frequency of each symbol.
  2. Create a leaf node for each symbol, weighted by frequency.
  3. Repeatedly merge the two lowest-weight nodes into a parent. The parent’s weight is the sum of its children.
  4. Continue until one root node remains. This is the Huffman tree.
  5. The code for each symbol is the path from root to leaf: left = 0, right = 1.

Worked Example: Encode the string ABRACADABRA.

SymbolFrequencyHuffman CodeFixed-Width Code
A50 (1 bit)000 (3 bits)
B210 (2 bits)001 (3 bits)
R2110 (3 bits)010 (3 bits)
C11110 (4 bits)011 (3 bits)
D11111 (4 bits)100 (3 bits)

Fixed-width encoding: 11 symbols × 3 bits (need 3 bits for 5 symbols) = 33 bits.
Huffman encoding: $5 \times 1 + 2 \times 2 + 2 \times 3 + 1 \times 4 + 1 \times 4 = 23$ bits.

Huffman achieves 30% compression on this small example. For English text with its highly unequal letter frequencies, the savings are even greater.

The key property that makes Huffman coding decodable is the prefix-free property: no codeword is a prefix of any other codeword. For example, if “A” is coded as 0, then no other symbol’s code can start with 0. This means the decoder can read the bitstream left to right and unambiguously determine where each codeword ends.

Building the Huffman Tree step by step for ABRACADABRA:

  1. Start with five leaf nodes: A(5), B(2), R(2), C(1), D(1).
  2. Merge the two smallest: C(1) and D(1) become CD(2).
  3. Merge the two smallest: B(2) and CD(2) become BCD(4). (Or R(2) and CD(2) — ties can be broken arbitrarily.)
  4. Merge R(2) and BCD(4) to get RBCD(6). (Or merge R(2) with something else — multiple valid trees exist.)
  5. Merge A(5) and RBCD(6) to get the root ARBCD(11).

Different tie-breaking choices produce different trees, but all valid Huffman trees for the same frequency distribution achieve the same optimal average code length.

📐 Theorem

Optimality of Huffman coding: Among all prefix-free codes (where no codeword is a prefix of another), Huffman coding produces the shortest average codeword length. The average length approaches the Shannon entropy $H$ as a lower bound.

2.13.3 How PNG Combines Techniques

Real-world image compression (like PNG) combines multiple layers:

  1. Row filtering: Each row is compared to the row above (delta encoding), producing smaller values for smooth gradients.
  2. LZ77: Finds repeated sequences and replaces them with back-references to earlier occurrences.
  3. Huffman coding: Encodes the LZ77 output with optimal variable-length codes.

This is why PNG excels at geometric art and screenshots (lots of repeated patterns) but performs poorly on photographs (use JPEG) or random noise (incompressible).

2.13.4 Compression as a Complexity Measure

Since Kolmogorov complexity is uncomputable, compression provides a practical substitute. The compressed size of a file is an upper bound on its Kolmogorov complexity. The closer the compressed size is to the original, the more “random” (complex) the data.

This gives us a concrete tool for analyzing art:

  • Create an image in p5.js and save it as PNG.
  • The PNG file size approximates the image’s Kolmogorov complexity.
  • Compare to the source code size. The ratio of visual complexity (PNG size) to descriptive complexity (code size) is a measure of how much “amplification” your program achieves.
💡 Key Concept

Compression ratio as art metric: A generative art program is like a compression algorithm running in reverse. Instead of finding the shortest description of existing data, it expands a short description (the code) into rich visual output. The best generative art achieves the highest “amplification factor” — the most visual complexity per byte of source code.

2.13.5 Lossless vs Lossy Compression

Compression algorithms fall into two categories:

Lossless compression (PNG, ZIP, FLAC) preserves every bit of the original data. The decompressed output is identical to the input. Used when exactness matters: text, source code, medical images.

Lossy compression (JPEG, MP3, H.264) discards information that is “less important” to human perception. A JPEG photo looks nearly identical to the original but is 10–50x smaller. Used when perceptual quality matters more than bit-perfect accuracy: photographs, music, video.

For generative art, the distinction is significant. If you save a p5.js canvas as PNG (lossless), the file size reflects the actual information content. If you save as JPEG (lossy), the file size reflects the perceptually relevant information content, which is always less.

2.13.6 The Pigeonhole Principle and Compression Limits

A fundamental result: no compression algorithm can compress all possible inputs. Here is why:

📐 Theorem (Pigeonhole Principle for Compression)

If a compression algorithm maps every $n$-bit string to a string of fewer than $n$ bits, then by the pigeonhole principle, some compressed strings must correspond to multiple originals, making decompression ambiguous. Therefore, any lossless compression algorithm must expand some inputs.

This is why RLE makes ABCDEFGH longer: the algorithm compresses repetitive data at the cost of expanding non-repetitive data. No free lunch.

✍ Knowledge Check

Why can't a lossless compression algorithm make every input smaller?

If you try to map every $n$-bit string to fewer than $n$ bits, there are more possible inputs ($2^n$) than possible outputs ($< 2^n$). By the pigeonhole principle, multiple inputs would map to the same output, making it impossible to decompress unambiguously. Therefore, any lossless algorithm must expand some inputs.

2.14 Anemic Cinema and Kinetic Art

2.14.1 Marcel Duchamp: Beyond the Readymade

📜 Historical Note

Marcel Duchamp (1887–1968) is one of the most influential artists of the 20th century. He is best known for his “readymades” — everyday objects (a urinal, a bicycle wheel) presented as art. But Duchamp was also deeply interested in motion, optics, and mechanical systems, which led him to create kinetic artworks decades before the Op Art movement.

2.14.2 Anemic Cinema (1926)

In collaboration with Man Ray and Marc Allégret, Duchamp created Anemic Cinema — a seven-minute film alternating between spinning discs bearing spiral patterns (which produce optical illusions of depth) and rotating discs displaying French puns arranged in spirals. The film was produced under Duchamp’s pseudonym “Rrose Sélavy” (a pun on “Eros, c’est la vie” — “Eros, that’s life”).

The title itself is an anagram: “anemic” rearranges to “cinema.” This reflects Duchamp’s love of wordplay and his wry commentary on the new medium of film. The film is both a genuine experiment in optical illusion and a conceptual joke — a characteristic Duchamp duality that foreshadowed decades of conceptual art.

2.14.3 Rotoreliefs (1935)

Duchamp’s Rotoreliefs are flat cardboard discs printed with patterns of eccentric circles, spirals, and geometric forms. When placed on a turntable spinning at 33⅓ RPM, the flat patterns create vivid illusions of three-dimensional cones, funnels, and undulating surfaces.

The key principle: a simple set of rules (geometry + constant rotation) produces an emergent experience (perceived depth and motion) that is far richer than the input. This is the same principle that makes generative art compelling.

2.14.4 The Op Art Connection

Duchamp’s optical experiments preceded the Op Art (Optical Art) movement of the 1960s by decades. Op Art artists like Bridget Riley and Victor Vasarely created paintings that produce optical illusions through precise geometric patterns. The movement was formalized by the 1965 MoMA exhibition The Responsive Eye.

Op Art shares a fundamental principle with generative art: a precise set of rules (geometric relationships, color contrasts) produces perceptual experiences that transcend the physical medium. The rules are simple; the experience is complex.

2.14.5 Adaptation as Translation

Moving art between media is an act of translation. You must decide what is essential to the work and what is merely a feature of the original medium:

AspectEssential?Reasoning
The geometric patternYesWithout the pattern, there is no optical illusion

The physical cardboard discNoThe disc is merely a delivery mechanism
Specific turntable speedPartiallyThe effect works at various speeds, but speed affects the experience

When we write p5.js code for a rotorelief, we preserve the pattern and rotation while discarding the physical medium. But we also add something new: interactivity. The digital version can let the viewer control speed, eccentricity, color, and number of rings — parameters that were fixed in Duchamp’s original.

2.14.6 Using LLMs as Creative Partners

Large Language Models represent a new link in the adaptation chain:

Original artwork → Description of rules → LLM prompt → Generated code → Digital artwork

The artistic decisions remain with you: what to create, how to evaluate results, when to iterate. The LLM handles the translation from natural language to code. This is analogous to Sol LeWitt writing instructions and assistants executing them, or Duchamp choosing a readymade and the gallery context doing the rest.

Tips for working with LLMs on creative code:

  • Describe the effect you want, not just the shapes.
  • Provide constraints (canvas size, color palette, number of elements).
  • Iterate: run the code, evaluate, describe changes.
  • Understand the generated code so you can modify it yourself.

2.15 Summary

This chapter covered the mathematical and conceptual foundations of generative art:

  1. p5.js descends from Processing (Reas & Fry, 2001, MIT Media Lab). It provides an accessible creative coding environment in the browser.
  2. CPU vs GPU: CPUs process sequentially (few powerful cores); GPUs process in parallel (thousands of small cores). Default p5.js uses the CPU; WebGL and shaders use the GPU.
  3. Geometric transformations (translate, rotate, scale) work by moving the coordinate system, not individual shapes. They are represented internally as matrix multiplication.
  4. push() and pop() save and restore the transformation state, enabling independent and nested transformations. They use a stack data structure (LIFO).
  5. Algorithmic art pioneers (Nees, Nake, Noll, Cohen) established that the algorithm is the artwork. Georg Nees’ Schotter (1968) visualizes the transition from order to disorder.
  6. Shannon entropy $H = -\sum p_i \log_2 p_i$ measures the average information content of a random source. High entropy means high unpredictability.
  7. Kolmogorov complexity $K(x)$ is the length of the shortest program that produces $x$. It is uncomputable but can be approximated by compression. Generative art thrives in the sweet spot: low program complexity, high visual complexity.
  8. PRNGs are deterministic algorithms that produce random-looking sequences from a seed. Seeds make generative art reproducible.
  9. Recursion (base case + recursive case) naturally expresses self-similar structures: fractal trees, Sierpinski triangles, Koch curves, and L-systems.
  10. Bitwise operations and enumeration let you systematically explore all configurations (subsets), as in Sol LeWitt’s Incomplete Open Cubes.
  11. Data compression (RLE, Huffman, PNG) exploits patterns to reduce data size and provides a practical measure of complexity.
  12. Kinetic art (Duchamp’s rotoreliefs, Op Art) demonstrates how simple geometric rules plus motion produce emergent perceptual experiences — the same principle underlying generative art.

These ideas form an interconnected web. Transformations enable the structured compositions that Nees and Molnár pioneered. Recursion generates the self-similar forms that Mandelbrot identified everywhere in nature. Information theory gives us the vocabulary to discuss why some images are interesting (moderate entropy) while others are boring (zero entropy) or overwhelming (maximum entropy). Compression connects information theory to practical computing. And the historical thread from Duchamp through Nees to modern generative art shows how these mathematical concepts have always been intertwined with artistic vision.

As you work through the exercises, pay attention to the relationships between code length (Kolmogorov complexity), visual richness (apparent complexity), and the transformation/recursion techniques that amplify simple rules into complex outputs. This amplification — short programs producing rich results — is the essence of computing arts.

2.16 Glossary

Affine transformation
A geometric transformation (translation, rotation, scaling, shear) that preserves parallel lines. Represented by matrix multiplication in homogeneous coordinates.
Base case
The condition in a recursive function that stops the recursion and returns a value directly.
Bit
The fundamental unit of information, representing a single binary digit (0 or 1). Named by Claude Shannon as a contraction of “binary digit.”
Bitwise operation
An operation that manipulates individual bits of a number: AND, OR, XOR, NOT, shifts.
Call stack
A region of memory that stores function call frames in LIFO order. Each recursive call adds a frame; returning removes one.
Entropy (Shannon)
A measure of the average information content (unpredictability) of a random variable. $H = -\sum p_i \log_2 p_i$.
Fractal
A geometric object exhibiting self-similarity at different scales. Examples: Koch curve, Sierpinski triangle, fractal trees.
Homogeneous coordinates
A coordinate system that adds an extra dimension (e.g., $(x, y, 1)$ for 2D) to represent translations as matrix multiplications.
Huffman coding
An optimal prefix-free compression algorithm that assigns shorter binary codes to more frequent symbols.
Kolmogorov complexity
The length of the shortest program that produces a given string. Uncomputable but approximable by compression.
L-system (Lindenmayer system)
A string-rewriting system used to model plant growth and generate fractal curves.
Linear Congruential Generator (LCG)
A PRNG defined by $x_{n+1} = (ax_n + c) \bmod m$. Simple and fast but with known statistical weaknesses.
PRNG (Pseudorandom Number Generator)
A deterministic algorithm producing a sequence that appears random. Same seed yields same sequence.
Recursion
A technique where a function calls itself with simpler inputs. Requires a base case to terminate.
Rotorelief
A flat disc with geometric patterns that creates optical illusions of depth when rotated. Invented by Marcel Duchamp (1935).
Run-Length Encoding (RLE)
A compression algorithm replacing runs of identical values with a count and the value.
Seed
The initial value given to a PRNG that determines the entire sequence of pseudorandom numbers.
Self-similarity
A property of objects whose parts resemble the whole at different scales.
Stack frame
A block of memory on the call stack containing a function’s local variables, parameters, and return address.
Stack overflow
An error caused by exceeding the call stack’s memory limit, typically from infinite or excessively deep recursion.
Surprisal
The information content of a single event: $I(x) = -\log_2 p(x)$ bits.
Transformation matrix
A matrix that, when multiplied by a coordinate vector, applies a geometric transformation.
Vera Molnár
Hungarian-French artist (1924–2023) who pioneered computer-generated geometric art, exploring the boundary between order and randomness.
Xorshift
A family of fast PRNGs based on XOR and bit-shift operations, invented by George Marsaglia (2003).
Perlin noise
A gradient noise function invented by Ken Perlin (1983) that produces smooth, natural-looking randomness. Available in p5.js as noise().
Op Art
Optical Art — an art movement of the 1960s that creates visual illusions through precise geometric patterns. Key artists: Bridget Riley, Victor Vasarely.
Pigeonhole principle
If $n$ items are placed into $m$ containers with $n > m$, at least one container must hold more than one item. Applied to compression: not all files can be made shorter.
Source coding theorem
Shannon’s theorem stating that entropy is the fundamental limit of lossless compression. No encoding can average fewer than $H$ bits per symbol.
Tail recursion
A form of recursion where the recursive call is the last operation in the function, enabling potential optimization into a loop.
Fractal dimension
A non-integer measure of a fractal’s geometric complexity. The Koch curve has dimension $\approx 1.26$; the Sierpinski triangle has dimension $\approx 1.585$.

✍ Question 1

In p5.js, what does rectMode(CENTER) change about how rectangles are drawn?

By default, rect(x, y, w, h) positions the rectangle with its top-left corner at (x, y). After calling rectMode(CENTER), the (x, y) values specify the center of the rectangle. This is essential when rotating shapes around their centers.

✍ Question 2

Georg Nees’ Schotter was first exhibited in which decade?

Georg Nees created Schotter in 1968, and his first computer art exhibition was at the University of Stuttgart in 1965. He was one of the earliest pioneers of algorithmic art.

✍ Question 3

What is the Kolmogorov complexity of a truly random 1000-bit string?

A truly random string has no compressible patterns. The shortest program to produce it must essentially contain the string itself (plus a small constant for the “print” instruction). Therefore, its Kolmogorov complexity is approximately equal to its length.

✍ Question 4

How many leaf nodes does a binary fractal tree have if it recurses to depth 8?

A binary tree (each branch splits into 2) at depth $d$ has $2^d$ leaf nodes. At depth 8: $2^8 = 256$ leaves. This exponential growth is characteristic of tree recursion and is why fractal trees look so complex despite being generated by short programs.

✍ Question 5

Which compression algorithm assigns shorter binary codes to more frequent symbols?

Huffman coding builds a binary tree where frequent symbols are near the root (short codes) and rare symbols are far from the root (long codes). RLE replaces runs of identical values. LZ77 finds repeated sequences. Base64 is an encoding, not a compression algorithm.
