15.2 Object-Oriented Programming (OOP)

Object-oriented programming is the single most important organizational tool for creative coding. Every particle system, every flock of agents, every game character in this course has been built using OOP principles. The core idea is simple: instead of managing dozens of loose variables, you bundle related data and behavior into a single class, and then create as many instances (objects) of that class as you need.

📖 Definition

Class — A blueprint or template that defines the properties (data) and methods (behavior) that a type of object will have. Think of it as a cookie cutter: the cutter defines the shape, but each cookie is its own independent thing.

Object (Instance) — A specific thing created from a class. Each instance has its own copy of the properties defined by the class, but shares the method definitions.

15.2.1 Classes, Constructors, and this

A class definition uses the class keyword. Inside it, the constructor() method runs once when you create a new instance with new. The keyword this refers to the specific instance being created or acted upon—it is what distinguishes “this ball’s x position” from “that ball’s x position.”

class Particle {
  // Constructor: runs once when you create a new instance
  constructor(x, y) {
    this.x = x;             // Property: this particle's x position
    this.y = y;             // Property: this particle's y position
    this.vx = random(-2, 2); // Random horizontal velocity
    this.vy = random(-3, 0); // Random upward velocity
    this.lifespan = 255;     // Also used as alpha transparency
    this.size = random(3, 10);
  }

  // Method: update this particle's state
  update() {
    this.x += this.vx;
    this.y += this.vy;
    this.vy += 0.05;       // Gravity pulls it down each frame
    this.lifespan -= 3;    // Fade out over time
  }

  // Method: draw this particle
  display() {
    noStroke();
    fill(255, 100, 50, this.lifespan);
    circle(this.x, this.y, this.size);
  }

  // Method: returns true when particle should be removed
  isDead() {
    return this.lifespan <= 0;
  }
}

To create an instance, use the new keyword. This calls the constructor, creates a fresh object with its own properties, and returns it:

let p = new Particle(200, 300);  // new calls the constructor
p.update();                       // Call the particle's update method
p.display();                      // Call the particle's display method
console.log(p.lifespan);          // Access a property: 252

15.2.2 Managing Many Instances

The real power of classes emerges when you manage many instances in an array. You can spawn new particles, update them all, draw them all, and remove dead ones—all with clean, readable code:

let particles = [];

// Spawn: add a new particle to the array
particles.push(new Particle(mouseX, mouseY));

// Update and display all particles
for (let p of particles) {
  p.update();
  p.display();
}

// Remove dead particles (iterate backwards when using splice)
for (let i = particles.length - 1; i >= 0; i--) {
  if (particles[i].isDead()) {
    particles.splice(i, 1);
  }
}
⚠ Common Mistake

Always iterate backwards when removing elements with splice(). If you iterate forwards and remove an element at index 3, the element that was at index 4 shifts down to index 3—but your loop counter has already moved past index 3, so that element gets skipped. Backwards iteration avoids this because removals only affect indices you have already visited.

15.2.3 Encapsulation: Keeping Logic Inside the Class

One of the most powerful aspects of OOP is encapsulation—the idea that each object manages its own internal state. The isDead() method is a perfect example: the main sketch does not need to know that particles have a lifespan property or what its threshold value is. It simply asks each particle “are you dead?” and the particle answers. If you later change the death condition (say, particles die when they go off-screen instead of when lifespan hits zero), you only change the isDead() method—the main sketch remains untouched.

This principle of “tell, don’t ask” makes your code more modular and easier to maintain. Instead of the main sketch reaching inside objects to read their properties and make decisions, the objects themselves contain the decision-making logic.

15.2.4 Inheritance: extends and super()

Sometimes you want a new class that is similar to an existing one but with some differences. Inheritance lets a child class (subclass) extend a parent class (superclass), inheriting all its properties and methods while adding or overriding some of its own.

class Shape {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }
  move(dx, dy) {
    this.x += dx;
    this.y += dy;
  }
}

class CircleShape extends Shape {
  constructor(x, y, r) {
    super(x, y);   // Call the parent constructor first
    this.r = r;     // Add a new property specific to circles
  }
  display() {
    circle(this.x, this.y, this.r * 2);
  }
}

The keyword extends declares the inheritance relationship. Inside the child constructor, you must call super() before using this—this runs the parent constructor to set up the inherited properties. You can also call super.methodName() to invoke the parent’s version of a method.

When different classes share a common interface (like a display() method), you can store them in the same array and loop through them uniformly. A CircleShape and a RectShape both respond to display(), but each draws something different. This is called polymorphism—the same method name, different behavior depending on the object’s class.

Inheritance is powerful but should be used sparingly. A good rule of thumb: use inheritance when there is a genuine “is-a” relationship (a CircleShape is a Shape). If the relationship is “has-a” (a Car has a Engine), prefer composition—storing one object as a property of another rather than using inheritance.

15.2.5 When to Use Classes

You do not need classes for everything. A simple sketch that draws a single pattern does not benefit from OOP. Use classes when you have:

  • Multiple instances of the same kind of thing (particles, agents, enemies, musical notes)
  • State and behavior that belong together (a ball has position, velocity, and knows how to bounce—these should not be separate)
  • Complex interactions between objects (collision detection, gravitational attraction, flocking behavior)
  • Code that is becoming unmanageable (if you have x1, y1, vx1, vy1, x2, y2, vx2, vy2..., it is time for a class)

The refactoring process is straightforward: identify the variables that belong together, bundle them into a class with a constructor, move the update logic into an update() method, move the drawing logic into a display() method, and replace your loose variables with an array of instances. Suddenly, adding 100 more entities is a one-line change instead of duplicating code 100 times.

Consider the difference between procedural and object-oriented approaches for managing two bouncing balls:

// PROCEDURAL: variables for each ball, duplicated logic
let x1 = 100, y1 = 200, vx1 = 2, vy1 = 3;
let x2 = 300, y2 = 150, vx2 = -1, vy2 = 2;

function draw() {
  x1 += vx1; y1 += vy1;
  if (x1 > width || x1 < 0) vx1 *= -1;
  if (y1 > height || y1 < 0) vy1 *= -1;
  circle(x1, y1, 30);

  x2 += vx2; y2 += vy2;
  if (x2 > width || x2 < 0) vx2 *= -1;
  if (y2 > height || y2 < 0) vy2 *= -1;
  circle(x2, y2, 30);
  // Adding a 3rd ball means copying all of this AGAIN
}

// OOP: one class, any number of instances
let balls = [];
function setup() {
  for (let i = 0; i < 100; i++) {
    balls.push(new Ball(random(width), random(height)));
  }
}
function draw() {
  for (let b of balls) { b.update(); b.display(); }
  // Adding more balls: just change the 100
}

The OOP version is not just shorter—it is fundamentally more scalable and maintainable. If you need to add a new behavior (say, color changing), you add it once in the class and every instance gets it automatically.

15.2.6 Common OOP Mistakes

Three mistakes trip up nearly every beginner learning OOP in JavaScript:

  • Forgetting this: Inside a method, x is a local variable that disappears when the method ends. this.x is the object’s property that persists. If you write x = 5 instead of this.x = 5 in a method, you are creating a temporary local variable, not setting the object’s property.
  • Forgetting new: Writing let b = Ball(100, 200) without new calls Ball as a regular function, which will throw an error or produce undefined. Always use new Ball(100, 200).
  • Forgetting super(): If your class extends another class, you must call super() in the constructor before using this. Omitting it produces a ReferenceError.
📜 Historical Note

The pattern of separating particles into constructor(), update(), display(), and isDead() methods comes from Daniel Shiffman’s The Nature of Code, which built on the Processing tradition established by Casey Reas and Ben Fry. This convention is not required by JavaScript—you can name your methods anything—but it has become the standard vocabulary of creative coding, and using it makes your code immediately readable to other creative coders.

15.3 Functional Programming (FP)

Functional programming is a style of writing code that treats functions as values—they can be stored in variables, passed as arguments to other functions, and returned as results. Instead of writing step-by-step instructions that modify data in place, functional programming emphasizes transforming data: you take an array, apply a transformation, and get a new array back, leaving the original unchanged.

You have already been using functional programming without realizing it. Every time you wrote particles.filter(p => p.lifespan > 0) to remove dead particles, or used p5.js’s map() function to convert a value from one range to another, you were thinking functionally.

15.3.1 Arrow Functions

Arrow functions are a concise syntax for writing functions, especially useful when passing small functions as arguments:

// Traditional function
function double(x) {
  return x * 2;
}

// Arrow function (equivalent)
let double = x => x * 2;

// Multiple parameters require parentheses
let add = (a, b) => a + b;

// No parameters require empty parentheses
let greet = () => "hello";

When the function body is a single expression, you can omit the curly braces and the return keyword—the expression’s value is returned automatically. For longer function bodies, use curly braces and an explicit return.

15.3.2 The Big Three: map(), filter(), reduce()

These three array methods are the workhorses of functional programming. They all create new arrays or values without modifying the original.

map() transforms every element in an array by applying a function to each one:

let numbers = [1, 2, 3, 4, 5];
let doubled = numbers.map(n => n * 2);     // [2, 4, 6, 8, 10]
let squared = numbers.map(n => n * n);     // [1, 4, 9, 16, 25]
// numbers is still [1, 2, 3, 4, 5] -- unchanged

filter() keeps only the elements for which a test function returns true:

let numbers = [1, 2, 3, 4, 5, 6];
let evens = numbers.filter(n => n % 2 === 0);  // [2, 4, 6]
let big = numbers.filter(n => n > 3);           // [4, 5, 6]

reduce() combines all elements into a single value by applying a function that accumulates a result:

let numbers = [1, 2, 3, 4, 5];
let sum = numbers.reduce((acc, n) => acc + n, 0);     // 15
let max = numbers.reduce((a, b) => a > b ? a : b);    // 5

The second argument to reduce() is the initial value of the accumulator. For summing, you start at 0. For building an object, you start with {}.

15.3.3 Other Useful Array Methods

Beyond the Big Three, several other functional array methods are valuable in creative coding:

forEach() is like map() but does not return a new array. Use it when you want to perform an action (like drawing) for each element:

particles.forEach(p => {
  p.update();
  p.display();
});

find() returns the first element that matches a condition. findIndex() returns its index:

let weakAgent = agents.find(a => a.energy < 50);
let weakIndex = agents.findIndex(a => a.energy < 50);

some() returns true if at least one element passes the test. every() returns true only if all elements pass:

let anyAlive = particles.some(p => p.alive);    // true if any are alive
let allDead = particles.every(p => !p.alive);    // true if all are dead

These methods read like English, making your code’s intent clear at a glance. “Do some particles exist that are alive?” is expressed naturally as particles.some(p => p.alive).

15.3.4 Pure Functions and Immutability

A pure function always returns the same output for the same input and does not modify anything outside itself (no side effects). In creative coding, pure functions are ideal for transformations and calculations. Drawing functions are inherently impure (they modify the canvas), and that is perfectly fine. The goal is not to make everything pure, but to separate pure logic from impure side effects when it makes the code clearer.

Immutability means not changing data after it is created. Instead of modifying an array in place, you create a new one. The map() and filter() methods embody this principle—they return new arrays and leave the originals untouched. For creative coding, strict immutability can be impractical (updating 1000 particle positions every frame by creating new arrays is wasteful), but the useful takeaway is: be intentional about when you mutate data.

15.3.5 Creative Coding Applications

Functional methods are particularly elegant for creative coding tasks. Compare removing dead particles imperatively versus functionally:

// Imperative: backwards splice loop
for (let i = particles.length - 1; i >= 0; i--) {
  if (particles[i].isDead()) particles.splice(i, 1);
}

// Functional: one line with filter
particles = particles.filter(p => !p.isDead());

You can also chain operations together. Since map(), filter(), and sort() all return new arrays, you can call them one after another in a pipeline:

let result = data
  .filter(d => d > 10)        // Keep values above 10
  .map(d => d * 2)            // Double each value
  .sort((a, b) => a - b);    // Sort ascending

Chaining reads like a recipe: “Take the data, keep only values above 10, double each one, then sort them in ascending order.” Each step is a clear, self-contained transformation. Compare this to the imperative equivalent, which requires a temporary array, a loop, an if statement, and a separate sort call—more code, more opportunities for bugs, and harder to read at a glance.

// Imperative equivalent (same result, more code)
let temp = [];
for (let i = 0; i < data.length; i++) {
  if (data[i] > 10) {
    temp.push(data[i] * 2);
  }
}
temp.sort((a, b) => a - b);

Both approaches produce the same result. The functional version is more concise and often easier to read once you are comfortable with filter and map. The imperative version may be slightly faster for very large arrays because it iterates only once rather than creating intermediate arrays, but for creative coding workloads this difference is negligible.

First, filter(n => n % 2 === 0) keeps only even numbers: [2, 4]. Then map(n => n * 10) multiplies each by 10: [20, 40]. The operations are chained—map receives the filtered array, not the original.

15.4 Data Structures

Every creative coding project needs to organize data. A particle system stores hundreds of particles. A grid-based cellular automaton stores thousands of cell states. An undo system remembers past states. The way you organize this data—the data structure you choose—determines what operations are easy or hard, fast or slow.

15.4.1 Arrays and Key Methods

Arrays are ordered lists, and they are the data structure you use most often. Here is a reference of the essential methods:

let a = [1, 2, 3];
a.push(4);           // Add to end: [1, 2, 3, 4]
a.pop();             // Remove from end, returns 4: [1, 2, 3]
a.unshift(0);        // Add to beginning: [0, 1, 2, 3]
a.shift();           // Remove from beginning, returns 0: [1, 2, 3]
a.splice(1, 1);      // Remove 1 item at index 1: [1, 3]
a.includes(3);       // true
a.indexOf(3);        // 1
a.slice(0, 2);       // Copy portion: [1, 3] (non-destructive)

15.4.2 Removing Items During Iteration

Removing items from an array during iteration is one of the most common sources of bugs in creative coding. When you remove an item from an array during forward iteration using splice(), all subsequent indices shift down by one. Your loop counter moves forward, but the array has shifted backward—so you skip the next element.

// WRONG -- skips elements after removal
for (let i = 0; i < particles.length; i++) {
  if (particles[i].isDead()) {
    particles.splice(i, 1);
    // The particle that was at index i+1 is now at index i
    // But i increments to i+1, skipping it!
  }
}

// CORRECT -- iterate backwards
for (let i = particles.length - 1; i >= 0; i--) {
  if (particles[i].isDead()) {
    particles.splice(i, 1);
    // Removal only affects indices we have already visited
  }
}

// ALSO CORRECT -- use filter (creates a new array)
particles = particles.filter(p => !p.isDead());

The filter() approach is often cleaner and less error-prone. It creates a new array containing only the elements that pass the test, leaving the original unchanged. The backwards-iteration approach is better when you need to perform additional operations during the removal process (like spawning explosion particles where a dead particle was).

15.4.3 Objects as Key-Value Pairs

JavaScript objects store data as named key-value pairs. They are ideal for configuration settings, named properties, and structured data:

let config = {
  gravity: 0.1,
  friction: 0.99,
  maxParticles: 500
};

config.gravity;            // 0.1 (dot notation)
config["friction"];        // 0.99 (bracket notation for dynamic keys)
config.newProp = true;     // Add a new property on the fly

15.4.4 Stacks (LIFO) and Queues (FIFO)

A stack follows the Last In, First Out principle: the most recently added item is the first one removed. Think of a stack of plates—you put plates on top and take them off the top. In JavaScript, use push() and pop() on an array:

let undoStack = [];
undoStack.push(currentState);        // Save state
let previous = undoStack.pop();      // Restore most recent state

A queue follows First In, First Out: the first item added is the first one processed, like a line at a store. Use push() and shift():

let eventQueue = [];
eventQueue.push(newEvent);          // Add to end
let next = eventQueue.shift();      // Process from front

Stacks are perfect for undo/redo systems (as seen in Chapter 4). Queues are perfect for mouse trails and event processing.

15.4.5 Choosing the Right Data Structure

Choosing the right data structure is one of the most important decisions in programming. Here is a quick reference to guide your choice:

SituationData StructureWhy
Collection of particles or agentsArrayOrdered, easy to iterate, add/remove
Configuration settingsObjectNamed keys, easy to read and modify
Grid simulation (Game of Life, heat)2D ArrayRow/column access, neighbor iteration
Undo/Redo historyStack (push/pop)LIFO order matches undo behavior

In practice, arrays and objects handle the vast majority of creative coding needs. Reach for more specialized structures (Map, Set, 2D arrays) when the situation calls for them. A good principle: start with the simplest data structure that works, and upgrade to a more specialized one only if you encounter a performance problem or if the code becomes hard to read.

📖 Definition

Double Buffering — A technique where two copies of a data structure are maintained. During each update cycle, you read from one copy and write to the other, then swap them. This prevents partial updates from affecting calculations within the same frame. Used extensively in cellular automata (Chapter 5) and the heat diffusion simulation in this chapter.

15.4.6 2D Arrays (Grids)

A 2D array is an array of arrays, forming a grid. This is essential for cellular automata (Chapter 5), tile maps, image processing, and grid-based simulations like heat diffusion. In these simulations, each cell stores a value (alive/dead, temperature, height, etc.) and interacts with its neighbors according to rules you define.

The key technique for grid simulations is double buffering: maintain two grids, read from one and write to the other, then swap them at the end of each frame. This prevents a cell’s update from affecting its neighbors’ calculations within the same frame—without double buffering, the order in which cells are updated would influence the result, creating directional artifacts.

// Create a 2D grid filled with zeros
let grid = [];
for (let i = 0; i < rows; i++) {
  grid[i] = [];
  for (let j = 0; j < cols; j++) {
    grid[i][j] = 0;
  }
}

// Access a cell
let val = grid[row][col];
grid[row][col] = 1;

// Convert pixel coordinates to grid coordinates
let col = floor(mouseX / cellSize);
let row = floor(mouseY / cellSize);

15.5 Interactivity

Interactivity is what transforms a generative sketch from something you watch into something you experience. p5.js provides a rich set of tools for responding to mouse actions, keyboard input, touch events, and GUI controls. This section covers the most important patterns.

15.5.1 Mouse Events and Built-in Variables

p5.js automatically tracks the mouse and provides built-in variables: mouseX and mouseY give the current position, pmouseX and pmouseY give the previous frame’s position (useful for computing speed), and mouseIsPressed is true whenever a mouse button is held down.

For discrete events, define callback functions that p5.js will call automatically:

function mousePressed() { /* fires once on click */ }
function mouseReleased() { /* fires once on release */ }
function mouseDragged() { /* fires while dragging */ }
function mouseWheel(event) {
  let delta = event.delta; // positive = scroll down
  return false; // prevent page scroll
}

15.5.2 Keyboard Events

The keyPressed() function fires once when a key goes down. Use the key variable for letter keys and keyCode for special keys like arrows:

function keyPressed() {
  if (key === 'r') { /* reset */ }
  if (keyCode === UP_ARROW) { /* move up */ }
}

// For continuous key detection (smooth movement), use keyIsDown() in draw():
if (keyIsDown(LEFT_ARROW)) player.x -= 3;

15.5.3 GUI Controls

p5.js can create HTML interface elements that interact with your canvas. Sliders, buttons, and inputs are positioned below or beside the canvas:

let slider = createSlider(0, 100, 50, 1);  // min, max, default, step
slider.position(10, height + 10);
let val = slider.value();                   // read the current value

let btn = createButton("Reset");
btn.position(10, height + 40);
btn.mousePressed(() => { /* handle click */ });

15.5.4 Responsive Canvas

When the browser window is resized, p5.js calls windowResized() if you define it. Use this to create sketches that adapt to any screen size:

function setup() {
  createCanvas(windowWidth, windowHeight);
}

function windowResized() {
  resizeCanvas(windowWidth, windowHeight);
}

With a responsive canvas, use relative coordinates (like width / 2 and height / 2 for the center) rather than hard-coded pixel values. This ensures your sketch looks correct at any size. For example, instead of circle(200, 200, 50), write circle(width / 2, height / 2, min(width, height) * 0.1)—the circle is always centered and scales proportionally with the canvas.

Responsive design is especially important if your final project will be exhibited in a gallery or presented on a projector, where the screen resolution may be very different from your development environment.

15.5.5 Touch Events

For mobile devices and touchscreens, p5.js provides touchStarted(), touchMoved(), and touchEnded(). The touches array contains all active touch points, each with .x and .y properties. Always return false from touch event functions to prevent the browser from scrolling or performing other default actions:

function touchStarted() {
  for (let t of touches) {
    // Each touch point has .x and .y
    particles.push(new Particle(t.x, t.y));
  }
  return false; // Prevent default browser behavior
}

15.5.6 Preventing Default Browser Behavior

By default, the browser handles certain events itself: right-click opens a context menu, spacebar scrolls the page, arrow keys scroll the page. To prevent these defaults in your sketch, return false from the relevant event handler:

function keyPressed() {
  // Prevent spacebar and arrow keys from scrolling
  if ([32, 37, 38, 39, 40].includes(keyCode)) {
    return false;
  }
}

function mousePressed() {
  // Prevent right-click context menu
  return false;
}

This is especially important for game-like sketches where arrow keys and spacebar are used for gameplay. Without preventing defaults, pressing the spacebar would cause the page to scroll down, ruining the experience.

15.5.7 Hit Detection

p5.js does not have built-in hit detection—you must compute it yourself. The two most common tests are point-in-circle and point-in-rectangle:

// Is the mouse over a circle?
function isMouseOverCircle(cx, cy, r) {
  return dist(mouseX, mouseY, cx, cy) < r;
}

// Is the mouse over a rectangle?
function isMouseOverRect(rx, ry, rw, rh) {
  return mouseX > rx && mouseX < rx + rw
      && mouseY > ry && mouseY < ry + rh;
}

You want to move a player character smoothly while the user holds down the right arrow key. Which approach should you use?

15.6 Pixel Manipulation

Every image on your screen is made of pixels, and every pixel is stored as four numbers: red, green, blue, and alpha. Direct pixel manipulation gives you complete control over the image—every filter, every effect, every glitch art technique ultimately comes down to reading and writing pixel values.

15.6.1 The pixels[] Array

The p5.js canvas stores all pixel data in a one-dimensional array called pixels[]. Each pixel occupies four consecutive slots in the array: R, G, B, A. Before you can read or write this array, you must call loadPixels(). After modifying it, you must call updatePixels() to push the changes to the screen.

💡 Key Concept

The pixel index formula converts 2D coordinates to a position in the 1D pixel array:

$$\text{index} = (y \times \text{width} + x) \times 4$$

Then pixels[index] is red, pixels[index + 1] is green, pixels[index + 2] is blue, and pixels[index + 3] is alpha. This formula works because pixels are stored row by row, left to right, top to bottom.

15.6.2 pixelDensity(1): Essential for Correct Access

On high-DPI (Retina) displays, p5.js doubles the pixel density by default. This means a 400×400 canvas actually has 800×800 pixels, and the pixels[] array is four times larger than expected. Always call pixelDensity(1) in setup() when doing direct pixel manipulation:

function setup() {
  createCanvas(400, 400);
  pixelDensity(1); // One canvas pixel = one screen pixel
}
⚠ Common Mistake

Forgetting pixelDensity(1) is the most common source of mysterious pixel manipulation bugs. Without it, your pixel math will be correct on standard displays but produce garbled results on Retina displays. Always include it.

15.6.3 Common Pixel Operations

Once you can read and write individual pixels, a wide range of image processing operations become straightforward. The most fundamental is converting a pixel to grayscale. The human eye is not equally sensitive to all colors—we perceive green as brightest, then red, then blue. The standard luminance formula weights the channels accordingly:

$$\text{gray} = 0.299 \cdot R + 0.587 \cdot G + 0.114 \cdot B$$

This formula, known as the ITU-R BT.601 luma formula, is used throughout image processing and appears in this textbook whenever we compute brightness from RGB values.

// Grayscale: weighted average matching human perception
let gray = 0.299 * r + 0.587 * g + 0.114 * b;

// Invert (negative): subtract each channel from 255
pixels[i] = 255 - pixels[i];

// Threshold (black and white): compare brightness to a cutoff
let val = gray > threshold ? 255 : 0;

// Brightness: add a constant to each channel
pixels[i] = constrain(pixels[i] + amount, 0, 255);

// Contrast: stretch values away from the midpoint (128)
pixels[i] = constrain((pixels[i] - 128) * factor + 128, 0, 255);

15.6.4 Working with Images

You can load an image and manipulate its pixels. Use preload() to load the image before setup() runs, draw it to the canvas, then access pixels[] to modify it:

let img;

function preload() {
  img = loadImage("photo.jpg");
}

function setup() {
  createCanvas(img.width, img.height);
  pixelDensity(1);
  image(img, 0, 0); // Draw image to canvas
  noLoop();
}

function draw() {
  loadPixels();
  // Now pixels[] contains the image data
  // Modify it, then:
  updatePixels();
}

You can also access an image’s pixel array directly without drawing it to the canvas first, using img.loadPixels() and img.pixels[]. This is useful when you want to read from the original image while writing to the canvas.

15.6.5 Pixel Sorting (Glitch Art)

Pixel sorting is a glitch art technique popularized by artist Kim Asendorf in 2012. The algorithm divides image rows into “runs” of pixels (defined by a brightness threshold), sorts each run by brightness (or hue, or any other criterion), and writes the sorted pixels back. The result is a striking visual effect where smooth gradients emerge within the sorted runs while the overall composition is preserved.

Here is the basic algorithm outline for implementing pixel sorting:

  1. Load the image and call loadPixels()
  2. For each row of the image, scan from left to right
  3. Find contiguous “runs” of pixels that meet a brightness threshold (e.g., brightness > 60)
  4. Extract the RGBA values for each run into a temporary array
  5. Sort the temporary array by brightness (or hue, saturation, or any criterion you choose)
  6. Write the sorted pixel values back into the pixels[] array at their original positions
  7. Call updatePixels() to display the result
// Simplified pixel sorting pseudocode
loadPixels();
for (let y = 0; y < height; y++) {
  let run = [];
  let runStart = -1;
  for (let x = 0; x < width; x++) {
    let idx = (y * width + x) * 4;
    let b = (pixels[idx] + pixels[idx+1] + pixels[idx+2]) / 3;
    if (b > 60) {
      if (runStart === -1) runStart = x;
      run.push({ r: pixels[idx], g: pixels[idx+1],
                 b: pixels[idx+2], a: pixels[idx+3], bright: b });
    } else {
      if (run.length > 1) {
        run.sort((a, b) => a.bright - b.bright);
        for (let i = 0; i < run.length; i++) {
          let wi = (y * width + runStart + i) * 4;
          pixels[wi] = run[i].r;
          pixels[wi+1] = run[i].g;
          pixels[wi+2] = run[i].b;
          pixels[wi+3] = run[i].a;
        }
      }
      run = []; runStart = -1;
    }
  }
}
updatePixels();

Experiment with different sorting criteria—sorting by hue produces rainbow streaks, sorting by saturation creates watercolor-like washes, and sorting by red channel alone produces eerie, channel-specific effects. You can also sort columns instead of rows for a rain-like dripping effect, or sort along diagonal lines for even more unusual results.

15.6.6 Performance Considerations

Direct pixel manipulation on the CPU is slow for large canvases because you are touching every pixel every frame. Some tips for better performance:

  • Use pixelDensity(1) to avoid processing 4× the pixels on Retina displays
  • Use smaller canvases (200×200 instead of 800×800) for real-time effects
  • Avoid get(x, y) for reading individual pixels—it is much slower than direct array access
  • Use noLoop() and redraw() for static image processing—no need to process 60 times per second
  • Move effects to GLSL shaders for real-time performance—the GPU processes all pixels in parallel (see Chapters 8–11)

15.7.2 Type Coercion Pitfalls

JavaScript automatically converts between types in many situations. This is called type coercion, and it is one of the most common sources of bugs:

"5" + 3      // "53" (concatenation, not addition!)
"5" - 3      // 2 (subtraction coerces string to number)
5 == "5"     // true (loose equality coerces types)
5 === "5"    // false (strict equality, no coercion -- ALWAYS USE THIS)
⚠ Common Mistake

Always use === (strict equality) instead of == (loose equality). Loose equality performs type coercion, which leads to surprising results like "" == false being true and null == undefined being true. Strict equality compares both value and type, eliminating these surprises.

15.7.3 Explicit Type Conversion

When you need to convert between types, do it explicitly rather than relying on JavaScript’s implicit coercion. Here are the standard conversion functions:

// String to number
let n = Number("42");          // 42
let n2 = parseInt("42px");     // 42 (ignores non-numeric suffix)
let n3 = parseFloat("3.14");   // 3.14

// Number to string
let s = String(42);            // "42"
let s2 = (42).toString();      // "42"
let s3 = (255).toString(16);   // "ff" (hexadecimal)
let s4 = (10).toString(2);     // "1010" (binary)

// To boolean
let b = Boolean(0);            // false
let b2 = Boolean("hello");    // true

A common pattern in creative coding is reading values from HTML inputs (which always return strings) and converting them to numbers: let speed = Number(speedInput.value());. Forgetting this conversion leads to string concatenation bugs that can be very confusing to debug.

15.7.4 Truthy and Falsy Values

In boolean contexts (like if statements), JavaScript treats certain values as false. These “falsy” values are: false, 0, "" (empty string), null, undefined, and NaN. Everything else is “truthy”—including empty arrays [] and empty objects {}, which surprises many beginners.

// These are all falsy:
if (0) { }          // does not execute
if ("") { }         // does not execute
if (null) { }       // does not execute

// These are all truthy (surprising ones):
if ([]) { }         // EXECUTES -- empty array is truthy!
if ({}) { }         // EXECUTES -- empty object is truthy!

This means you cannot use if (myArray) to check if an array has elements—an empty array [] is truthy, so the condition would always pass. Use if (myArray.length > 0) instead.

Understanding truthiness is important for defensive programming. A common pattern for providing default values uses the logical OR operator:

// If config.speed is falsy (undefined, 0, etc.), use 3 as default
let speed = config.speed || 3;

// Modern JavaScript has the nullish coalescing operator (??)
// which only triggers for null/undefined, not for 0 or ""
let speed = config.speed ?? 3;  // Uses 3 only if speed is null/undefined

15.7.5 Template Literals and String Methods

Template literals (backtick strings) allow embedded expressions and multi-line strings:

let info = `Canvas: ${width}x${height}, FPS: ${frameRate().toFixed(1)}`;

// Useful string methods
let s = "hello world";
s.length;                  // 11
s.toUpperCase();           // "HELLO WORLD"
s.split(" ");              // ["hello", "world"]
s.includes("world");       // true
s.slice(0, 5);             // "hello"
s.replace("world", "p5"); // "hello p5"

15.7.6 Typed Arrays

Regular JavaScript arrays can hold any type of value. Typed arrays hold only one type of number, stored as raw binary data. They are used for audio buffers (Float32Array), image pixel data (Uint8ClampedArray—this is what the pixels[] array actually is), and communication with WebGL shaders.

let buf = new Float32Array(256); // 256 floats, initialized to 0
buf[0] = sin(0.1);               // Write a value

let px = new Uint8ClampedArray(4); // Auto-clamps to 0-255
px[0] = 300;  // Stored as 255 (clamped to maximum)
px[1] = -50;  // Stored as 0 (clamped to minimum)

15.7.7 Bit Operations

Bit operations work on the binary representation of numbers. They appear in creative coding for XOR patterns, pixel packing, and fast math tricks:

let val = (x ^ y) & 0xFF;      // XOR pattern, masked to 0-255
let r = (packed >> 16) & 0xFF; // Extract red from packed RGBA
let floored = (3.7) | 0;       // Fast floor: 3

Several misconceptions about copyright and licensing are widespread. Here are the most important ones to avoid:

  • “I found it on the internet, so it’s free to use.” — No. Everything on the internet is copyrighted by default. Only works with explicit licenses or public domain dedications are free to use.
  • “I changed it enough that it’s mine now.” — This is a legal gray area. Substantial modification creates a “derivative work,” but you still need permission (a license) to create derivatives of the original.
  • “It’s for a school project, so fair use applies.” — Fair use is a defense, not a blanket right, and it is narrower than most people think. Educational use is one factor courts consider, but it does not automatically make all uses fair.
  • Mixing incompatible licenses. — You cannot combine GPL code with proprietary code or with code under certain other licenses. Always check license compatibility before mixing code from different sources.

15.8.4 The Free Culture Movement

📜 Historical Note

Lawrence Lessig articulated the Free Culture movement in his 2004 book Free Culture. He argued that culture has always been built on remixing, referencing, and building on the past, but modern copyright law (which has expanded from an original 14-year term to life plus 70 years) stifles creativity and cultural participation. Creative Commons licenses were created as a practical response—letting creators voluntarily share their work under terms they choose, rather than defaulting to “all rights reserved.”

The Free Software Foundation focuses on four freedoms: the freedom to run, study, redistribute, and distribute modified versions of software. The GPL was created to protect these freedoms through copyleft. Open Source is a related but distinct movement that emphasizes practical benefits (better code quality, collaboration) over philosophical freedoms.

15.9 Games and Visual Art

You have been using a game loop since day one of this course without calling it that. The setup()/draw() pattern in p5.js is structurally identical to the core loop of every video game: initialize, then loop forever through input processing, state updates, and rendering. The difference is in intent: games emphasize rules, goals, and player agency; creative coding emphasizes aesthetics, emergence, and exploration. Art games deliberately blur these boundaries.

📜 Historical Note

The boundary between games and art has been explored by many artists and designers. Jason Rohrer’s Passage (2007) is a 5-minute game about life and death—you walk right, you age, you die. thatgamecompany’s Flower (2009) lets you be the wind, guiding petals across landscapes with no enemies, no score, no failure. Max Bittker’s Sandspiel is a falling-sand simulation in the browser—not a game (no goals), but deeply playful and interactive. These works challenge us to think about what makes something a “game” versus an “interactive experience.”

15.9.1 Collision Detection

Collision detection answers the fundamental question: “Are these two things overlapping?” The math is straightforward. For two circles, they collide when the distance between their centers is less than the sum of their radii:

$$d(c_1, c_2) = \sqrt{(x_2 - x_1)^2 + (y_2 - y_1)^2} < r_1 + r_2$$

function circlesCollide(x1, y1, r1, x2, y2, r2) {
  return dist(x1, y1, x2, y2) < r1 + r2;
}

For rectangles, the standard approach is AABB (Axis-Aligned Bounding Box) collision. Two rectangles overlap when they overlap on both the x-axis and the y-axis simultaneously. This is expressed as four conditions that must all be true:

// Rectangle-rectangle (AABB: Axis-Aligned Bounding Box)
function rectsCollide(x1, y1, w1, h1, x2, y2, w2, h2) {
  return x1 < x2 + w2 && x1 + w1 > x2
      && y1 < y2 + h2 && y1 + h1 > y2;
}

// Point in circle: for mouse interaction / hit detection
function pointInCircle(px, py, cx, cy, r) {
  return dist(px, py, cx, cy) < r;
}

// Point in rectangle: for button clicks
function pointInRect(px, py, rx, ry, rw, rh) {
  return px > rx && px < rx + rw
      && py > ry && py < ry + rh;
}

Point-in-circle and point-in-rectangle are used constantly for hit detection—determining whether the mouse cursor is over a shape for interactive buttons, drag-and-drop, and selection. They are the same formulas as circle and rectangle collision, but with one object reduced to a single point.

15.9.2 State Machines

Most games have multiple states: a title screen, gameplay, pause menu, and game over screen. A state machine is a pattern for managing transitions between these states:

let state = "title";

function draw() {
  if (state === "title") drawTitle();
  else if (state === "playing") drawGame();
  else if (state === "gameover") drawGameOver();
}

function keyPressed() {
  if (state === "title" && keyCode === ENTER) state = "playing";
  if (state === "gameover" && keyCode === ENTER) state = "title";
}

The state machine pattern keeps your code organized as complexity grows. Without it, a single draw() function would need to handle all possible game states with tangled if statements. With a state machine, each state has its own dedicated function with its own logic, clearly separated from the others. The state transitions are explicit and easy to follow.

You can visualize the transitions as a diagram:

[Title] --ENTER--> [Playing] --collision--> [Game Over]
                      ^  |                       |
                      |  P                    ENTER
                      |  v                       |
                   [Paused]                   [Title]

15.9.3 Game Feel (“Juice”)

Small visual and audio responses to player actions make interactions feel satisfying. These techniques—collectively called “juice”—include screen shake on impact, easing animations for scores, particle bursts on collection events, and color flashes on collision:

// Screen shake: translate by random offset on impact
if (shakeAmount > 0) {
  translate(random(-shakeAmount, shakeAmount),
            random(-shakeAmount, shakeAmount));
  shakeAmount *= 0.9; // Decay
}

// Animated score: smoothly count up to the target
displayScore = lerp(displayScore, targetScore, 0.1);

15.9.4 Timers and Spawning

Games need to track time for countdowns, cooldowns, and enemy spawning. There are two approaches. Frame-based timers use the frameCount variable:

// Spawn an enemy every 60 frames (about 1 second at 60 fps)
if (frameCount % 60 === 0) {
  spawnEnemy();
}

Millisecond-based timers are more accurate and independent of frame rate, making them better for gameplay timing:

let lastSpawnTime = 0;
let spawnDelay = 2000; // 2 seconds

function draw() {
  let now = millis();
  if (now - lastSpawnTime > spawnDelay) {
    spawnEnemy();
    lastSpawnTime = now;
  }
}

15.9.5 Design Principles for Art Games

If you want to create a game that is also a work of art, consider these principles:

  • Constraint is creative. Limiting mechanics to one or two actions forces you to make them meaningful. A game where you can only move and collect is more expressive than one with 20 buttons.
  • Juice matters. Small visual and audio responses to player actions make interactions feel satisfying—screen shake, particle bursts, color flashes, easing animations.
  • Emergence over scripting. Simple rules that interact to produce complex behavior are more interesting (and more like art) than heavily scripted sequences. This is the same principle behind cellular automata and agent systems.
  • Let the player interpret. Games do not need to explain everything. Abstract visuals and ambiguous mechanics invite the player to make their own meaning.

15.10 Sound and Audio Programming

Sound is a core dimension of computing arts. The p5.sound library (bundled with p5.js) wraps the Web Audio API, providing oscillators, envelopes, filters, FFT analysis, and sound file playback. The most important thing to know before writing any audio code: modern browsers require a user gesture (click, tap, or key press) before audio can play. You handle this with userStartAudio().

⚠ Common Mistake

If your audio code produces no sound, the most likely cause is the browser’s audio autoplay policy. Always call userStartAudio() inside a user event handler like mousePressed() or keyPressed(). Without this, the browser’s AudioContext remains suspended and no sound will play.

15.10.1 Oscillators

An oscillator generates a periodic waveform—a repeating signal that produces a tone. p5.sound provides four waveform types: sine (pure, smooth), square (hollow, buzzy—retro game sounds), sawtooth (bright, rich—synth leads), and triangle (soft, mellow).

let osc = new p5.Oscillator("sine");
osc.freq(440);       // Frequency in Hz (A4)
osc.amp(0.3);        // Amplitude: 0 (silence) to 1 (maximum)
osc.start();
osc.amp(0, 0.5);     // Fade out over 0.5 seconds

15.10.2 Envelopes (ADSR)

An envelope shapes how a sound’s amplitude changes over time. The standard model is ADSR: Attack (ramp up to peak), Decay (drop to sustain level), Sustain (held level), and Release (fade to silence).

let env = new p5.Envelope();
env.setADSR(0.01, 0.2, 0.3, 0.5); // Attack, Decay, Sustain, Release
env.setRange(0.5, 0);              // Peak amplitude, release amplitude
env.play(osc);                     // Trigger the envelope on an oscillator

Think of ADSR like pressing and releasing a piano key. When you press the key, the sound quickly rises to full volume (Attack), then settles to a comfortable level (Decay to Sustain). While you hold the key, the sound stays at the Sustain level. When you release the key, it fades to silence (Release). Different ADSR settings create very different characters: a very short attack and release creates a plucky sound (like a harpsichord), while a slow attack creates a swelling pad sound.

15.10.3 Filters

A filter removes or emphasizes certain frequencies from a sound. A low-pass filter removes frequencies above the cutoff (making the sound darker and more muffled), while a high-pass filter removes frequencies below it (making the sound thinner and brighter). A band-pass filter keeps only frequencies near the cutoff, creating a nasal or resonant quality. Route an oscillator through a filter by disconnecting it from the default output and connecting it to the filter:

let filter = new p5.LowPass();
osc.disconnect();
osc.connect(filter);
filter.freq(1000);  // Cutoff frequency in Hz
filter.res(5);      // Resonance: emphasis at the cutoff

15.10.4 FFT Analysis

The FFT (Fast Fourier Transform) decomposes a sound into its frequency components, enabling audio-reactive visualizations:

let fft = new p5.FFT(0.8, 256);   // smoothing, number of bins
let spectrum = fft.analyze();      // frequency amplitudes (0-255)
let waveform = fft.waveform();     // time-domain values (-1 to 1)
let bass = fft.getEnergy("bass");  // energy in the bass range

You can use these values to drive visuals—spawn particles when the bass hits, change colors with the mid-range, pulse shapes with the treble. This is the foundation of audio-reactive art (see Chapter 12).

15.10.5 Playing Sound Files

You can load and play audio files (MP3, WAV, OGG) using loadSound() in preload():

let sound;

15.10.6 Audio-Reactive Visuals

Connecting audio analysis to visual output is one of the most compelling applications of sound programming. The basic pattern is: analyze the audio with FFT, extract energy in different frequency bands, and use those values to control visual parameters:

let spectrum = fft.analyze();
let bass = fft.getEnergy("bass");       // 20-140 Hz
let mid = fft.getEnergy("mid");         // 400-2600 Hz
let treble = fft.getEnergy("treble");   // 2600-5200 Hz

// Use these values to drive visuals
let bgPulse = map(bass, 0, 255, 0, 30);
background(bgPulse, 0, bgPulse);

let circleSize = map(mid, 0, 255, 10, 200);
circle(width/2, height/2, circleSize);

For a complete audio-reactive project, consider using microphone input (new p5.AudioIn()) so the sketch responds to live sound in the room, or load an MP3 file for a more controlled experience. Either way, call fft.setInput(source) to tell the FFT which audio source to analyze.

The key to effective audio-reactive visuals is mapping ranges. The energy values from FFT range from 0 to 255, but your visual parameters might need completely different ranges. Use p5.js’s map() function to convert between them:

// Map bass energy (0-255) to circle size (10-200)
let circleSize = map(bass, 0, 255, 10, 200);

// Map mid energy (0-255) to hue (0-360)
let hue = map(mid, 0, 255, 0, 360);

// Map treble energy (0-255) to stroke weight (0.5-5)
let weight = map(treble, 0, 255, 0.5, 5);

Experiment with different mappings to find what looks and feels most natural. Some energy-to-visual mappings feel intuitive (bass = size, because bass is “big”), while others create surprising and interesting effects (bass = color speed, treble = position offset).

15.10.7 The Web Audio API

p5.sound is built on the Web Audio API, a lower-level browser API that gives you more control. The Web Audio API works by connecting nodes in a graph—conceptually similar to modular synthesis, where you patch cables between modules:

// [OscillatorNode] --> [GainNode] --> [BiquadFilterNode] --> [speakers]

For serious audio work, consider exploring the Web Audio API directly or using Tone.js, a more modern and fully-featured audio library. But p5.sound is the most accessible starting point and integrates cleanly with p5.js sketches.

✍ Knowledge Check

Why must you call userStartAudio() in a p5.js sketch that uses sound?

Modern browsers enforce an autoplay policy that suspends the AudioContext until the user interacts with the page (click, tap, or key press). userStartAudio() resumes the AudioContext after such a gesture, allowing sound to play. Without it, all audio output is silently blocked.

✍ Question 1

You have an array of 200 particles. You want to remove all particles whose lifespan property is less than or equal to zero. Which approach is correct?

Option (b) uses filter() to create a new array containing only living particles—clean, correct, and functional. Option (c) would produce an array of booleans, not particles. Option (d) uses reduce() with incorrect syntax.

✍ Question 2

In a class definition, what does super(x, y) do inside a child class constructor?

super() calls the parent class’s constructor. When a child class uses extends, its constructor must call super() before using this. This ensures the parent’s initialization logic runs first, setting up any inherited properties. The arguments you pass to super() are forwarded to the parent constructor.

✍ Question 3

What is the pixel index formula for accessing the red channel of the pixel at position $(x, y)$ on a canvas of width $w$?

Pixels are stored row by row, left to right, top to bottom. y * w skips past $y$ full rows of width $w$, then + x moves to the correct column. The * 4 accounts for the four values per pixel (R, G, B, A). The red channel is at this index, green at index + 1, blue at index + 2, and alpha at index + 3.

✍ Question 4

You publish code on GitHub without adding a LICENSE file. What are other people legally allowed to do with your code?

Copyright is automatic. Without a license, the default is “all rights reserved.” GitHub’s Terms of Service allow others to view and fork your repository on GitHub itself, but they have no legal right to copy, modify, or distribute the code for their own use. This is why explicitly choosing a license matters—even something as simple as MIT unlocks the code for everyone.

✍ Question 5

Which data structure is most appropriate for implementing an undo/redo system?

Undo/redo follows a Last In, First Out pattern: the most recent action is the first one undone. A stack (using push() and pop()) naturally models this. Each action pushes a state onto the undo stack; pressing undo pops it off and pushes it onto a redo stack. A queue (FIFO) would undo the oldest action first, which is not what users expect.
