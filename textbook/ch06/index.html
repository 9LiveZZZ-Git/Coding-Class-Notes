6.1 Introduction

This chapter brings together two of the most powerful techniques in creative computing: GLSL pattern generation and cellular automata. At first glance these domains may seem unrelated — one is about drawing with mathematics on the GPU, the other is about simulating grids of cells that evolve over time. But they share a profound connection: both produce complex visual output from simple, local rules applied uniformly across space.

In a GLSL fragment shader, each pixel computes its color independently using a mathematical function of its coordinates. In a cellular automaton, each cell computes its next state independently using a rule based on its neighbors. Both systems are massively parallel, and both demonstrate that extraordinary richness can emerge from simplicity.

💡 Key Concept

Emergence — Complex global behavior arising from simple local rules. This is the unifying theme of this chapter: whether we are computing pixel colors with math or evolving cell states with lookup tables, the expressive power comes not from the complexity of the rules themselves, but from their uniform, parallel application across a large space.

The connection between these two domains runs deeper than analogy. In Section 6.9, we will see that cellular automata can be implemented directly as GLSL shaders using the frame feedback technique, unifying the two approaches into a single GPU-based pipeline. The Game of Life becomes a fragment shader; each pixel is both a pixel and a cell.

This chapter also touches on one of the deepest questions in science: where does complexity come from? Both GLSL patterns and cellular automata demonstrate that you do not need complex rules to produce complex output. You need simple rules, applied in parallel, with local interactions. This is the same principle that underlies crystal growth, snowflake formation, animal coat patterns, and neural computation. As Stephen Wolfram argued in A New Kind of Science, simple programs may be more fundamental to understanding nature than the traditional approach of simple equations.

6.2 GLSL Pattern Fundamentals

6.2.1 The Fragment Shader Mindset

In a fragment shader you do not "draw" shapes imperatively ("draw a circle here, draw a line there"). Instead, you write a function that answers one question for every pixel: "What color should I be?"

The shader runs simultaneously for every pixel on the screen. Each pixel knows only its own coordinates. It cannot (in most cases) ask what color its neighbor is. The entire pattern must be described as a pure mathematical function of position. This constraint is also a superpower: because every pixel is independent, the GPU can compute them all in parallel, achieving real-time performance even for complex patterns at high resolutions.

Think of it this way: in a traditional drawing program, you might write code like "move the pen to position (100, 200), draw a circle with radius 50, fill it with red." This is imperative drawing. In a fragment shader, you instead write: "for ANY position $(x, y)$, if the distance from $(x, y)$ to $(100, 200)$ is less than 50, the color is red; otherwise it is white." This is functional drawing — you define the image as a mathematical function rather than a sequence of drawing commands.

This functional approach has profound consequences:

  • Resolution independence: The function works at any resolution. Zoom in and the pattern remains sharp because you are evaluating a mathematical function, not sampling stored pixels.
  • Infinite canvas: The function is defined for all $(x, y)$, not just those within a bounded image. You can scroll, zoom, and pan freely.
  • Composability: Functions compose naturally. You can combine, transform, and nest them to build complex patterns from simple parts.
  • Parallelism: Because each pixel is independent, a GPU with thousands of cores can compute all pixels simultaneously.
📖 Definition

Fragment Shader — A program that runs once per pixel (fragment) on the GPU. It receives the pixel's screen coordinates and must output a color (typically as an RGBA vec4). Fragment shaders are written in GLSL (OpenGL Shading Language) and execute in massively parallel fashion. A modern GPU can run fragment shaders for millions of pixels simultaneously, which is why shaders achieve real-time performance even for mathematically complex patterns.

To understand the execution model more concretely: for a 400x400 canvas, the fragment shader runs 160,000 times per frame — once for each pixel. At 60 fps, that is 9.6 million shader invocations per second. A modern GPU handles this trivially because its thousands of shader cores execute many of these invocations simultaneously. The fragment shader is the most computationally intensive stage of the graphics pipeline, and it is where all the creative work happens in shader art.

The key uniforms (values passed from the CPU to all shader invocations) that we use throughout this chapter are:

  • u_resolution (vec2): The canvas size in pixels, e.g., (400.0, 400.0)
  • u_time (float): Elapsed time in seconds since the shader started running
  • u_mouse (vec2): The mouse position in pixels (if supported by the environment)

6.2.2 UV Coordinates and Normalization

The first step in any shader pattern is establishing a coordinate system. Raw pixel coordinates (gl_FragCoord.xy) range from 0 to the canvas width/height in pixels. This is awkward for mathematical patterns because the coordinate range depends on the canvas size. A circle with radius 100 pixels would look different on a 200x200 canvas versus a 1000x1000 canvas. Instead, we normalize the coordinates to a standard range:

// Simple normalization: 0 to 1 (origin at bottom-left)
vec2 uv = gl_FragCoord.xy / u_resolution;

// Centered at origin, aspect-ratio corrected
vec2 uv = (gl_FragCoord.xy - 0.5 * u_resolution.xy) / u_resolution.y;

The first form maps coordinates to $[0, 1]$ in both axes. This is useful for simple patterns but distorts the aspect ratio on non-square canvases.

The second form is the most commonly used in shader art. It first subtracts half the resolution to center the origin, then divides by u_resolution.y only (not .xy). After this transformation:

  • The center of the screen is at $(0, 0)$
  • The top and bottom edges are at $y = +0.5$ and $y = -0.5$
  • The left and right edges depend on the aspect ratio (for a square canvas, $x$ also ranges from $-0.5$ to $+0.5$; for a 16:9 canvas, $x$ ranges from about $-0.89$ to $+0.89$)
  • Dividing by u_resolution.y (not .xy) ensures that circles look like circles, not ellipses — this preserves the aspect ratio
⚠ Common Mistake

Beginners often normalize by dividing by u_resolution.xy (both components), which maps both axes to the same range but distorts the aspect ratio. Circles become ellipses, squares become rectangles. Always divide by a single scalar (typically u_resolution.y) to maintain correct proportions. You can also use min(u_resolution.x, u_resolution.y) to ensure the entire $[-0.5, 0.5]$ range is visible on both axes.

A third common normalization maps coordinates to $[-1, 1]$ on both axes:

// -1 to 1, centered (may distort aspect ratio)
vec2 uv = (2.0 * gl_FragCoord.xy - u_resolution.xy) / u_resolution.y;

This is popular in ShaderToy and gives a larger working range. Choose whichever normalization suits your pattern; just be consistent.

6.2.3 Essential GLSL Functions

GLSL provides a rich set of built-in functions for pattern creation. Understanding these is the key to unlocking shader art:

💡 Key Concept

smoothstep is the anti-aliasing workhorse. Instead of a hard edge (step), smoothstep(a, b, x) creates a soft transition over the interval $[a, b]$. The formula is a cubic Hermite polynomial: $3t^2 - 2t^3$ where $t = \frac{x - a}{b - a}$, clamped to $[0, 1]$. This eliminates jagged edges on curved boundaries.

Let us look at some of these functions in more detail, since they form the vocabulary of all shader patterns:

step() and smoothstep() Compared

step(edge, x) is a binary threshold: it returns exactly 0 or 1 with no transition. This produces hard, aliased edges — fine for some designs, but jagged on curves. smoothstep(edge0, edge1, x) provides a smooth transition between 0 and 1 over the range $[\text{edge0}, \text{edge1}]$. The width of this transition zone controls the "softness" of the edge:

// Hard edge: circle with aliased boundary
float circle_hard = step(0.25, length(uv));

// Soft edge: anti-aliased circle
float circle_soft = smoothstep(0.25, 0.26, length(uv));

// Very soft edge: blurry glow
float circle_glow = smoothstep(0.1, 0.4, length(uv));

fract() and floor(): The Tiling Duo

fract(x) and floor(x) are complementary: for any value $x$, we have $x = \text{floor}(x) + \text{fract}(x)$. Together they let you work with tiled patterns:

  • floor(uv * freq) gives you the cell index — which tile you are in (an integer)
  • fract(uv * freq) gives you the position within the cell — where you are inside the tile (0 to 1)

This decomposition is the foundation of all tiled patterns: grids, dot arrays, brick patterns, and more.

mix(): Linear Interpolation

mix(a, b, t) computes $a \cdot (1 - t) + b \cdot t$. When $t = 0$ you get $a$; when $t = 1$ you get $b$; in between you get a weighted blend. This is fundamental for color gradients, blending between patterns, and smooth transitions. It works on all GLSL types: float, vec2, vec3, vec4.

// Color gradient from red to blue based on x position
vec3 col = mix(vec3(1,0,0), vec3(0,0,1), uv.x + 0.5);

// Blend between two patterns based on distance
float pattern = mix(stripes, dots, smoothstep(0.2, 0.3, length(uv)));

6.2.4 Distance and Angle: The Polar Toolkit

Two operations form the foundation of most patterns:

Distance from a point:

float d = length(uv);              // distance from the origin
float d = distance(uv, somePoint); // distance from any point

Angle from a point:

float angle = atan(uv.y, uv.x);   // returns -PI to PI

Together, these convert Cartesian coordinates $(x, y)$ to polar coordinates $(r, \theta)$, which are the natural language for circular and radial patterns.

$$r = \sqrt{x^2 + y^2}, \qquad \theta = \text{atan2}(y, x)$$

Why are polar coordinates so useful for patterns? Because many natural patterns have rotational symmetry: ripples on water, flower petals, spider webs, radar displays. In polar coordinates, a circle is simply $r = \text{const}$, and a spiral is $r = a + b\theta$. These are far simpler expressions than their Cartesian equivalents.

⚠ Common Mistake

The GLSL atan(y, x) function (two-argument arctangent) returns values in $[-\pi, \pi]$. Be aware of the discontinuity at $\theta = \pm\pi$ (the negative x-axis). Patterns that cross this line may show a seam. To avoid the seam, you can remap: float angle = atan(uv.y, uv.x) + 3.14159; shifts the discontinuity to the positive x-axis, and angle / 6.28318 normalizes to $[0, 1]$.

6.2.5 GLSL Color Palettes

One of the most effective techniques for creating visually appealing shader art is using cosine-based color palettes, popularized by Inigo Quilez. The formula:

$$\text{color}(t) = a + b \cdot \cos\bigl(2\pi(c \cdot t + d)\bigr)$$

where $a$, $b$, $c$, and $d$ are vec3 parameters controlling the palette. In GLSL:

vec3 palette(float t) {
  vec3 a = vec3(0.5, 0.5, 0.5);
  vec3 b = vec3(0.5, 0.5, 0.5);
  vec3 c = vec3(1.0, 1.0, 1.0);
  vec3 d = vec3(0.00, 0.33, 0.67);
  return a + b * cos(6.28318 * (c * t + d));
}

By varying the four parameters, you can create an infinite variety of smooth, aesthetically pleasing color palettes. The input $t$ is typically derived from position, distance, or time. A common shorthand used in many shader examples is:

vec3 col = 0.5 + 0.5 * cos(u_time + uv.xyx + vec3(0, 2, 4));

This creates a smoothly shifting rainbow palette where the three color channels are offset by different amounts.

6.3.1 Stripes

Stripes are perhaps the simplest repeating pattern. They use periodic functions to create alternating bands of color. The key insight is that floor(uv.y * freq) converts a smooth coordinate into an integer "band index," and taking mod 2 alternates between 0 and 1:

$$\text{stripe} = \text{mod}\bigl(\lfloor y \cdot f \rfloor, \; 2\bigr)$$

where $f$ is the frequency (number of stripe pairs). Try changing the frequency and direction in the shader below:

Variations to explore:

  • uv.x for vertical stripes
  • uv.x + uv.y for diagonal stripes at 45 degrees
  • sin(uv.y * freq * 6.28) * 0.5 + 0.5 for smooth gradient stripes instead of hard-edged bands
  • Add u_time to the coordinate for animation: stripes that scroll
  • sin(uv.x * freq * 6.28) * sin(uv.y * freq * 6.28) for a grid of soft diamonds
  • Use non-uniform frequency: floor(uv.y * (8.0 + 4.0 * sin(uv.x * 3.0))) for stripes with variable spacing

The stripe pattern is deceptively fundamental. Many complex patterns are built by applying stripes in transformed coordinate spaces. Stripes in polar coordinates become concentric rings (r-stripes) or fan blades (theta-stripes). Stripes in a warped coordinate space become organic, flowing bands. The humble stripe is the atom of shader patterns.

6.3.2 Checkerboard

The checkerboard pattern combines stripes in both dimensions using a parity check. We compute integer cell coordinates in both $x$ and $y$, add them, and take mod 2. This works because:

  • even + even = even (white)
  • odd + odd = even (white)
  • even + odd = odd (black)

$$\text{checker} = \text{mod}\bigl(\lfloor x \cdot f \rfloor + \lfloor y \cdot f \rfloor, \; 2\bigr)$$

The checkerboard is mathematically interesting because it is equivalent to the XOR of two stripe patterns: horizontal stripes XOR vertical stripes. This connection to XOR (exclusive or) appears throughout pattern generation and cellular automata — Rule 90 is also based on XOR, and it produces the Sierpinski triangle.

You can create colored checkerboards by using the checker value to index into different color palettes:

vec3 colA = vec3(0.9, 0.1, 0.2);  // Red squares
vec3 colB = vec3(0.1, 0.2, 0.9);  // Blue squares
vec3 color = mix(colA, colB, checker);

For a 3D-looking checkerboard (alternating raised/lowered squares), add a subtle brightness variation based on the cell position within each square:

vec2 cellUV = fract(uv * freq);
float edge = smoothstep(0.0, 0.05, cellUV.x) * smoothstep(0.0, 0.05, cellUV.y)
           * smoothstep(0.0, 0.05, 1.0 - cellUV.x) * smoothstep(0.0, 0.05, 1.0 - cellUV.y);
float shaded = checker * edge;  // Squared-off tiles with rounded edges

6.3.3 Circles (Distance-Based)

A filled circle is defined by a distance threshold. We compute the distance from each pixel to a center point, then test whether that distance is less than a radius. Using smoothstep instead of step gives us an anti-aliased edge:

float d = length(uv);
float circle = 1.0 - smoothstep(0.25, 0.26, d);

The smoothstep(0.25, 0.26, d) creates an anti-aliased edge over a 0.01-unit transition zone. The outer 1.0 - inverts it so the circle interior is white (1.0) and the exterior is black (0.0).

This pattern is incredibly versatile. By modifying the distance function, you can create many different shapes:

// Ring (annulus): two smoothsteps for inner and outer edge
float ring = smoothstep(0.24, 0.25, d) - smoothstep(0.29, 0.30, d);

// Soft glow: exponential falloff from center
float glow = exp(-d * 5.0);

// Multiple circles at different positions
float d1 = length(uv - vec2(-0.2, 0.0));
float d2 = length(uv - vec2( 0.2, 0.0));
float circles = max(1.0 - smoothstep(0.15, 0.16, d1),
                    1.0 - smoothstep(0.15, 0.16, d2));

The circle is the fundamental building block of signed distance fields (SDFs), which we explored in Chapter 5. Here we use it purely for pattern creation, but the same distance-based approach extends to any shape that can be described by a distance function.

6.3.4 Concentric Rings

Rings combine distance with periodic functions. By passing the distance through sin(), we create repeating bands that radiate from the center. Subtracting u_time inside the sin() makes the rings expand outward:

$$\text{rings} = \sin(r \cdot f - t) \cdot 0.5 + 0.5$$

The parameter $f$ controls the ring density (higher = more tightly packed rings), and $t$ (typically u_time) controls the animation speed. The * 0.5 + 0.5 remaps the sine wave from $[-1, 1]$ to $[0, 1]$ for use as a brightness value.

Variations:

  • Sharp rings: Use step(0.5, sin(d * 40.0)) instead of the smooth sine for alternating black/white bands with hard edges
  • Pulsing rings: Modulate the frequency with time: sin(d * (30.0 + 10.0 * sin(u_time)) - u_time * 2.0)
  • Eccentric rings: Offset the center from (0,0) to create rings centered at a different point. Move the center with the mouse for interactive patterns.
  • Rings from multiple centers: Compute distances from several points and combine: sin(d1 * 40.0) + sin(d2 * 40.0) creates interference/Moire fringes

6.3.5 Fan (Angular) Patterns

Fan patterns use the angle in polar coordinates. atan(uv.y, uv.x) returns an angle from $-\pi$ to $\pi$. Passing this through sin() with a multiplier creates segments that radiate from the center like the spokes of a wheel:

float angle = atan(uv.y, uv.x);     // -PI to PI
float fan = sin(angle * 10.0) * 0.5 + 0.5;

The multiplier (10.0 in this case) controls the number of fan blades: sin(angle * N) produces $2N$ alternating sectors (N bright and N dark). With step instead of sin, you get sharp-edged pie slices; with sin, you get smooth gradient fans.

Combining angular patterns with distance creates spirals:

// Spiral: angle + distance creates spiral arms
float spiral = sin(angle * 3.0 + length(uv) * 20.0 - u_time) * 0.5 + 0.5;

The length(uv) term adds a radial offset to the angular pattern, causing the bands to twist into spiral arms. The number of arms equals the angular multiplier (3 in this case). Subtracting u_time makes the spiral rotate.

6.3.6 Tiled Patterns with fract()

The fract() function is the key to tiling. It returns the fractional part of a number, effectively wrapping coordinates back to the $[0, 1)$ range. This means you can define a pattern in one "cell" and have it repeat infinitely across the canvas:

vec2 cell = fract(uv * 8.0);  // 8x8 grid of repeating cells
float d = length(cell - 0.5); // distance from cell center
float dot = 1.0 - smoothstep(0.15, 0.16, d); // circle in each cell

The technique works in three steps:

  1. Scale: Multiply UV coordinates by the tile frequency (uv * 8.0). This stretches the coordinate space so that [0,1] now maps to [0,8].
  2. Tile: Apply fract() to wrap back to [0,1] within each tile. Now you have a local coordinate system that repeats 8 times.
  3. Pattern: Use the local coordinates to draw any shape within one tile. It automatically repeats across the grid.

You can get the tile index (which tile you are in) using floor():

vec2 id = floor(uv * 8.0);    // Integer tile ID (0,0) to (7,7)
vec2 local = fract(uv * 8.0); // Position within tile (0 to 1)

The tile ID is useful for making each tile slightly different — for example, by using it as a seed for a hash function to randomize colors or sizes:

float r = hash(id);                  // Random value per tile
float size = 0.1 + 0.15 * r;        // Random circle size
float d = length(local - 0.5);
float dot = 1.0 - smoothstep(size, size + 0.01, d);

More advanced tiling techniques include:

Brick Pattern (Offset Tiling)

Offset every other row by half a tile width to create a brick-laying pattern:

vec2 p = uv * vec2(8.0, 16.0);        // More rows than columns
float row = floor(p.y);
p.x += mod(row, 2.0) * 0.5;           // Shift odd rows by half
vec2 cell = fract(p);
float d = length(cell - 0.5);

Hexagonal Tiling

Hexagonal grids provide 6-fold symmetry, which is more isotropic than the 4-fold symmetry of square grids. Implementing hex tiling in GLSL requires some coordinate transformations but produces beautiful, natural-looking tessellations.

Truchet Tiles

Truchet tiles use a single tile design with rotational variants. Each cell randomly selects an orientation (0, 90, 180, or 270 degrees), creating maze-like or circuit-board patterns. The tile ID (from floor()) is used as a hash seed to determine the rotation:

vec2 id = floor(uv * 8.0);
float rotation = floor(hash(id) * 4.0) * 1.5708; // 0, PI/2, PI, 3PI/2
vec2 local = fract(uv * 8.0) - 0.5;
local = rotate2D(local, rotation);
// Draw arcs or diagonal lines in the rotated local space

Truchet tiles are a beautiful example of how randomness and simple geometry combine to produce complex, visually interesting patterns. They are widely used in generative art and fabric design.

📜 Historical Note

Truchet Tiles are named after Sébastien Truchet, a French Dominican priest and mathematician who studied them in 1704. He was interested in how simple square tiles with diagonal lines, when placed in random orientations, could create complex and aesthetically pleasing patterns. The modern resurgence of interest in Truchet tiles came with the work of Cyril Stanley Smith (1987) and their adoption by the generative art and creative coding communities. In GLSL, Truchet tiles are easy to implement and produce mesmerizing results.

6.3.7 Moire Patterns and Interference

Moire patterns arise from the interference between two similar repeating patterns. When two sets of concentric rings are centered at slightly different points, the phase differences create large-scale interference fringes that are far more complex than either pattern alone:

float d1 = length(uv - vec2(-0.1, 0.0));
float d2 = length(uv - vec2( 0.1, 0.0));
float moire = sin(d1 * 60.0) * 0.5 + sin(d2 * 60.0) * 0.5;

Moire patterns are a real-world phenomenon: you can see them when two chain-link fences overlap, when a fine-striped shirt is filmed on TV, or when two glass panes with etched lines are overlaid. In mathematics, they arise from the beat frequency between two similar periodic signals:

$$\sin(f_1 x) + \sin(f_2 x) = 2 \cos\!\left(\frac{f_1 - f_2}{2} x\right) \sin\!\left(\frac{f_1 + f_2}{2} x\right)$$

When $f_1$ and $f_2$ are close, the cosine term produces a slow oscillation (the Moire fringe) that modulates the fast sine pattern. In 2D with concentric rings, this produces spectacular hyperbolic interference fringes.

Interference patterns more generally occur whenever two or more wave-like patterns are combined. The resulting pattern contains spatial frequencies not present in either original, creating richness from simplicity — a theme that connects directly to cellular automata.

6.3.8 Rotation

Rotation transforms the coordinate space before computing the pattern. The standard 2D rotation matrix is:

$$R(\theta) = \begin{pmatrix} \cos\theta & -\sin\theta \\ \sin\theta & \cos\theta \end{pmatrix}$$

vec2 rotate2D(vec2 p, float angle) {
  float c = cos(angle);
  float s = sin(angle);
  return mat2(c, s, -s, c) * p;
}

Applying this before a pattern computation rotates the entire pattern. Animate the angle with u_time for spinning effects:

// Spinning checkerboard
vec2 ruv = rotate2D(uv, u_time * 0.5);
float checker = mod(floor(ruv.x * 8.0) + floor(ruv.y * 8.0), 2.0);

You can also use rotation for more subtle effects, like creating kaleidoscope-style symmetry by rotating copies of the same pattern:

// 6-fold symmetry (kaleidoscope)
float angle = atan(uv.y, uv.x);
float sector = 6.28318 / 6.0;            // 60 degrees
angle = mod(angle, sector);               // Fold into one sector
angle = abs(angle - sector * 0.5);        // Mirror within sector
vec2 kuv = vec2(cos(angle), sin(angle)) * length(uv);

6.3.9 Noise and Randomness

GLSL does not have a built-in random number generator. Instead, shader artists use hash functions — deterministic functions that produce pseudo-random output from their input. A common 2D hash function:

float hash(vec2 p) {
  return fract(sin(dot(p, vec2(127.1, 311.7))) * 43758.5453);
}

This takes a 2D position and returns a pseudo-random float in $[0, 1)$. Despite being completely deterministic (the same input always gives the same output), the output appears random. This is used for:

  • Random initial states for CAs on the GPU
  • Adding variation to tiled patterns (each tile gets a different random value from its cell index)
  • Building value noise and gradient noise (Perlin noise) for organic textures
⚠ Common Mistake

The hash function above relies on floating-point precision of the sin and dot functions. On some GPUs (especially mobile), the precision is lower, and the "random" values may show visible patterns or banding. For more robust hashing, use integer-based hash functions or multiple iterations.

6.4 Pattern Composition

Individual patterns are building blocks. The real creative power comes from composing multiple patterns together. If each pattern is a function $f_i : \mathbb{R}^2 \to [0, 1]$, composition creates a new function from the old ones. There are several fundamental composition techniques:

6.4.1 Arithmetic Composition

The simplest way to combine patterns is through arithmetic operations on their output values. If we treat each pattern as a function that maps position to a brightness value in $[0, 1]$, then:

  • Multiplication ($C = A \times B$) creates intersection-like effects: both patterns must be bright for the result to be bright. If either pattern is dark (0), the result is dark. This is useful for masking and for creating patterns that appear only where two conditions are simultaneously met.
  • Addition ($C = \text{clamp}(A + B, 0, 1)$) creates union-like effects: either pattern contributes brightness. Bright regions from both patterns appear in the result. Clamping prevents values from exceeding 1.0.
  • Subtraction ($C = \text{clamp}(A - B, 0, 1)$) carves one pattern out of another. Pattern B's bright regions are "cut" from pattern A. This is useful for creating shapes with holes or borders.
  • Maximum ($C = \max(A, B)$) takes the brightest value at each pixel. Similar to addition but preserves the original brightness levels without clipping.
  • Minimum ($C = \min(A, B)$) takes the darkest value. Creates a strict intersection: only where both patterns are bright.
  • XOR / Mod 2 ($C = \text{mod}(A + B, 2)$) creates alternating interference effects. Where both patterns are 0 or both are 1, the result is 0; where they differ, the result is 1. This produces striking Op-Art-style interference fringes.
  • mix() ($C = \text{mix}(A, B, t)$) blends smoothly between two patterns using a third value $t$ as a crossfader. When $t$ itself is a pattern, you get spatial blending where different regions of the screen show different patterns.

6.4.2 Masking

Use one pattern as a mask to control where another pattern appears. A mask is simply a pattern used as a multiplier: where the mask is 1.0, the other pattern shows through; where the mask is 0.0, the result is black. For example, use a circle as a mask for a stripe pattern:

float stripes = mod(floor(uv.y * 20.0), 2.0);
float mask = 1.0 - smoothstep(0.3, 0.31, length(uv));
float result = stripes * mask;  // stripes only inside the circle

Masking is multiplication, which means it can also be used for more subtle effects. A soft-edged mask (created with a wide smoothstep transition) creates a gradual fade rather than a hard cutoff. You can use any pattern as a mask: distance fields, angular patterns, noise, or even the output of another composition.

A common use of masking is creating windowed patterns — a complex pattern that only appears in certain regions of the screen, with a simpler background elsewhere:

// Complex pattern visible only in a ring shape
float ring = smoothstep(0.2, 0.21, d) * (1.0 - smoothstep(0.35, 0.36, d));
vec3 result = mix(backgroundColor, patternColor, ring);

6.4.3 Domain Warping

Domain warping is one of the most powerful and visually stunning composition techniques. Instead of combining the outputs of two patterns, you use one pattern to distort the input coordinates of another. This creates organic, fluid effects that are impossible to achieve with simple arithmetic:

$$\text{color} = f\bigl(\mathbf{uv} + g(\mathbf{uv})\bigr)$$

where $f$ is the base pattern and $g$ is the warping function. The result looks like the base pattern has been stirred, melted, or viewed through rippled glass.

The warp function $g$ is typically built from sin/cos waves that vary across the canvas. The warpAmount parameter controls the intensity of the distortion. Small values (0.01–0.05) give a subtle wobble; large values (0.2+) produce dramatic swirling. Adding u_time to the wave phases animates the distortion.

Domain warping can be iterated (applied recursively) for increasingly organic results. Inigo Quilez demonstrated that applying warp-of-warp-of-warp produces patterns resembling clouds, marble, and geological strata:

// First warp
vec2 q = vec2(fbm(uv), fbm(uv + vec2(1.7, 9.2)));
// Second warp using q
vec2 r = vec2(fbm(uv + 4.0*q), fbm(uv + 4.0*q + vec2(8.3, 2.8)));
// Final pattern
float f = fbm(uv + 4.0*r);

(Here fbm stands for fractal Brownian motion, a noise-based function we will study in later chapters. The same principle works with simpler warp functions.)

6.4.4 Combining Rings and Fans

One of the most visually striking compositions combines radial (rings) and angular (fans) patterns. Using XOR (mod 2 addition) creates a complex interference pattern reminiscent of Op Art:

float d = length(uv);
float angle = atan(uv.y, uv.x);

float rings = step(0.0, sin(d * 30.0 - u_time * 2.0));
float fan   = step(0.0, sin(angle * 8.0 + u_time));

// XOR combination
float pattern = mod(rings + fan, 2.0);

This XOR combination produces a pattern where the rings and fans create a constantly shifting "chessboard in polar coordinates." Where both patterns agree (both black or both white), the result is black; where they disagree, the result is white. The animation creates a hypnotic spinning effect. This type of pattern is central to Op Art (Optical Art), the art movement pioneered by Bridget Riley and Victor Vasarely in the 1960s, which used precise geometric patterns to create visual illusions of movement and depth.

The XOR technique also connects directly to cellular automata: Rule 90 (the Sierpinski triangle rule) is exactly XOR applied to the left and right neighbors. Both the visual XOR patterns in GLSL and the computational XOR in CAs demonstrate the same principle: combining two simple regular structures through XOR produces complex, fractal-like output.

6.4.5 Layered Composition

Complex shader art typically uses many layers composed together. A typical approach:

  1. Background gradient: A smooth color wash based on position or distance
  2. Primary pattern: The main geometric pattern (e.g., tiled shapes)
  3. Secondary pattern: An overlay or modifier (e.g., domain warping)
  4. Lighting/shading: Simulated 3D effects using distance fields
  5. Vignette: Darkening the edges for visual focus
  6. Color grading: Final color adjustments (contrast, saturation, palette mapping)

Each layer is computed independently and combined using mix, multiplication, or addition. The vignette effect (darkening edges) is particularly common:

// Simple vignette: multiply by (1 - distance^2)
float vignette = 1.0 - dot(uv, uv) * 2.0;
color *= clamp(vignette, 0.0, 1.0);
💡 Key Concept

Thinking in Layers: When designing a shader pattern, sketch out the layers on paper first. What is the background? What is the primary pattern? How will you add visual interest and color? This layer-by-layer approach makes complex shaders manageable and debuggable — you can comment out individual layers to isolate problems.

6.5 Cellular Automata: History and Theory

6.5.1 Origins: Von Neumann and Ulam

📜 Historical Note

In the 1940s, John von Neumann was working on the problem of self-reproducing machines — could a machine build a copy of itself? His colleague Stanislaw Ulam suggested that instead of thinking about physical machines with moving parts, von Neumann should work in an abstract mathematical universe: a grid of cells, each following simple rules. This was the birth of cellular automata.

Von Neumann designed a cellular automaton with 29 states per cell that was capable of universal construction — it could build any pattern within the grid, including a complete copy of itself. The automaton operated on a 2D grid and included components analogous to a construction arm, a blueprint reader, and a copy mechanism. His work established two foundational ideas:

  1. Self-reproduction is possible in a computational system. A pattern of cells can contain the "blueprint" for itself and the "machinery" to read that blueprint and assemble a copy. Von Neumann proved that such a machine would need at least two parts: the constructor (which builds based on instructions) and the instructions themselves (which get copied verbatim to the offspring).
  2. Simple rules can support universal computation. If the rules are rich enough, any computation whatsoever can be carried out within the automaton.

These ideas anticipated concepts in molecular biology by over a decade. When Watson and Crick discovered the structure of DNA in 1953, biologists realized that living cells work exactly as Von Neumann had described: DNA is the blueprint, the ribosome is the constructor that reads instructions and builds proteins, and when a cell divides, the DNA is copied verbatim to the daughter cell. Von Neumann had derived the logical necessity of this architecture from first principles, before anyone knew how biology actually implemented it.

In the 1960s and 1970s, cellular automata gained wider attention. Edgar Codd simplified Von Neumann's design to 8 states. John Conway created the Game of Life in 1970, which brought CAs to popular awareness. Then in the 1980s, Stephen Wolfram began a systematic study of the simplest possible CAs — the elementary (1D) rules — and discovered that even these minimal systems could produce behavior of remarkable complexity. Wolfram's work culminated in his 2002 book A New Kind of Science, a 1,280-page treatise arguing that simple computational programs (particularly CAs) are the key to understanding complexity in nature.

Today, CAs are used in a wide range of applications beyond theoretical computer science:

  • Cryptography: Rule 30 has been used as a pseudorandom number generator
  • Traffic modeling: Rule 184 and Nagel-Schreckenberg models simulate traffic flow and jamming
  • Physics: Lattice gas automata simulate fluid dynamics; lattice Boltzmann methods are CA-based fluid simulators used in engineering
  • Biology: CAs model tumor growth, epidemics, forest fires, and predator-prey dynamics
  • Art and design: CAs generate patterns for textiles, architectural facades, and generative art
  • Music: CA rules have been used to generate melodies, rhythms, and harmonic progressions

6.5.2 Formal Definition

📖 Definition

Cellular Automaton (CA) — A discrete computational system consisting of four components:

  1. Lattice: A regular grid of cells (1D line, 2D grid, or higher dimensional)
  2. States: A finite set of possible values for each cell (e.g., {0, 1} or {dead, alive})
  3. Neighborhood: A fixed pattern defining which cells influence each cell (e.g., the 3-cell neighborhood in 1D, or the Moore 8-cell neighborhood in 2D)
  4. Transition Rule: A function $f$ that computes a cell's next state from its neighborhood's current states

All cells update synchronously (simultaneously) at each time step.

The synchronous update is critical. If cells updated one at a time (asynchronously), the behavior would depend on the order of updates, breaking the parallel symmetry that gives CAs their character. Asynchronous CAs exist and are studied (they model certain physical systems more accurately), but they produce fundamentally different dynamics and are much harder to analyze.

Formally, given a lattice $L$, a state space $S$, a neighborhood function $N$, and a transition rule $f$, the CA evolves as:

$$s_i^{t+1} = f\!\left(s_{N(i)}^t\right) \qquad \text{for all cells } i \in L$$

where $s_i^t$ is the state of cell $i$ at time $t$, and $s_{N(i)}^t$ denotes the states of all cells in the neighborhood of $i$ at time $t$. The superscript $t$ on the right side (not $t+1$) emphasizes the synchronous nature: all reads come from time $t$, all writes go to time $t+1$.

6.5.3 Neighborhoods

Different neighborhood definitions lead to different CA behaviors:

1D Neighborhoods

  • Radius-1: Left, center, right (3 cells). This is the "elementary" CA neighborhood.
  • Radius-2: Two cells on each side plus center (5 cells). $2^{32} = 4$ billion possible rules.

2D Neighborhoods

  • Von Neumann neighborhood: The 4 orthogonal neighbors (N, S, E, W). Named after John von Neumann. Produces more rectilinear patterns.
  • Moore neighborhood: All 8 surrounding cells (including diagonals). Used by the Game of Life. Produces more isotropic (rotationally symmetric) behavior.
  • Extended neighborhoods: All cells within radius $r$ (e.g., a 5x5 square for $r = 2$). More neighbors = smoother behavior but exponentially more possible rules.

6.5.4 The Space of Possible Rules

For a 1D CA with $k$ states and a neighborhood of $n$ cells, the number of possible rules is:

$$N_{\text{rules}} = k^{k^n}$$

For elementary CAs ($k = 2$, $n = 3$): $2^{2^3} = 2^8 = 256$ rules. This is small enough to enumerate exhaustively, which is exactly what Wolfram did.

For a 2D CA with 2 states and the Moore neighborhood ($n = 9$ including the cell itself): $2^{2^9} = 2^{512} \approx 1.3 \times 10^{154}$. This is incomprehensibly large — far more than the number of atoms in the observable universe ($\sim 10^{80}$). Exhaustive enumeration is impossible; we can only explore specific rules of interest (like Life's B3/S23).

This combinatorial explosion means that the Game of Life, remarkable as it is, represents one point in an unimaginably vast space of 2D CAs. We have barely scratched the surface of what 2D CAs can do.

6.5.5 Totalistic and Outer Totalistic Rules

For 2D CAs with more than a handful of neighbors, specifying a separate output for every possible neighborhood configuration is impractical. A practical simplification is the totalistic approach: instead of considering the exact configuration of neighbors, only count the total number of live neighbors.

In a totalistic rule, the next state depends only on the sum of the neighbor states (including the cell itself). In an outer totalistic rule, the next state depends on the current state of the cell AND the sum of its neighbors' states (but not the exact arrangement). The Game of Life is outer totalistic: it distinguishes between a live cell and a dead cell, but treats all arrangements with the same neighbor count identically.

This is a dramatic reduction in the number of rules. For the Moore neighborhood with 2 states:

  • Full rule: $2^{512}$ possibilities (every configuration specified independently)
  • Outer totalistic: $2^{18}$ = 262,144 possibilities (for each of the 2 cell states and 9 possible neighbor counts, specify an output)
  • Life-like (B/S notation): $2^{18}$ = 262,144 possibilities (same as outer totalistic for 2-state systems)

262,144 is large but manageable — computational surveys have explored significant fractions of this space, discovering thousands of interesting Life-like rules.

6.5.6 Boundary Conditions

Since any real grid is finite, we must decide how to handle cells at the edges. The choice of boundary conditions affects the dynamics significantly:

  • Toroidal (periodic): The grid wraps around — the right edge connects to the left, and the top connects to the bottom. Every cell has exactly the same number of neighbors. This is the most common choice for studying CAs because it eliminates edge artifacts.
  • Fixed: Cells outside the grid are assumed to always be in a fixed state (usually dead/0). This creates a "wall" effect at the boundaries.
  • Reflective: Cells at the edge "mirror" their interior neighbors. The edge acts like a mirror.
  • Infinite: Conceptually, the grid extends infinitely in all directions. In practice, this is implemented with dynamic data structures that grow as the CA expands (used in advanced Life simulators like Golly with the HashLife algorithm).

For the exercises and implementations in this chapter, we use toroidal boundary conditions unless otherwise stated. In code, toroidal wrapping is implemented with the modulo operator:

// Toroidal wrapping in JavaScript
let nx = (x + dx + cols) % cols;  // wraps x to [0, cols-1]
let ny = (y + dy + rows) % rows;  // wraps y to [0, rows-1]

// In GLSL with texture coordinates (automatically wraps if
// texture wrap mode is set to GL_REPEAT)

The + cols before the modulo handles the case where x + dx is negative (i.e., going past the left edge). In JavaScript, -1 % cols returns a negative number, but (-1 + cols) % cols correctly wraps to cols - 1.

6.6 Elementary Cellular Automata

6.6.1 Wolfram's Numbering System

Stephen Wolfram introduced a systematic numbering scheme for the 256 elementary (1D, 2-state, 3-neighbor) CAs. Each rule is identified by a number from 0 to 255. The rule number, written in binary, directly encodes the transition table.

Consider the 8 possible 3-cell neighborhoods, listed from 111 (=7) down to 000 (=0):

Neighborhood111110101100011010001000
Rule 3000011110
Rule 9001011010
Rule 11001101110

Reading Rule 30's output row as binary: 00011110 = 30 in decimal. This is how the rule gets its name. Similarly, Rule 90 = 01011010 in binary, and Rule 110 = 01101110.

💡 Key Concept

The rule number IS the lookup table. To decode a rule: convert the rule number to 8-bit binary. Bit $i$ of the result tells you the output when the neighborhood (left, center, right) has the binary value $i$. The neighborhood value is computed as $L \cdot 4 + C \cdot 2 + R$.

6.6.2 Implementation

The algorithm is straightforward:

  1. Initialize a 1D array of cells, typically with a single "on" cell in the center
  2. For each generation, compute the next state of each cell by looking up its 3-cell neighborhood in the rule table
  3. Draw each generation as a row of pixels, building a 2D image over time (time flows downward)

The key implementation detail is converting a rule number into a lookup table. The rule number (0–255) is treated as an 8-bit binary number. Bit $i$ of this number gives the output for neighborhood configuration $i$:

// Convert rule number to lookup table
function ruleToTable(ruleNum) {
  let table = new Array(8);
  for (let i = 0; i < 8; i++) {
    table[i] = (ruleNum >> i) & 1;  // Extract bit i
  }
  return table;
}

// Apply rule to a neighborhood
// The neighborhood value L*4 + C*2 + R indexes into the table
function applyRule(table, left, center, right) {
  let idx = left * 4 + center * 2 + right;
  return table[idx];
}

Let us trace through an example. For Rule 90 (binary: 01011010):

  • Bit 0 (neighborhood 000): 0 — dead stays dead
  • Bit 1 (neighborhood 001): 1 — right neighbor alive, center becomes alive
  • Bit 2 (neighborhood 010): 0 — only center alive, it dies
  • Bit 3 (neighborhood 011): 1 — center and right alive, stays alive
  • Bit 4 (neighborhood 100): 1 — left neighbor alive, center becomes alive
  • Bit 5 (neighborhood 101): 0 — left and right alive, center dies
  • Bit 6 (neighborhood 110): 1 — left and center alive, stays alive
  • Bit 7 (neighborhood 111): 0 — all alive, center dies

Notice that Rule 90 is equivalent to XOR: the next state equals left XOR right (the center cell's own state is irrelevant!). This produces the Sierpinski triangle, which is also generated by Pascal's triangle modulo 2 — a beautiful mathematical connection between number theory, combinatorics, and cellular automata.

📐 Theorem

Rule 90 and Pascal's Triangle. The output of Rule 90 from a single-cell initial condition is identical to Pascal's triangle modulo 2. Entry $(n, k)$ of Pascal's triangle (the binomial coefficient $\binom{n}{k}$) is the value of the cell at position $k$ in generation $n$ of Rule 90 (starting from a single cell at position 0). The Sierpinski triangle emerges because $\binom{n}{k} \bmod 2 = 1$ if and only if every bit of $k$ is also a bit of $n$ (Kummer's theorem). This produces a self-similar fractal with fractal dimension $\frac{\log 3}{\log 2} \approx 1.585$.

Bit Manipulation Approach

Since the rule table is encoded directly in the bits of the rule number, you can also implement the rule lookup without an explicit table, using bitwise operations:

function wolframRule(ruleNumber, left, center, right) {
  let idx = left * 4 + center * 2 + right;
  return (ruleNumber >> idx) & 1;
}

This single-line implementation extracts bit idx from the rule number. It is equivalent to building the lookup table and indexing into it, but more concise.

6.7 Conway's Game of Life

6.7.1 Rules

📜 Historical Note

The Game of Life was invented by mathematician John Horton Conway in 1970 and popularized by Martin Gardner's column in Scientific American. Conway spent years searching for rules that were neither too chaotic (most initial patterns die out) nor too explosive (patterns grow without bound). The result: a set of rules so simple they fit in a sentence, yet capable of supporting universal computation.

The Game of Life operates on a 2D grid with the Moore neighborhood (8 surrounding cells). Its rules are elegant in their simplicity:

  1. Birth: A dead cell with exactly 3 live neighbors becomes alive
  2. Survival: A live cell with 2 or 3 live neighbors stays alive
  3. Death by isolation: A live cell with fewer than 2 live neighbors dies
  4. Death by overcrowding: A live cell with more than 3 live neighbors dies

In the notation used for "Life-like" automata, these rules are written as B3/S23 (Birth with 3 neighbors, Survival with 2 or 3).

Let us trace through one generation to see how these rules work. Consider this 5x5 grid (# = alive, . = dead):

The center cell (row 2, col 2) is alive with 4 live neighbors. Since 4 is not 2 or 3, it dies by overcrowding. The cell at (row 1, col 1) is dead with 2 live neighbors. Since 2 is not 3, it stays dead. The cell at (row 0, col 2) is dead with 1 live neighbor — stays dead. The cell at (row 1, col 2) is alive with 2 live neighbors — survives.

The result after one generation is a "plus" pattern that has changed shape — demonstrating how even simple configurations evolve in non-obvious ways.

💡 Key Concept

Why B3/S23? Conway tested many different birth/survival rules before settling on B3/S23. His criteria: the rules should produce behavior that is (1) not too boring (patterns should not immediately die or reach static equilibrium) and (2) not too chaotic (patterns should not explode to fill the grid). B3/S23 sits at the sweet spot — it is a Class IV system at the edge of chaos, capable of supporting stable structures, moving patterns, and universal computation.

  • Methuselahs: Small patterns that take a very long time to stabilize. The "R-pentomino" (5 cells) takes 1,103 generations to reach a stable state and produces 6 gliders along the way.
  • Puffers: Moving patterns that leave a trail of debris behind them, like a spaceship with exhaust.
  • Rakes: Puffers whose debris consists of spaceships — essentially mobile glider guns.

Yet these complex, named, interacting structures emerge from the collective behavior of millions of cells following those simple rules. This is the deepest lesson of the Game of Life: complexity is not designed; it emerges.

To put the richness of Life in perspective: the LifeWiki (a community-maintained encyclopedia) catalogs thousands of named patterns, including over 50 known spaceships of various sizes and speeds, oscillators with periods ranging from 2 to hundreds of thousands of generations, and patterns that compute prime numbers, display digital clocks, and simulate other cellular automata — including simulating the Game of Life itself.

📐 Theorem

Turing Completeness of Life. The Game of Life is Turing complete, meaning it can simulate any computation that a Turing machine can perform. This was demonstrated by constructing logic gates (AND, OR, NOT) from glider streams, which can then be combined into arbitrary circuits. The existence of glider guns (to produce signals) and collision reactions (to process signals) provides the necessary computational primitives. In principle, the Game of Life can simulate itself, run arbitrary programs, and even compute non-computable questions with the same limitations as any other Turing machine.

6.8 Wolfram's Classification

After exhaustively exploring all 256 elementary CAs (and many more complex systems), Stephen Wolfram proposed a classification of CA behavior into four classes. This classification applies not just to 1D elementary CAs but to cellular automata of all types, including the Game of Life.

6.8.1 Class I: Uniform (Fixed Point)

From almost any initial condition, the system quickly evolves to a uniform, homogeneous state. All initial structure is destroyed — erased by the rule. The system reaches a fixed point where every cell has the same value, and nothing changes thereafter.

Physical analog: A gas cooling to a uniform temperature. A drop of dye diffusing until the liquid is uniformly colored.

Computational power: None. These systems cannot store, transmit, or process information.

Examples: Rule 0 (all cells die), Rule 255 (all cells live), Rule 32, Rule 160.

6.8.2 Class II: Periodic (Oscillating)

The system evolves to a set of simple, stable, or periodic structures. Different regions of the grid may settle into different fixed points or oscillation cycles, but each region evolves independently. Changes remain localized — a perturbation at one location does not propagate to affect distant cells. There is no long-range communication or information transfer.

Physical analog: A crystal lattice. A repeating wallpaper pattern. Pendulums that settle into fixed oscillation frequencies.

Computational power: Limited to finite-state machines. Each cell is essentially an independent oscillator or fixed point. The system can "remember" its initial conditions (locally) but cannot compute anything non-trivial.

Examples: Rule 90 (produces the Sierpinski triangle fractal — beautiful but predictable), Rule 184 (traffic model where particles move right), Rule 4.

Note that Rule 90 generates a fractal, which may seem "complex," but it is entirely predictable and can be computed in closed form: $\text{cell}(x, t) = \text{XOR}(L, R)$, which is equivalent to Pascal's triangle modulo 2.

6.8.3 Class III: Chaotic (Random)

The system produces aperiodic, seemingly random behavior. Structures appear and disappear unpredictably. The output looks like noise — statistical tests for randomness are often passed by the output. Crucially, small changes to the initial condition produce drastically different outcomes, a property known as sensitive dependence on initial conditions (the "butterfly effect").

Physical analog: Turbulent fluid flow. Static on a television. Brownian motion.

Computational power: These systems are effective pseudorandom number generators, but their apparent randomness makes them poor at organized computation. Information enters the system but is immediately scrambled and distributed.

Examples: Rule 30 (used as a PRNG in Mathematica), Rule 45, Rule 73, Rule 150.

Wolfram famously used Rule 30 to generate "random" numbers in his software because, despite being completely deterministic, its center column passes statistical tests for randomness as well as most dedicated pseudorandom algorithms.

The apparent randomness of Class III systems raises a deep philosophical question: if a deterministic system can produce output indistinguishable from true randomness, what does "random" even mean? The answer, in the context of computational complexity theory, is that randomness is not an intrinsic property of a sequence but rather a measure of the computational resources needed to predict the next element. If no shortcut exists for prediction (short of running the actual computation), the output is effectively random, even though the underlying process is deterministic.

6.8.4 Class IV: Complex (Edge of Chaos)

The system produces complex, long-lived structures that interact in intricate ways. There is a mixture of order and chaos: stable regions coexist with moving structures, and their interactions are unpredictable in the general case. This class is often described as existing at the "edge of chaos" — balanced between the rigid predictability of Class II and the formless randomness of Class III.

Physical analog: Living systems. Weather patterns. Economies. The brain.

Computational power: These systems can perform universal computation. They can store information (in stable structures), transmit it (via moving structures), and process it (via collisions between structures). This makes the evolution of Class IV automata computationally irreducible — there is no shortcut to predicting their future state; you must simulate every step.

Examples: Rule 110 (proven Turing complete by Matthew Cook, 2004), Conway's Game of Life, Rule 54.

💡 Key Concept

The Edge of Chaos Hypothesis: Class IV systems are special because they balance order and randomness. This balance enables computation — the ability to store, transmit, and process information. Chris Langton proposed that this transition region between order and chaos is where the most complex and interesting behavior occurs, and where life itself may exist. Class IV automata can support universal computation precisely because they can maintain and manipulate structured information over time.

6.8.5 How to Classify a Rule

In practice, classifying a CA rule involves running it from various initial conditions and observing the long-term behavior:

  1. Start from a single cell: Initialize with one "on" cell in an otherwise empty grid. Class I rules will quickly die; Class II will produce a small periodic or fixed pattern; Class III will produce an expanding, random-looking triangle; Class IV will produce a complex mixture.
  2. Start from random initial conditions: Fill the grid with 50% random on/off cells. Class I rules converge to uniform; Class II to periodic patches; Class III to persistent chaos; Class IV to a mixture of stable structures and moving patterns.
  3. Test sensitivity: Run the same rule twice with initial conditions that differ by one cell. Class I and II: the difference has no effect (or only local effect). Class III: the outputs diverge rapidly (sensitive dependence). Class IV: the difference may propagate in complex ways.

The classification is not always clear-cut. Some rules are borderline between classes, and there is ongoing debate about the precise mathematical criteria for each class. In practice, visual inspection and statistical analysis of the output are the primary classification methods.

Wolfram proposed that the classification is universal — it applies not just to CAs but to all computational systems: differential equations, Turing machines, biological systems, and economic models. The claim is controversial but deeply thought-provoking: it suggests a unified framework for understanding complexity across all domains.

An interesting question for the artistically inclined: which class produces the most aesthetically interesting output? Many artists find Class IV systems the most compelling, because they combine structure (readable, identifiable forms) with surprise (unpredictable evolution). Class III systems can also be visually striking in their chaotic texture, while Class II systems produce the most "designed-looking" patterns. Class I is generally too simple to be interesting on its own, though it can serve as a dramatic contrast element when combined with other patterns.

The relationship between Wolfram's classification and aesthetic judgment is itself an open and fascinating question at the intersection of art, mathematics, and cognitive science. Why do humans find certain levels of complexity more beautiful than others? The "edge of chaos" hypothesis suggests a deep connection between computational complexity and perceptual interest.

6.9 Cellular Automata in GLSL

6.9.1 The Frame Feedback Technique

The key challenge in implementing CAs on the GPU is that a cell needs to read its neighbors' states from the previous frame. In normal GLSL rendering, each pixel is independent and cannot read other pixels in the current frame. The solution is frame feedback (also called ping-pong buffering or double-buffered rendering):

  1. Initialize: Create two textures (framebuffers) of the same size. Fill one with the initial CA state (each pixel represents one cell, with red channel = alive/dead).
  2. Read: In the fragment shader, bind the previous frame's texture as a uniform sampler. Use texture2D() to sample the neighbors from this texture.
  3. Compute: Apply the CA transition rule to compute the new state of each cell.
  4. Write: Output the result to the current framebuffer.
  5. Swap: Exchange the two framebuffers. The current output becomes the next frame's input.

This "ping-pong" technique ensures that all cells read from the same previous state (synchronous update), while writing to a separate buffer. Without double-buffering, cells updated early in the frame would contaminate the reads of cells updated later.

On ShaderToy, this is achieved using the multi-pass buffer system: Buffer A can read its own previous output via the iChannel0 sampler. The "Common" tab is used for shared code, and the "Image" tab renders the final visualization (often with colorization and scaling applied to the raw CA state from Buffer A).

The performance advantage is dramatic. A CPU-based Game of Life on a 1000x1000 grid must iterate over 1 million cells each frame. A GPU-based version processes all 1 million cells simultaneously, achieving frame rates 100-1000x faster on the same grid size. This is why GPU-based CAs can run at 4K resolution in real time.

💡 Key Concept

Texture as State: In GPU-based CAs, the texture IS the grid. Each texel (texture pixel) represents one cell. The red channel stores the cell's state. Sampling a texel at an offset reads a neighbor's state. This is a fundamentally different use of textures: instead of storing images, we store computational state that evolves over time.

6.9.2 Game of Life Shader

Here is the core fragment shader for Game of Life on the GPU. The key technique is to sample the 8 neighbors from the previous frame's texture using texture2D(). Each neighbor is at a known offset from the current pixel:

// In a real implementation with frame feedback:
vec2 pixel = 1.0 / u_resolution;  // Size of one pixel in UV space

// Sample the 8 neighbors
float n = 0.0;
n += texture2D(u_state, uv + vec2(-pixel.x, -pixel.y)).r;  // top-left
n += texture2D(u_state, uv + vec2(      0.0, -pixel.y)).r;  // top
n += texture2D(u_state, uv + vec2( pixel.x, -pixel.y)).r;  // top-right
n += texture2D(u_state, uv + vec2(-pixel.x,       0.0)).r;  // left
n += texture2D(u_state, uv + vec2( pixel.x,       0.0)).r;  // right
n += texture2D(u_state, uv + vec2(-pixel.x,  pixel.y)).r;  // bottom-left
n += texture2D(u_state, uv + vec2(      0.0,  pixel.y)).r;  // bottom
n += texture2D(u_state, uv + vec2( pixel.x,  pixel.y)).r;  // bottom-right

This sums the red channel values of the 8 surrounding pixels, giving us the neighbor count as a float. We then apply the Game of Life rules using float comparisons (not integer equality, to avoid precision issues). 6.9.3 Continuous Cellular Automata: SmoothLife

SmoothLife (Rafler, 2011) extends the Game of Life from a discrete grid with binary states to continuous space with continuous states:

  • Continuous states: Cell values can be any real number between 0.0 and 1.0, rather than just "dead" or "alive"
  • Continuous neighborhoods: Instead of counting the 8 discrete Moore neighbors, SmoothLife uses circular regions with inner radius $r_i$ (the cell's own area) and outer radius $r_o$ (the neighborhood). The "neighbor count" becomes a weighted integral over the annular region.
  • Smooth transition functions: Birth and survival thresholds use smooth sigmoid functions instead of hard integer cutoffs. This removes the discreteness artifacts and produces fluid, organic behavior.

The transition function in SmoothLife uses a sigmoid of the form:

$$\sigma(x, a, \alpha) = \frac{1}{1 + e^{-\frac{4}{\alpha}(x - a)}}$$

where $a$ is the threshold and $\alpha$ controls the smoothness of the transition. As $\alpha \to 0$, the sigmoid approaches a step function, recovering the discrete Game of Life rules. For positive $\alpha$, the transition is smooth, allowing cells to be "partially alive" — a value of 0.7 means the cell is 70% alive.

The birth and death rules are composed from these sigmoids in a way that generalizes the B3/S23 rule to continuous values. The key parameters are:

  • $b_1, b_2$: birth thresholds (the integral must be between these values for a dead cell to come alive)
  • $d_1, d_2$: death thresholds (the integral must be between these values for a live cell to survive)
  • $r_i, r_o$: inner and outer radii of the neighborhood annulus
  • $\alpha_n, \alpha_m$: smoothness parameters for the neighborhood and transition functions

The result is mesmerizing: organic, blob-like patterns that resemble biological processes. You can observe:

  • Cell division: A blob pinches in the middle and separates into two independent blobs
  • Growth: Blobs expand when conditions are favorable (surrounded by enough empty space)
  • Movement: Blobs glide across the field, changing direction when they encounter obstacles
  • Death: Blobs shrink and disappear when surrounded by too many or too few neighbors
  • Interaction: Blobs can merge, bounce off each other, or annihilate on collision

SmoothLife demonstrates that the principles of cellular automata extend naturally beyond discrete grids. The discreteness of the Game of Life is not essential — the essential ingredient is local rules applied uniformly.

SmoothLife runs beautifully on the GPU as a fragment shader because the integral over the circular neighborhood can be computed efficiently using texture sampling with mipmaps, or approximated with a finite sum over sampled points. Several implementations exist on ShaderToy (search for "SmoothLife" or "Lenia"), and they are among the most visually striking demos on the platform. Lenia, created by Bert Chan, extends SmoothLife with more flexible growth functions and kernel shapes, producing an even richer ecosystem of artificial life forms.

6.9.4 Other GPU-Accelerated CAs

The frame feedback technique is not limited to the Game of Life. Any CA can be ported to the GPU using the same approach:

  • Elementary CAs: Though 1D, they can be rendered as a 2D texture where each row is one generation. The shader reads the previous row to compute the next.
  • Multi-state CAs: Use the RGBA channels to store states beyond binary. Red might encode the primary state, green the "age" of a cell for colorful visualization.
  • Larger-than-Life: CAs with extended neighborhoods (radius > 1). On the GPU, sampling more neighbors is inexpensive since texture reads benefit from hardware caching.
  • Continuous CAs (SmoothLife, Lenia): Naturally suited to GPU implementation because continuous states map directly to floating-point texel values.
  • Multi-layered CAs: Use multiple texture channels or multiple render passes to simulate interacting layers (like the reaction-diffusion systems in Chapter 7).

The common thread: if a system updates each cell based only on its local neighborhood, it maps naturally to a GPU fragment shader. The GPU does not care about the specific rules — it just needs to know which texels to read and what function to apply.

💡 Key Concept

Generalization Path: Discrete grid + binary states (Game of Life) → Discrete grid + continuous states (multi-state CAs) → Continuous space + continuous states (SmoothLife) → Continuous space + continuous states + multiple chemicals (Reaction-Diffusion systems, Chapter 7). Each generalization preserves the core principle of local rules producing emergent global behavior.

6.10.1 Parallel Computation

Both GLSL shaders and CAs are fundamentally parallel systems. Every pixel/cell computes independently based on local information. This parallelism is not an optimization trick — it is inherent to the model. The rules are defined locally, and the global pattern emerges from the collective behavior of all the local computations.

This correspondence is not merely theoretical. In practice, running CAs on the GPU via fragment shaders achieves performance that is 2–3 orders of magnitude faster than CPU implementations for large grids. A Game of Life grid of 4096 x 4096 cells (16 million cells) can be updated at 60 frames per second on a modern GPU, while a JavaScript implementation on the CPU would struggle with a 500 x 500 grid at the same frame rate.

The key insight is that both GLSL shaders and CAs are instances of the same abstract computational model: data-parallel local computation. Each processing unit (pixel/cell) reads local data, applies a function, and writes the result. No global coordination is needed. This is why GPUs — designed for exactly this kind of computation — are the natural hardware for both tasks.

📖 Definition

SIMD (Single Instruction, Multiple Data) — A parallel computing model where the same operation is applied simultaneously to many data elements. GPUs are SIMD architectures: the same fragment shader (single instruction) runs on thousands of pixels (multiple data) at once. Cellular automata are naturally SIMD because the same rule applies to every cell simultaneously.

6.10.2 From Rules to Patterns

The deep connection between these two domains is the principle of emergent complexity from simple rules:

  • In GLSL, the "rule" is a mathematical function. sin(length(uv) * 40.0) produces concentric rings. mod(floor(uv.x * 8.0) + floor(uv.y * 8.0), 2.0) produces a checkerboard. These patterns emerge from the uniform application of the function across all pixels.
  • In a CA, the rule is a lookup table or conditional. "If alive and 2-3 neighbors, survive" produces gliders, oscillators, and Turing-complete computation. These structures emerge from the uniform application of the rule across all cells, over many generations.

Both systems are deterministic — the same input always produces the same output. Yet both can produce behavior that appears random (Rule 30, domain-warped shaders). The complexity is not in the rules but in the interaction between the rules and the spatial/temporal structure.

This has a practical implication for creative computing: you do not need complex code to create complex art. Some of the most visually stunning shader pieces on ShaderToy are under 50 lines of code. The most captivating CA patterns come from rules that fit in a single sentence. The art lies not in piling on more code, but in choosing the right simple operations and combining them thoughtfully.

This is also why these techniques are so appealing for artistic exploration. Because the rules are simple, you can modify them quickly and see the results instantly. A small change to a frequency, a warp amount, or a threshold can transform the entire visual character of a piece. This tight feedback loop between intention and result is the essence of creative coding.

As an exercise in this philosophy, try taking any of the interactive demos in this chapter and making the smallest possible change that produces the most dramatic visual difference. Change one number, one operator, or one function call. You will often be surprised by how much a single modification can transform the output — this is emergence in action, and it is what makes these simple systems so endlessly fascinating to explore.

6.10.3 Patterns in Nature

Many patterns in the natural world can be understood through the same principles we have studied in this chapter:

  • Animal coat patterns (zebra stripes, leopard spots, giraffe patches): Alan Turing proposed in 1952 that these arise from reaction-diffusion systems, which are continuous analogs of cellular automata. We will explore this connection in Chapter 7.
  • Seashell patterns: The pigmentation patterns on certain cone snails (Conus textile) are almost identical to the output of 1D cellular automata like Rule 30. The shell grows at a constant rate, and the pigment-producing cells at the growing edge follow CA-like rules based on their neighbors.
  • Crystal growth: Snowflakes form six-fold symmetric patterns because water molecules attach to the growing crystal based on local conditions (temperature, humidity) at each point on the surface. This is a continuous-space analog of a 2D CA.
  • Moire and interference: Butterfly wing scales produce iridescent colors through interference of light waves at the nanoscale — the same principle as our GLSL Moire patterns, operating on electromagnetic waves instead of pixel values.
📜 Historical Note

The connection between cellular automata and natural patterns was one of Wolfram's central arguments in A New Kind of Science (2002). He proposed that many natural systems are best modeled not by differential equations (the traditional approach) but by simple computational programs — cellular automata, substitution systems, and similar discrete systems. Whether or not one accepts this strong claim, the visual correspondence between CA output and natural patterns is striking and suggestive.

6.10.4 Computational Irreducibility

One of Wolfram's most provocative ideas is computational irreducibility: for Class IV CAs (and many other systems), there is no shortcut to predicting the outcome. You cannot determine the state at time step $t$ without simulating all $t$ steps. The system is its own fastest simulator.

This has profound implications:

  • Science: Some systems may be inherently unpredictable not because we lack information, but because prediction requires as much computation as the system itself.
  • Art: Computationally irreducible systems are inherently surprising. Even the programmer who wrote the rules cannot predict what patterns will emerge without running the simulation. This makes CAs a genuine tool for discovery in creative computing.
  • Philosophy: Free will, creativity, and novelty in nature might be related to computational irreducibility. If the universe is performing a computation that cannot be shortcut, then the future is genuinely open and undetermined in a practical sense.

6.11.5 Tips for Shader Development

Regardless of which tool you use, these practices will accelerate your learning:

  • Start simple. Begin with a solid color, then add one operation at a time. Always understand each line before adding the next.
  • Visualize intermediate values. Output any intermediate value as a color to see what it looks like: gl_FragColor = vec4(vec3(d), 1.0); where d is whatever variable you want to inspect.

  • Read other people's shaders. ShaderToy's gallery is an incredible learning resource. Find a shader you like, read the code, and try modifying it.

6.12 Summary

This chapter covered two complementary approaches to generating complex visual patterns from simple rules:

GLSL Shaders use mathematical functions to determine the color of each pixel based on its position. Key techniques include:

  • Distance functions for circles and rings (length(), distance())
  • Periodic functions for stripes and grids (floor(), fract(), mod(), sin())
  • Polar coordinates for angular patterns (atan())
  • Rotation matrices for transforming coordinate space
  • Composition through arithmetic operations, masking, and domain warping
  • Anti-aliasing with smoothstep()

Cellular Automata use discrete grids where each cell updates based on its neighbors' states:

  • Elementary CAs (1D, 2-state, 3-neighbor) with Wolfram's 256 rules, each identified by a number 0–255 that directly encodes the lookup table
  • Conway's Game of Life (2D, B3/S23) with emergent structures: still lifes, oscillators, spaceships, guns, and Methuselahs
  • Life-like automata family: different B/S rules produce dramatically different behavior (HighLife, Seeds, Day & Night)
  • Wolfram's four classes: uniform (I), periodic (II), chaotic (III), complex (IV)
  • Class IV systems at the "edge of chaos" support universal computation; their evolution is computationally irreducible
  • GPU implementation via frame feedback (ping-pong buffering) in GLSL
  • SmoothLife: Generalization to continuous space and states, producing organic behavior

The unifying principle: complex, beautiful, and surprising behavior can emerge from simple rules applied uniformly across space.

Looking ahead, Chapter 7 will explore reaction-diffusion systems, which can be understood as a further generalization of cellular automata to continuous chemical concentrations. The same principle — local rules, parallel application, emergent complexity — will produce patterns like animal coat markings, coral growth, and chemical oscillations. The ideas in this chapter form the foundation for that exploration.

Chapter 6 Review Quiz

Test your understanding of GLSL patterns and cellular automata.

✍ Question 1

In a GLSL fragment shader, which operation creates a tiled repeating pattern from continuous UV coordinates?

fract(uv * frequency) is the standard technique for creating tiled patterns. Multiplying by a frequency controls how many tiles appear, and fract() wraps the coordinates back to $[0, 1)$ in each tile, creating seamless repetition. You can then define any pattern within a single cell and it will repeat across the entire canvas.

✍ Question 2

What is "domain warping" in the context of GLSL pattern creation?

Domain warping uses one function to distort the input coordinates of another: $f(\mathbf{uv} + g(\mathbf{uv}))$. Rather than combining pattern outputs (which is arithmetic composition), domain warping operates on the input space. This produces organic, fluid distortions that make patterns look melted, stirred, or alive. It is one of the most powerful techniques in shader art.

✍ Question 3

How many possible elementary cellular automata rules exist (1D, 2 states, 3-cell neighborhood)?

With a 3-cell neighborhood and 2 states, there are $2^3 = 8$ possible neighborhood configurations. Each configuration maps to an output of 0 or 1, giving $2^8 = 256$ possible rules. This is why Wolfram numbered them 0 through 255 — each number corresponds to a unique rule table when written in 8-bit binary.

✍ Question 4

Which elementary CA rule has been proven Turing complete?

Rule 110 was proven Turing complete by Matthew Cook in 2004. This means that with the right initial conditions, Rule 110 can simulate any computation. It is a Class IV automaton, exhibiting the mixture of order and chaos that enables universal computation. Rule 30 is Class III (chaotic but not known to be Turing complete), Rule 90 is Class II (periodic), and Rule 184 is Class II (models traffic flow).

✍ Question 5

In the Game of Life, what is a "glider"?

A glider is a spaceship — a pattern that moves across the grid. The smallest glider consists of 5 cells and moves one cell diagonally every 4 generations. The existence of gliders (and glider guns that produce them) is what makes the Game of Life Turing complete: glider streams can be used as signals for logic gates. Option (d) describes a "gun" (like the Gosper Glider Gun), not a glider itself.
