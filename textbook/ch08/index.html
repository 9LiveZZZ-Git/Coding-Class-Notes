8.1 Introduction: What Is Simulation?

Imagine you are standing on a bridge, watching leaves fall from a tree into a river below. Each leaf follows a slightly different path. Some tumble end-over-end. Others glide smoothly before catching a gust of wind. A few land in the water and are carried downstream, spinning in tiny eddies.

You could paint this scene—freezing one moment in time. Or you could build a simulation—a set of rules that, when run, produces an infinite number of such moments, each one unique, each one plausible.

📖 Definition

Simulation — A computational model that evolves over time. You define the rules that govern behavior, set initial conditions, press play, and watch the system unfold. The result is not a single image but a living, breathing system that can surprise even its creator.

A simulation requires three ingredients:

  1. Rules — the laws that govern how things behave (gravity pulls objects downward, birds avoid collisions).
  2. Time — the simulation steps forward frame by frame, applying the rules repeatedly.
  3. Initial conditions — the starting state (where are the objects? how fast are they moving?).

Change any of these three and you get a different outcome. This is what makes simulation so compelling for art: the same set of rules can produce endlessly varied results.

In this chapter, we will build two kinds of simulations:

  1. Particle systems — collections of simple objects subject to physical forces, used for effects like fire, smoke, sparks, and fountains.
  2. Agent systems — collections of autonomous objects that perceive their neighbors and make decisions, used for flocking, swarming, and artificial life.

Both systems share a common foundation in physics, so we will start there.

📜 Historical Note

William Reeves introduced particle systems in his landmark 1983 paper “Particle Systems—A Technique for Modeling a Class of Fuzzy Objects.” Working at Lucasfilm (which would become Pixar), Reeves used particles to create the “Genesis Effect” explosion sequence in Star Trek II: The Wrath of Khan (1982). His insight was that some phenomena—fire, clouds, water spray—are better modeled as collections of many tiny, short-lived elements rather than as solid surfaces. This idea has since become one of the foundational techniques in computer graphics and creative coding.

8.2 Vectors: The Language of Motion

Before we can simulate anything that moves, we need a way to describe motion. In the physical world, motion involves both a speed (how fast) and a direction (which way). A quantity that has both magnitude and direction is called a vector.

In a 2D simulation, every moving object has three key vectors:

  • Position — “the object is at (200, 150)”
  • Velocity — “the object is moving 3 pixels right and 2 pixels down per frame”
  • Acceleration — “the object’s velocity is increasing by 0.1 pixels per frame downward”

In p5.js, we create vectors with the createVector(x, y) function:

let position     = createVector(200, 150);
let velocity     = createVector(3, -2);
let acceleration = createVector(0, 0.1);

Vectors support arithmetic operations that let us combine and transform them:

// Adding vectors: move the object by its velocity
position.add(velocity);

// Scaling vectors: halve the force
let force = createVector(1, 0);
force.mult(0.5);

// Getting magnitude (length): how fast are we going?
let speed = velocity.mag();

// Normalizing (making length 1, keeping direction)
let direction = velocity.copy().normalize();

These operations form the building blocks of every simulation we will write. The relationship between position, velocity, and acceleration is the heartbeat of motion:

$$\text{velocity} \mathrel{+}= \text{acceleration}$$

$$\text{position} \mathrel{+}= \text{velocity}$$

Each frame, acceleration changes velocity, and velocity changes position. This two-step update is called Euler integration—the simplest numerical method for simulating motion. It is not the most accurate method, but it is easy to understand and works well for creative coding purposes.

Let’s see this in action. The sketch below creates a ball that drifts across the canvas, accelerating slowly to the right:

Try changing the acceleration vector. What happens if you set it to createVector(0, 0.05)? The ball will fall downward—that is gravity. What about createVector(-0.01, -0.01)? Now it accelerates up and to the left. Every force in our simulations will work through this same mechanism: changing the acceleration vector.

✍ Knowledge Check

In a simulation, what is the correct order of operations each frame?

The correct sequence is: (1) apply all forces to compute acceleration, (2) add acceleration to velocity, (3) add velocity to position, (4) reset acceleration to zero so forces can be freshly accumulated next frame. This is the standard Euler integration loop used throughout physics simulation.

8.3 Newton’s Second Law: $F = ma$

Isaac Newton’s second law of motion tells us how forces relate to motion:

$$F = m \cdot a$$

Rearranged for our purposes:

$$a = \frac{F}{m}$$

In plain language: acceleration equals force divided by mass. A heavy object (large $m$) accelerates less than a light object (small $m$) under the same force. Push a bowling ball and a tennis ball with the same force—the tennis ball shoots away while the bowling ball barely moves.

In a simulation, this means:

  1. We know the forces acting on an object (gravity, wind, springs, etc.).
  2. We divide the total force by the object’s mass to get its acceleration.
  3. We use the acceleration to update the velocity.
  4. We use the velocity to update the position.

For simplicity, we often assume mass = 1, which means $a = F$. Here is the basic pattern in code—a class called Mover that encapsulates an object with physics:

class Mover {
  constructor(x, y) {
    this.position     = createVector(x, y);
    this.velocity     = createVector(0, 0);
    this.acceleration = createVector(0, 0);
    this.mass         = 1;
  }

  applyForce(force) {
    // a = F / m
    let f = p5.Vector.div(force, this.mass);
    this.acceleration.add(f);
  }

  update() {
    this.velocity.add(this.acceleration);
    this.position.add(this.velocity);
    this.acceleration.mult(0); // Reset each frame
  }

  display() {
    circle(this.position.x, this.position.y, this.mass * 16);
  }
}
📖 Definition

Newton’s Second Law — The net force acting on an object equals its mass times its acceleration: $F = ma$. In simulation code, we rearrange this to $a = F/m$ so we can compute the resulting acceleration from any applied force.

Let’s walk through the two key methods:

applyForce(force):

  • let f = p5.Vector.div(force, this.mass); — We divide the force by mass to get this force’s contribution to acceleration. We use the static p5.Vector.div() to create a new vector rather than modifying the original force. This is important because the same force vector might be applied to multiple objects.
  • this.acceleration.add(f); — We add the contribution to the accumulated acceleration. Multiple forces can be applied per frame, and they all add up (this is called superposition).

update():

  • this.velocity.add(this.acceleration); — Acceleration changes velocity.
  • this.position.add(this.velocity); — Velocity changes position.
  • this.acceleration.mult(0); — We must reset acceleration to zero. Forces in our model are re-applied every frame. If we did not reset, the acceleration from previous frames would persist and grow without bound.
⚠ Common Mistake

Forgetting to reset acceleration at the end of update() is the most common physics simulation bug. Without the line this.acceleration.mult(0), forces accumulate across frames, causing objects to accelerate wildly out of control. Every frame, you must start with a clean slate and re-apply all active forces.

8.4 Gravity

Gravity is the simplest force to simulate: a constant downward pull.

let gravity = createVector(0, 0.1);
mover.applyForce(gravity);

This creates a force vector pointing straight down (positive y is down in p5.js) with a magnitude of 0.1 pixels per frame squared. Every frame, the object’s downward velocity increases by 0.1—it falls faster and faster, just like a real falling object.

In the real world, gravitational force depends on mass: $F_g = mg$, where $g$ is the gravitational acceleration constant (about $9.8 \, \text{m/s}^2$ on Earth). If we want heavier objects to fall at the same rate as lighter ones (as they do in reality, ignoring air resistance), we should scale the gravitational force by mass:

let gravity = createVector(0, 0.1 * mover.mass);
mover.applyForce(gravity);
// Inside applyForce, we divide by mass, so the mass cancels out

This ensures that inside applyForce, the division by mass cancels the multiplication, and all objects accelerate at the same rate regardless of their mass. This is the principle Galileo demonstrated centuries ago by (allegedly) dropping balls of different weights from the Leaning Tower of Pisa.

Let’s see gravity in action with a bouncing ball:

p5.js Bouncing Ball with Gravity

Notice how the ball loses a little energy on each bounce (this.velocity.y *= -0.8). The -1 reverses the direction, and the 0.8 reduces the speed by 20%. This simulates an inelastic collision—some kinetic energy is lost to heat and deformation, just like a real ball bouncing on a floor. Try changing -0.8 to -1.0 for a perfectly elastic bounce, or -0.5 for a very “dead” ball.

✍ Knowledge Check

Why do we multiply the gravitational force by an object’s mass before applying it?

Gravitational force is $F = mg$. When applyForce computes $a = F/m$, the mass cancels: $a = mg/m = g$. This means all objects experience the same gravitational acceleration regardless of mass—matching real-world physics (when air resistance is negligible).

8.5 Building a Particle System

Now that we understand the physics of a single moving object, we can scale up to many objects. A particle system manages a collection of short-lived physics objects called particles. Each particle is born, lives briefly, and dies. This cycle of birth and death is what gives particle systems their dynamic, ever-changing quality.

8.5.1 What Is a Particle?

A particle is a simplified physics object with a limited lifespan. Unlike permanent objects in a simulation, particles are ephemeral—they appear, do their thing, and vanish. This is perfect for modeling phenomena like fire (each spark lasts only a moment), smoke (each puff dissipates), and water spray (each droplet splashes and disappears).

A particle needs:

  • Position, velocity, acceleration — for physics (same as our Mover).
  • Lifespan — a timer that counts down from some maximum to zero.
  • Visual properties — color, size, transparency (often tied to lifespan so the particle fades as it dies).

8.5.2 The Particle Class

class Particle {
  constructor(x, y) {
    this.position     = createVector(x, y);
    this.velocity     = createVector(random(-1, 1), random(-3, 0));
    this.acceleration = createVector(0, 0);
    this.lifespan     = 255.0;
    this.size         = random(4, 12);
    this.col          = color(
      random(200, 255), random(80, 180), random(0, 60)
    );
  }

  applyForce(force) {
    this.acceleration.add(force);
  }

  update() {
    this.velocity.add(this.acceleration);
    this.position.add(this.velocity);
    this.acceleration.mult(0);
    this.lifespan -= 2.0;
  }

  display() {
    noStroke();
    let alpha = this.lifespan;
    let currentSize = map(this.lifespan, 255, 0, this.size, 0);
    fill(red(this.col), green(this.col), blue(this.col), alpha);
    circle(this.position.x, this.position.y, currentSize);
  }

  isDead() {
    return this.lifespan <= 0;
  }
}

Let’s walk through the important details:

  • this.velocity = createVector(random(-1, 1), random(-3, 0)); — Each particle starts with a random velocity: slight horizontal spread and upward motion. The randomness ensures particles don’t all go in exactly the same direction, creating a natural-looking spray.
  • this.lifespan = 255.0; — We use 255 because it conveniently maps to p5.js’s alpha channel (0–255). When lifespan is 255, the particle is fully opaque. When it reaches 0, the particle is fully transparent and ready to be removed.
  • this.lifespan -= 2.0; — Each frame, the particle ages. At 2.0 per frame and 60 fps, each particle lives about 2 seconds (255 / 2 / 60 ≈ 2.1 seconds).
  • map(this.lifespan, 255, 0, this.size, 0) — The particle shrinks as it ages. When lifespan is 255, size is at its maximum. When lifespan is 0, size is 0. Combined with the fading alpha, this creates a natural dissipation effect.
  • isDead() — Returns true when the particle’s lifespan has expired. The system uses this to know when to remove the particle.

8.5.3 The Emitter Class

An emitter is the manager of a particle system. It has a position (the “source” from which particles appear) and an array of particles. Its job is to create new particles, apply forces to all of them, update their physics, remove dead ones, and draw the survivors.

class Emitter {
  constructor(x, y) {
    this.origin    = createVector(x, y);
    this.particles = [];
  }

  emit(count) {
    for (let i = 0; i < count; i++) {
      this.particles.push(
        new Particle(this.origin.x, this.origin.y)
      );
    }
  }

  applyForce(force) {
    for (let p of this.particles) {
      p.applyForce(force);
    }
  }

  update() {
    // Iterate BACKWARD so we can safely remove elements
    for (let i = this.particles.length - 1; i >= 0; i--) {
      this.particles[i].update();
      if (this.particles[i].isDead()) {
        this.particles.splice(i, 1);
      }
    }
  }

  display() {
    for (let p of this.particles) {
      p.display();
    }
  }
}
⚠ Common Mistake

Why iterate backward? When we call this.particles.splice(i, 1) to remove a dead particle, all elements after index i shift down by one. If we iterated forward, the next element would be skipped because it moved into the slot we just processed. Iterating backward avoids this: removing element i only affects indices greater than i, which we have already processed.

💡 Key Concept

The power of particle systems lies in quantity and randomness. No single particle is interesting on its own—it is just a fading dot. But hundreds of them, each with slightly different initial conditions, create the visual impression of fire, smoke, sparks, or rain. The individual is simple; the collective is rich.

8.6 From Particles to Agents

So far, our particles have been entirely passive. They have no awareness of their surroundings. They simply respond to external forces—gravity pulls them down, wind pushes them sideways. A particle is a leaf in the wind: it goes wherever physics takes it.

An agent is fundamentally different. An agent is active: it perceives its local environment and makes decisions about how to move. It is a bird in a flock, not a leaf in the wind.

Here is the key insight: both particles and agents use the same underlying physics. They both have position, velocity, and acceleration. They both obey the same update loop (accumulate forces, update velocity, update position, reset acceleration). The difference lies in where the forces come from:

  • Particle system — Forces are externally applied: gravity, wind, drag. The particle has no say in the matter.
  • Agent system — Forces are self-generated from perception. The agent looks at its neighbors, decides where it wants to go, and computes a force to steer itself in that direction.

Think of it this way: a leaf and a bird are both subject to gravity and air resistance. But the bird also has muscles and a brain. It can choose to fly toward a neighbor, away from a predator, or toward a food source. These choices become forces that it applies to itself.

This shift—from passive to active, from externally driven to self-directed—is what turns a particle system into an agent system. And it is what makes agent systems produce the stunning emergent behaviors that we see in nature: flocking, schooling, swarming, and herding.

8.7 Autonomous Agents

Craig Reynolds, the computer scientist who created the famous “Boids” algorithm, described four defining characteristics of an autonomous agent:

  1. It exists in an environment — it has a position in space and can move through it.
  2. It has limited perception — it can detect nearby neighbors and environmental features, but it does not have a god’s-eye view of the entire world. A bird in a flock can see the birds around it, but not the entire flock.
  3. It has no leader — there is no central controller telling each agent what to do. Each agent acts independently based on its own local perception.
  4. It has desires (goals) — encoded as behavioral rules like “stay near the group,” “avoid collisions,” or “move in the same direction as my neighbors.”

The combination of autonomy and local perception is what makes agent systems produce emergent behavior. No one designs the flocking pattern. No one tells the flock to split around an obstacle and rejoin on the other side. These complex global behaviors emerge from simple local rules.

📜 Historical Note

Craig Reynolds presented his Boids algorithm at SIGGRAPH 1987 in the paper “Flocks, Herds, and Schools: A Distributed Behavioral Model.” Reynolds showed that realistic flocking behavior could be produced by three simple rules applied to each individual “boid” (a blend of “bird” and “droid”). The paper was revolutionary because it demonstrated that complex collective behavior does not require complex individual intelligence—it emerges from the interaction of simple agents following simple rules. The Boids algorithm has since been used in countless films, games, and art installations.

✍ Knowledge Check

Which of the following is NOT a characteristic of an autonomous agent according to Reynolds?

A defining feature of autonomous agents is that they have no leader. There is no central controller issuing commands. Each agent acts independently based on its own local perception and built-in behavioral rules. The absence of centralized control is precisely what makes emergent behavior possible and surprising.

8.8 Reynolds’ Steering Model

How does an agent translate its “desires” into motion? Reynolds proposed an elegantly simple formula:

$$\vec{f}_{\text{steering}} = \vec{v}_{\text{desired}} - \vec{v}_{\text{current}}$$

In words: the steering force is the difference between where you want to go and where you are going.

Think of it like driving a car. Your desired velocity is the direction you want to travel (say, toward a gas station). Your current velocity is the direction you are actually traveling (straight down the highway). The steering force is how much you need to turn the wheel to get from your current heading to your desired heading. If you’re already heading toward the gas station, the steering force is zero—no correction needed.

In code, this looks like:

// General pattern for computing a steering force
let desired = /* some target direction at maxSpeed */;
let steering = p5.Vector.sub(desired, this.velocity);
steering.limit(this.maxForce);

Two important constraints keep the agent behaving realistically:

  • maxSpeed — The maximum speed an agent can travel. The desired velocity is always set to this magnitude. Without this limit, agents would accelerate to infinite speeds.
  • maxForce — The maximum steering force. This limits how quickly an agent can turn. Lower values produce smoother, more graceful turns (like a large bird or a car at speed). Higher values produce sharper, more responsive turns (like a housefly or a sports car).

The beauty of this model is that every agent behavior—seek, flee, separate, align, cohere—can be expressed as a different way of computing the desired velocity. The steering formula stays the same. Only the goal changes.

💡 Key Concept

Steering = Desired − Current. This single formula is the engine behind all Reynolds-style agent behaviors. To make an agent seek a target, compute a desired velocity toward the target. To make it flee, compute a desired velocity away from a threat. To make it align with neighbors, compute a desired velocity matching their average heading. The steering formula converts any goal into a force.

8.9 Implementing Boids

Reynolds’ Boids algorithm produces realistic flocking from just three rules, each applied locally to every boid:

  1. Separation — steer away from nearby neighbors to avoid crowding.
  2. Alignment — steer toward the average heading of nearby neighbors.
  3. Cohesion — steer toward the average position of nearby neighbors.

Let’s implement each rule step by step.

8.9.1 The Boid Class Foundation

A boid is an agent with position, velocity, acceleration, and two constraint values:

class Boid {
  constructor(x, y) {
    this.position     = createVector(x, y);
    this.velocity     = p5.Vector.random2D();
    this.velocity.setMag(random(2, 4));
    this.acceleration = createVector(0, 0);
    this.maxForce     = 0.2;
    this.maxSpeed     = 4;
  }

  applyForce(force) {
    this.acceleration.add(force);
  }

  update() {
    this.velocity.add(this.acceleration);
    this.velocity.limit(this.maxSpeed);
    this.position.add(this.velocity);
    this.acceleration.mult(0);
  }
}
  • p5.Vector.random2D() creates a unit vector pointing in a random direction, so each boid starts flying a random way.
  • setMag(random(2, 4)) sets the initial speed to a random value between 2 and 4 pixels per frame.
  • this.maxForce = 0.2 limits how sharply the boid can turn.
  • this.maxSpeed = 4 limits how fast it can go.

8.9.2 Rule 1: Separation

Separation prevents boids from crowding each other. Each boid looks at all neighbors within a perception radius and computes a force pointing away from them. The closer a neighbor is, the stronger the repulsion.

separation(boids) {
  let perceptionRadius = 50;
  let steering = createVector(0, 0);
  let total = 0;

  for (let other of boids) {
    let d = dist(
      this.position.x, this.position.y,
      other.position.x, other.position.y
    );

    if (other !== this && d < perceptionRadius) {
      let diff = p5.Vector.sub(this.position, other.position);
      diff.div(d * d);  // Closer = stronger push
      steering.add(diff);
      total++;
    }
  }

  if (total > 0) {
    steering.div(total);            // Average
    steering.setMag(this.maxSpeed); // Desired velocity
    steering.sub(this.velocity);    // Steering = desired - current
    steering.limit(this.maxForce);  // Limit the force
  }

  return steering;
}

The key line is diff.div(d * d). By dividing by the square of the distance, we make the repulsion much stronger for very close boids. A boid 5 pixels away pushes 100 times harder than a boid 50 pixels away (because $50^2 / 5^2 = 100$). This inverse-square relationship is common in physics—gravity and electrostatic forces follow the same pattern.

8.9.3 Rule 2: Alignment

Alignment makes boids fly in the same direction as their neighbors. We average the velocity vectors of all nearby boids and steer toward that average heading:

alignment(boids) {
  let perceptionRadius = 50;
  let steering = createVector(0, 0);
  let total = 0;

  for (let other of boids) {
    let d = dist(
      this.position.x, this.position.y,
      other.position.x, other.position.y
    );

    if (other !== this && d < perceptionRadius) {
      steering.add(other.velocity);
      total++;
    }
  }

  if (total > 0) {
    steering.div(total);            // Average velocity of neighbors
    steering.setMag(this.maxSpeed); // Desired velocity in that direction
    steering.sub(this.velocity);    // Steering = desired - current
    steering.limit(this.maxForce);
  }

  return steering;
}

This is simpler than separation—no distance weighting needed. We just average the velocities and apply the steering formula.

8.9.4 Rule 3: Cohesion

Cohesion steers boids toward the center of their local group. We average the positions of all nearby boids to find the group center, then steer toward it:

cohesion(boids) {
  let perceptionRadius = 50;
  let steering = createVector(0, 0);
  let total = 0;

  for (let other of boids) {
    let d = dist(
      this.position.x, this.position.y,
      other.position.x, other.position.y
    );

    if (other !== this && d < perceptionRadius) {
      steering.add(other.position);
      total++;
    }
  }

  if (total > 0) {
    steering.div(total);            // Average position of neighbors
    steering.sub(this.position);    // Vector from me to group center
    steering.setMag(this.maxSpeed); // Desired velocity toward center
    steering.sub(this.velocity);    // Steering = desired - current
    steering.limit(this.maxForce);
  }

  return steering;
}

Notice the extra step: steering.sub(this.position). After computing the average position (the group center), we need a vector pointing from our position toward that center. This becomes the desired direction of travel.

8.9.5 Combining the Three Rules

Each rule produces a steering force. We combine them with weights that control the relative importance of each behavior:

flock(boids) {
  let sep = this.separation(boids);
  let ali = this.alignment(boids);
  let coh = this.cohesion(boids);

  // Weight the forces
  sep.mult(1.5);  // Separation slightly stronger
  ali.mult(1.0);
  coh.mult(1.0);

  // Apply all forces
  this.applyForce(sep);
  this.applyForce(ali);
  this.applyForce(coh);
}

Changing these weights dramatically changes the flock’s behavior:

  • High separation, low cohesion — Boids spread out and avoid each other, like pigeons on a sidewalk.
  • Low separation, high cohesion — Boids collapse into a tight, dense cluster.
  • High alignment — Boids move in long parallel streams, like a school of fish in a current.
  • Balanced weights — The classic flocking behavior: dynamic, organic, constantly shifting.

8.11 Comparing Particle Systems and Agent Systems

Now that we have built both a particle system and an agent system, let’s compare them side by side:

Aspect Particle System Agent System (Boids)
Update loop Apply forces, integrate, die Perceive neighbors, compute steering, integrate
Awareness None—particles are blind Local (perception radius)
Lifespan Finite (particles die) Infinite (agents persist)
Interaction Independent (no particle-to-particle awareness) Mutual influence (each agent responds to neighbors)
Emergence Visual (fire, smoke from many dots) Behavioral (flocking from three rules)
Typical count Hundreds to thousands Tens to hundreds
Interesting because Collective visual impression Collective behavioral patterns
Source of forces External (gravity, wind, drag) Self-generated from perception

Both systems share the same core physics loop: accumulate forces, update velocity, update position, reset acceleration. The fundamental difference lies in where the forces come from: externally applied (particle system) vs. self-generated from perception (agent system).

✍ Knowledge Check

What is the fundamental difference between a particle and an agent?

Both particles and agents use the same physics (vectors, forces, Euler integration). The key difference is that particles are passive—they respond only to external forces like gravity—while agents are active: they perceive their neighbors and generate their own steering forces. This self-direction is what allows agent systems to produce emergent behaviors like flocking.

8.12 Beyond Boids: Other Agent Behaviors

8.12.1 Braitenberg Vehicles

Before Reynolds and Boids, the Italian-Austrian neuroscientist Valentino Braitenberg proposed a thought experiment in his 1984 book Vehicles: Experiments in Synthetic Psychology. Braitenberg imagined the simplest possible agents: tiny vehicles with sensors and motors, connected by wires. Despite having no brain, no memory, and no explicit “intelligence,” these vehicles exhibit behaviors that look intelligent to an observer.

A Braitenberg vehicle has:

  • Two sensors (one on the left, one on the right) that detect a stimulus (like light).
  • Two motors (one on the left, one on the right) that drive the wheels.
  • Wiring that connects sensors to motors.

The wiring determines the behavior:

  • Direct wiring (left sensor → left motor, right sensor → right motor): The vehicle turns away from the stimulus. Braitenberg called this “fear”—the vehicle avoids light.
  • Crossed wiring (left sensor → right motor, right sensor → left motor): The vehicle turns toward the stimulus. Braitenberg called this “aggression”—the vehicle charges at the light.

Move your mouse around and observe the difference in behavior. The green vehicles home in on the light like heat-seeking missiles. The blue vehicles veer away. The remarkable thing is that both behaviors arise from the same mechanism—sensors connected to motors—with only the wiring pattern changed. Braitenberg argued that we instinctively anthropomorphize these simple machines, attributing “fear” and “aggression” to them even though they have no internal experience whatsoever.

8.12.2 Artificial Life

Agent systems are the foundation of artificial life (ALife) research. By giving agents rules for eating, reproducing, and dying, you can create simulated ecosystems that exhibit life-like properties: adaptation, self-organization, reproduction, and evolution.

Some notable examples from the history of artificial life:

  • Karl Sims’ Evolved Virtual Creatures (1994) — Sims evolved 3D creatures in simulation, starting from random body plans and neural networks, and selecting for creatures that could swim, walk, or compete for resources. The results were strange and surprising—creatures developed body forms and locomotion strategies that no human designer would have imagined.
  • Lenia (2019, Bert Wang-Chak Chan) — A continuous generalization of Conway’s Game of Life. Lenia produces stunning life-like patterns: creatures that glide, pulsate, divide, and interact. Unlike the sharp on/off cells of the Game of Life, Lenia uses smooth, continuous values, resulting in organic, almost biological forms.
  • Clusters (Jeffrey Ventrella) — A system where agents attract and repel based on “types,” forming self-assembling structures that resemble molecules, organisms, and social groups.
📜 Historical Note

The field of artificial life was formally launched at a workshop organized by Christopher Langton at Los Alamos National Laboratory in 1987—the same year Reynolds published his Boids paper. Langton coined the term “artificial life” and defined it as the study of “life as it could be” in addition to “life as we know it.” The field sits at the intersection of computer science, biology, and philosophy, asking fundamental questions about what it means to be alive.

8.13 Emergence

If there is one idea that unites everything in this chapter, it is emergence.

📖 Definition

Emergence — The phenomenon where complex global patterns or behaviors arise from simple local rules and interactions. The emergent behavior is not explicitly programmed—it is a consequence of the system’s structure and dynamics.

Consider what we have built in this chapter:

  • A particle system with simple rules (gravity, fading lifespan) produces the visual impression of fire or a fountain—nobody programmed “fire.”
  • A boids simulation with three local rules (separation, alignment, cohesion) produces flocking—nobody programmed “flock.”
  • Braitenberg vehicles with sensor-motor wiring produce behaviors that look like fear and aggression—nobody programmed emotions.

In each case, the interesting behavior exists at the collective level, not the individual level. A single particle is just a fading dot. A single boid just wanders. But put them together with simple rules, and something greater appears—something that was not designed, not anticipated, not explicitly specified in the code.

This is the philosophical heart of simulation-based art. You are not painting an image. You are defining rules and watching something emerge. The artist becomes a designer of systems, not a designer of outcomes. The results are always partially surprising, partially beyond your control. This tension between authorship and emergence is what makes simulation one of the most compelling creative tools in computing arts.

We have seen emergence before in this course: cellular automata in Chapter 6, reaction-diffusion in Chapter 7. Particle systems and agent systems extend the idea into the domain of continuous motion, forces, and behavior. The thread connecting all of these is the same: simple rules, local interactions, and the unexpected beauty that arises from their combination.

💡 Key Concept

No designer necessary. The flock has no choreographer. The fire has no painter. The Braitenberg vehicle has no psychologist. Emergence means that complex, interesting, beautiful behavior can arise from the interaction of simple parts following simple rules. This is perhaps the deepest lesson of simulation: you do not need to specify the outcome. You only need to specify the process.

Chapter 8 Review Quiz

Test your understanding of particle systems, agent systems, and simulation.

✍ Question 1

In the Euler integration loop, why must we reset acceleration to zero at the end of each frame?

Forces in our simulation model are applied fresh every frame. If we do not reset acceleration to zero, the forces from previous frames persist and add to new forces, causing acceleration to grow without bound. The reset ensures a clean slate: only the forces that are actively applied this frame contribute to the object’s acceleration.

✍ Question 2

In a particle system, what determines when a particle should be removed?

Each particle has a lifespan that counts down each frame. When it reaches zero, the isDead() method returns true, and the emitter’s update() method removes it from the array using splice(). This birth-death cycle is what gives particle systems their dynamic, ever-changing quality.

✍ Question 3

What are the three rules of Reynolds’ Boids algorithm?

Reynolds’ three rules are: (1) Separation—steer away from nearby neighbors to avoid crowding, (2) Alignment—steer toward the average heading of neighbors, and (3) Cohesion—steer toward the average position of neighbors. These three local rules, applied to each individual boid, produce realistic flocking behavior at the collective level.

✍ Question 4

In Reynolds’ steering model, how is the steering force computed?

Reynolds’ steering formula is: $\vec{f}_{\text{steering}} = \vec{v}_{\text{desired}} - \vec{v}_{\text{current}}$. The steering force is the difference between where you want to go (desired velocity) and where you are currently going (current velocity). This force corrects the agent’s trajectory toward its goal.

✍ Question 5

In a Braitenberg vehicle with crossed wiring, the left sensor drives the right motor and vice versa. What behavior does this produce?

With crossed wiring, a stronger signal on the left sensor speeds up the right motor, which turns the vehicle to the left—toward the stimulus. This produces approach behavior that Braitenberg called “aggression.” Direct wiring (left sensor → left motor) produces the opposite: the vehicle turns away (“fear”).
