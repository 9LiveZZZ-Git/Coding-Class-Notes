## Introduction

How do you measure the "complexity" of an image? Is a red circle "simpler" than a screen full of random colored pixels? Intuitively, yes -- but can we make this precise?

This lesson explores the deep connections between:

- **Kolmogorov complexity** -- the theoretical minimum description of something
- **Data compression** -- practical algorithms that exploit patterns
- **Generative art** -- creating images from short programs
- **Low-complexity art** -- Jurgen Schmidhuber's thesis that beauty relates to compressibility

---

## Part 1: Kolmogorov Complexity

### The Core Idea

The **Kolmogorov complexity** of an object (a string, an image, a piece of music) is the length of the **shortest program** that produces that object.

Consider two strings:

- String A: `aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa` (32 a's)
- String B: `kx7qm2p9wvf4zt1a8jn3yl6bc0hurdes` (32 random characters)

String A can be generated by a very short program:

```js
// Program for String A (very short)
for (let i = 0; i < 32; i++) print("a");
```

But String B has no pattern. The shortest program that generates it is essentially "print this exact string" -- which is about as long as the string itself.

**Kolmogorov complexity** formalizes this: String A has low complexity (it can be described briefly), while String B has high complexity (it cannot be compressed).

### Why It Matters for Art

This gives us a way to think about the "information content" of visual art:

- A **red circle on a white background** has very low Kolmogorov complexity. You can describe it in a few lines of code.
- A **canvas of random colored noise** has very high Kolmogorov complexity. There is no short description; you need to specify every pixel.

### Important Theoretical Note

Kolmogorov complexity is **uncomputable** -- you cannot write a program that calculates the exact Kolmogorov complexity of an arbitrary input. This is related to the halting problem. But we can approximate it using practical compression algorithms.

## Part 2: The Red Circle vs. Color Noise -- A Concrete Example

Let us compare two images and think about their representations.

### Image 1: A Red Circle

```js
function setup() {
  createCanvas(400, 400);
  noLoop();
}

function draw() {
  background(255);     // White background
  fill(255, 0, 0);     // Red fill
  noStroke();
  ellipse(200, 200, 300, 300); // Centered circle
}
```

This program is about **180 characters** of source code. The image it produces, saved as a raw bitmap (400 x 400 pixels x 3 bytes per pixel), would be **480,000 bytes**. Saved as a PNG (which compresses), it might be around **5,000 bytes**.

### Image 2: Random Color Noise

```js
function setup() {
  createCanvas(400, 400);
  noLoop();
}

function draw() {
  loadPixels();
  for (let i = 0; i < pixels.length; i++) {
    pixels[i] = floor(random(256));
  }
  updatePixels();
}
```

This program is about **180 characters** too. But the image it produces is **incompressible**:

- Raw bitmap: **480,000 bytes**
- PNG: approximately **470,000 bytes** (PNG cannot compress random data much)
- Source code: **~180 bytes** -- but wait, this does not produce the same image every time!

Here is the crucial distinction: the noise program generates **a different random image** each time. To reproduce **one specific** noise image, you would need to store every pixel value. The program is short, but it does not produce a **specific** image without a specific random seed.

If you add `randomSeed(42)`, now the program is deterministic and always produces the same image. The source code is about 200 bytes, but the image "looks like" it contains 480,000 bytes of information. The secret is that the pseudorandom number generator (PRNG) is doing the heavy lifting -- it **expands** a short seed into a long sequence.

### Size Comparison Table

| Representation | Red Circle | Random Noise (one specific instance) |
|---------------|-----------|--------------------------------------|
| Raw bitmap (400x400, 24-bit) | 480,000 bytes | 480,000 bytes |
| PNG | ~5,000 bytes | ~470,000 bytes |
| p5.js source code | ~180 bytes | ~480,000 bytes (must list every pixel) |
| p5.js source + seed | ~180 bytes | ~200 bytes (but requires PRNG) |

The red circle compresses dramatically in PNG because PNG exploits repeated patterns (large regions of the same color). The noise image does not compress because there are no patterns to exploit.

## Part 3: Data Compression Algorithms

### Run-Length Encoding (RLE)

**Run-Length Encoding** is the simplest compression algorithm. It replaces runs of repeated values with a count and the value.

Example:
```
Input:  AAAAAABBBCCDDDDDDDD
Output: 6A3B2C8D
```

The input is 19 characters; the output is 8 characters. Compression ratio: 8/19 = 42%.

### RLE in Code

```js
function rleEncode(input) {
  let result = "";
  let i = 0;

  while (i < input.length) {
    let char = input[i];
    let count = 1;

    // Count consecutive identical characters
    while (i + count < input.length && input[i + count] === char) {
      count++;
    }

    result += count + char;
    i += count;
  }

  return result;
}

function setup() {
  createCanvas(600, 400);
  noLoop();
}

function draw() {
  background(255);
  fill(0);
  textSize(14);
  textFont('monospace');

  let examples = [
    "AAAAAABBBCCDDDDDDDD",
    "ABCABCABC",
    "AAAAAAAAAAAAAAAA",
    "ABABABABABABABAB"
  ];

  text("Run-Length Encoding Examples:", 20, 30);

  for (let i = 0; i < examples.length; i++) {
    let input = examples[i];
    let encoded = rleEncode(input);
    let ratio = (encoded.length / input.length * 100).toFixed(1);

    let y = 70 + i * 70;
    text("Input:    " + input, 20, y);
    text("Encoded:  " + encoded, 20, y + 20);
    text("Ratio:    " + encoded.length + "/" + input.length +
         " = " + ratio + "%", 20, y + 40);
  }
}
```

Notice that RLE works well for data with long runs (like `AAAAAAAAAA`) but can actually make data **larger** if there are no runs (like `ABABABAB` becomes `1A1B1A1B1A1B1A1B`).

### Where RLE Works Well

RLE is effective for:
- Simple images with large areas of solid color (like our red circle)
- Fax machine transmissions (mostly black and white)
- Early video game graphics

It is not effective for:
- Photographs (colors change at every pixel)
- Random data (no repeated runs)

### Huffman Coding

**Huffman coding** is a more sophisticated compression algorithm. It assigns shorter binary codes to more frequent symbols and longer codes to less frequent ones.

#### The Idea

In standard ASCII, every character uses 8 bits. But if the letter 'e' appears 100 times and the letter 'z' appears once, it is wasteful to use 8 bits for both. Huffman coding might assign:
- 'e' = 2 bits (because it is common)
- 'z' = 12 bits (because it is rare)

Overall, this saves space because the common characters use fewer bits.

#### Building a Huffman Tree

1. Count the frequency of each symbol.
2. Create a leaf node for each symbol.
3. Repeatedly merge the two nodes with the lowest frequencies into a new parent node.
4. The resulting tree determines the codes: left branch = 0, right branch = 1.

#### Example

For the string "AABBBCCCCDDDDD":

| Symbol | Frequency |
|--------|-----------|
| A      | 2         |
| B      | 3         |
| C      | 4         |
| D      | 5         |

Huffman tree construction:
1. Merge A(2) and B(3) into AB(5)
2. Merge C(4) and AB(5) into CAB(9)
3. Merge D(5) and CAB(9) into root(14)

Resulting codes (one possible assignment):
- D: 0 (1 bit)
- C: 10 (2 bits)
- A: 110 (3 bits)
- B: 111 (3 bits)

Original: 14 characters x 8 bits = 112 bits
Encoded: (2x3) + (3x3) + (4x2) + (5x1) = 6 + 9 + 8 + 5 = 28 bits

That is a 75% reduction.

#### Visualizing Huffman Coding

```js
function setup() {
  createCanvas(600, 500);
  noLoop();
}

function draw() {
  background(255);

  // The string to encode
  let str = "AABBBCCCCDDDDD";

  // Count frequencies
  let freq = {};
  for (let ch of str) {
    freq[ch] = (freq[ch] || 0) + 1;
  }

  // Display frequency analysis
  fill(0);
  textSize(16);
  textFont('monospace');
  text('String: "' + str + '"', 20, 30);
  text("Length: " + str.length + " chars = " + (str.length * 8) + " bits (ASCII)", 20, 55);

  text("Symbol Frequencies:", 20, 90);
  let symbols = Object.keys(freq).sort((a, b) => freq[a] - freq[b]);
  for (let i = 0; i < symbols.length; i++) {
    let ch = symbols[i];
    let barWidth = freq[ch] * 30;

    // Draw bar
    fill(100, 150, 255);
    noStroke();
    rect(120, 105 + i * 30, barWidth, 20);

    // Label
    fill(0);
    textSize(14);
    text(ch + ": " + freq[ch], 20, 120 + i * 30);
  }

  // Show Huffman codes (pre-computed for this example)
  let codes = { 'D': '0', 'C': '10', 'A': '110', 'B': '111' };

  let y = 250;
  text("Huffman Codes:", 20, y); y += 25;

  let totalBits = 0;
  for (let ch of symbols) {
    let code = codes[ch];
    let bits = code.length * freq[ch];
    totalBits += bits;
    text(ch + " -> " + code + "  (" + code.length + " bits x " +
         freq[ch] + " = " + bits + " bits)", 20, y);
    y += 25;
  }

  y += 10;
  text("Total encoded: " + totalBits + " bits", 20, y); y += 25;
  text("Original (ASCII): " + (str.length * 8) + " bits", 20, y); y += 25;
  text("Compression ratio: " + (totalBits / (str.length * 8) * 100).toFixed(1) + "%", 20, y);

  // Visualize the encoded string
  y += 35;
  text("Encoded bitstream:", 20, y); y += 20;
  let bitstream = "";
  for (let ch of str) {
    bitstream += codes[ch];
  }
  textSize(12);
  text(bitstream, 20, y);
  text("(" + bitstream.length + " bits)", 20, y + 20);
}
```

## Part 4: PNG Compression -- How It Works

PNG (Portable Network Graphics) uses a combination of:

1. **Filtering:** Each row of pixels is compared to the previous row to find patterns.
2. **DEFLATE compression:** A combination of LZ77 (finding repeated sequences) and Huffman coding.

This is why PNG works well for images with large uniform areas:

- **Red circle:** Most pixels are either pure white (255, 255, 255) or pure red (255, 0, 0). Huge runs of identical values. PNG compresses this dramatically.
- **Random noise:** Every pixel is different. No patterns to exploit. PNG adds its own overhead (headers, checksums), so the file might even be slightly larger than raw pixel data.

### Why Source Code Can Be Smaller Than PNG

The red circle's p5.js source code is ~180 bytes. Its PNG is ~5,000 bytes. The source code is a more compact representation because:

1. It describes the **process** that generates the image, not the image itself.
2. The p5.js runtime and the computer's graphics system do the work of turning that process into pixels.
3. The description "draw a red circle at the center" is vastly shorter than the 160,000 pixels that result.

This is the essence of Kolmogorov complexity: the shortest **program** that produces an output is often much shorter than the output itself.

## Part 5: Low-Complexity Art and Schmidhuber's Thesis

### Jurgen Schmidhuber's Theory of Beauty

Computer scientist Jurgen Schmidhuber proposed that humans find beauty in patterns that are **surprising but compressible**. We are drawn to things that seem complex at first glance but reveal an underlying simplicity -- an elegant rule or structure.

His theory suggests:

1. **Too simple** (a blank canvas): boring, no surprise.
2. **Too complex** (random noise): overwhelming, no pattern to discover.
3. **Compressible complexity** (a fractal, a Schotter, a fugue): interesting because the observer's brain can discover the hidden regularity, and this discovery process itself is pleasurable.

### Low-Complexity Art

"Low-complexity art" refers to visual art that can be described by very short programs. The art is "in the algorithm" rather than in the pixels.

Examples:
- Georg Nees' *Schotter*: A short program (grid + increasing randomness)
- Piet Mondrian's compositions: Rectangles, primary colors, simple rules
- Sol LeWitt's wall drawings: Described entirely by text instructions
- Fractal art: Infinite detail from finite rules

### A Low-Complexity Art Generator

```js
function setup() {
  createCanvas(400, 400);
  noLoop();
}

function draw() {
  background(255);
  noFill();
  stroke(0);
  strokeWeight(1);

  // This simple program generates a visually complex pattern
  for (let i = 0; i < 200; i++) {
    let t = i * 0.1;
    let x = width / 2 + sin(t * 1.3) * (150 - i * 0.5);
    let y = height / 2 + cos(t * 0.7) * (150 - i * 0.5);
    let size = 50 - i * 0.2;
    ellipse(x, y, size, size);
  }
}
```

This program is about 300 bytes of source code, but produces a visually rich, intricate pattern. The ratio of visual complexity to program length is high -- this is low-complexity art.

### Another Example: Moire Patterns

```js
function setup() {
  createCanvas(500, 500);
  noLoop();
}

function draw() {
  background(255);
  noFill();
  stroke(0, 50);
  strokeWeight(0.5);

  // Two sets of concentric circles, slightly offset
  // The interference creates complex moire patterns
  for (let r = 5; r < 500; r += 5) {
    ellipse(240, 250, r * 2, r * 2);
    ellipse(260, 250, r * 2, r * 2);
  }
}
```

Just two sets of concentric circles. The source code is trivially short, but the visual result -- a moire interference pattern -- appears startlingly complex. The "complexity" is an artifact of perception, not of information content.

## Part 6: The Relationship Between Program Length and Image Complexity

We can think of images on a spectrum:

```
Simple                                                        Complex
(low Kolmogorov complexity)                  (high Kolmogorov complexity)
    |                                                             |
    |  blank       geometric     Schotter     photograph    random
    |  canvas      shapes                                   noise
    |                                                             |
    Short program                                      Long program
    (or no program needed)                    (must specify every pixel)
```

### Key Insight for Generative Art

Generative art lives in the sweet spot: programs that are relatively short but produce images that appear complex. The artist's job is to find **compact rules** that generate **rich visual output**.

This is why generative art is deeply connected to computation: you are searching for short programs with interesting outputs. This is, in a sense, the same thing a compression algorithm does -- finding short descriptions of complex data.

## Part 7: Interactive Compression Demo

```js
function setup() {
  createCanvas(600, 400);
}

function draw() {
  background(240);

  // Left half: ordered pattern (compressible)
  push();
  let cols = 12;
  let rows = 16;
  let cellSize = 20;

  fill(0);
  noStroke();
  textSize(14);
  text("Ordered (compressible)", 20, 20);

  for (let r = 0; r < rows; r++) {
    for (let c = 0; c < cols; c++) {
      // Checkerboard -- very compressible
      if ((r + c) % 2 === 0) {
        fill(0);
      } else {
        fill(255);
      }
      stroke(200);
      strokeWeight(0.5);
      rect(20 + c * cellSize, 30 + r * cellSize, cellSize, cellSize);
    }
  }
  pop();

  // Right half: random pattern (incompressible)
  push();
  fill(0);
  noStroke();
  textSize(14);
  text("Random (incompressible)", 320, 20);

  randomSeed(123); // Fixed seed for consistency
  for (let r = 0; r < rows; r++) {
    for (let c = 0; c < cols; c++) {
      if (random() > 0.5) {
        fill(0);
      } else {
        fill(255);
      }
      stroke(200);
      strokeWeight(0.5);
      rect(320 + c * cellSize, 30 + r * cellSize, cellSize, cellSize);
    }
  }
  pop();

  // Labels
  fill(0);
  noStroke();
  textSize(12);
  text("Rule: (row + col) % 2", 20, 370);
  text("Program: ~20 characters", 20, 385);
  text("Rule: none (random)", 320, 370);
  text("Program: must list each cell", 320, 385);

  noLoop();
}
```

## Key Takeaways

- **Kolmogorov complexity** measures the length of the shortest program that produces a given output.
- **Low-complexity images** (geometric shapes, patterns) compress well; **high-complexity images** (random noise) do not.
- **RLE** replaces repeated sequences with counts; it works well for images with large uniform areas.
- **Huffman coding** assigns short codes to common symbols and long codes to rare ones.
- **PNG** combines filtering and DEFLATE compression, making it excellent for images with patterns.
- **Source code** can be a more compact representation than the image itself -- this is the foundation of generative art.
- **Schmidhuber's thesis**: beauty arises from patterns that are surprising yet compressible.
- Generative art exists at the intersection of short programs and rich visual output.
